# `city service`

Manage service runtime state and forward service commands.

## Command hierarchy

- Top-level command: `city service`
- Action commands: `city chat` / `city task` / `city memory` / `city skill`

These action commands belong to the service system and are not top-level commands.

## Usage

```bash
city service <subcommand>
```

## Subcommands

- `list`
- `status <serviceName>`
- `start <serviceName>`
- `stop <serviceName>`
- `restart <serviceName>`
- `command <serviceName> <command> --payload <json>`

## Common Scheduled Execution

All service action commands now support common scheduling options:

- `--delay <ms>`: delay execution
- `--time <time>`: schedule execution (Unix sec/ms timestamp or timezone-aware ISO datetime)

Notes:

- scheduled jobs are persisted in the current agent project at `.downcity/schedule.sqlite`
- unfinished jobs are restored after runtime restart
- `--delay` and `--time` cannot be used together
- this is for one-shot delayed/scheduled execution; use `task` service for cron and longer workflows

For detailed behavior, recovery rules, and HTTP examples, see:

- [`Service Action Scheduling`](/en/docs/commands/service/scheduling)

## Notes

`city service` is path-bound to an agent project.

- You can pass `--path <agentProjectPath>`
- Or pass `--agent <agentName>` to resolve from console registry
- If omitted, it uses current directory `.`
- The resolved path must be registered in console registry
- The resolved path must contain `downcity.json` and `PROFILE.md`

Inside agent shell, when `--agent` is omitted and `--path` uses default, it first resolves by injected `DC_AGENT_NAME` via registry; default `.` also prefers injected `DC_AGENT_PATH`.

When both are passed, `--agent` takes precedence.

Start agent runtime first:

```bash
city agent start
```
