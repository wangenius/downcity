---
title: Login and Token Flow
description: "How guest UserClient, the accounts service, your own backend, user_token, and normal UserClient connect together."
---

Every login flow around `@downcity/gate` converges to one goal:

> get `studio_id + user_token` into the studio frontend, then enter normal `UserClient` calls.

## Pattern one: your backend owns login

```txt
frontend submits login data
  -> your backend validates the account
  -> backend uses AdminClient to request user_token
  -> returns studio_id + user_token
  -> frontend creates a normal UserClient
```

Good fit when:

- you already have an account system
- you need more complex org, permission, or risk-control logic

## Pattern two: the accounts service owns login

```txt
frontend creates a guest UserClient
  -> calls accounts.register / login
  -> service returns user_token
  -> frontend creates a normal UserClient
```

Good fit when:

- you want the fastest path to minimal registration and login
- you do not want to maintain your own user and session tables first

## What a guest UserClient is

It is simply a `UserClient` with `base_url` only:

```ts
const guest = new UserClient({
  base_url: "https://base.example.com",
});
```

At this stage it is usually used only for guest-access Actions such as:

- `accounts.register`
- `accounts.login`
- `accounts.oauth/start`
- `accounts.oauth/result`

## Switching to a normal UserClient

```ts
const user = new UserClient({
  base_url: "https://base.example.com",
  studio_id: "studio_xxx",
  user_token: session.user_token,
});
```

At that point the studio enters real user context:

- it can call AI services
- it can call custom services that require user identity
- it can call services such as `accounts.me` and `usage.me`
