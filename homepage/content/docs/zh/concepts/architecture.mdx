# 架构总览

Downcity 最容易理解的方式，是先看两条主干：

- `console`：全局控制与运维面
- `agent`：项目级运行与执行面

在 agent runtime 内部，再由 `service`、`plugin`、`capability`、`asset` 组成执行层。

## 分层结构

1. **Console 层（全局）**
- 命令：`city console ...`、`city console config ...`
- 职责：进程管理、registry、模型池、全局存储

2. **Agent 层（项目）**
- 命令：`city agent create/start/status/restart ...`
- 职责：加载单个项目，持有单个 runtime，把痕迹写入 `.ship/`

3. **Service 层（核心工作流）**
- 例如 `chat`、`skill`、`task`、`memory`
- 负责业务流程与用户主路径

4. **Plugin 层（可选增强）**
- 例如 `voice`
- 暴露 actions、capabilities、hooks、system prompt 片段

5. **Capability + Asset 层**
- `capability`：稳定调用契约，例如 `audio.transcribe`
- `asset`：插件依赖的可安装资源，负责 install/check/config/use

## 架构规则

- 控制与执行分离：`console` 负责管，`agent` 负责跑。
- 工作流与增强分离：`service` 负责流程，`plugin` 负责增强。
- 契约与实现分离：service 调 capability，不碰 plugin 私有细节。
- 行为与依赖分离：plugin 声明所需 asset，asset 自己管理安装与可用性。
