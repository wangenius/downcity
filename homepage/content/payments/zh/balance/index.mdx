`@downcity/services` 提供的是一层全局用户钱包，而不是一套写死的价格规则。

它负责四件事：

- 用户余额账户
- 余额流水
- 充值单
- redeem_code

它不替你决定：

- 哪些 action 应该收费
- 每次该扣多少
- 哪些产品免费，哪些产品收费

这些规则推荐由 Service 自己在内部 `hook` 里直接调用服务来完成。

## 这组文档怎么读

建议按下面顺序读：

1. 先读 [全局钱包模型](/zh/payments/balance/global-wallet)，理解为什么余额不和 downcity 绑定。
2. 再读 [Hook 中直接扣费](/zh/payments/balance/hook-charging)，看推荐接法。
3. 再读 [充值单与流水](/zh/payments/balance/topups-and-ledger)，看“先创建、后确认”的充值链路。
4. 最后读 [redeem_code 兑换](/zh/payments/balance/redeem-codes)，看一次性礼品码如何直接入账。

## 最小启用方式

```ts
import { BalanceService } from "@downcity/services";

const balance = new BalanceService({
  init_credits: 100_000_000,
});

base.use(balance);
```

## 最重要的边界

### 余额是全局用户钱包

一个用户在多个产品之间共用同一份余额。

### 扣费策略在业务层

你应该在自己的 `hook` 中写：

- 调用前 `balance_precheck()`
- 调用成功后扣费

而不是把价格规则塞进服务配置里，也不是在 `base.use(service)` 的地方外部追加。

## 常见组合

- 和 [@downcity/services](/zh/payments) 配合，让登录后的用户拿到统一钱包
- 和 [@downcity/services](/zh/payments) 配合，把“发生了什么”与“扣了多少钱”拆开治理
- 和 [@downcity/services](/zh/payments) 配合，把支付结果映射成充值到账
- 单独使用 `redeem_code`，发礼品码、补偿码、活动赠送码
