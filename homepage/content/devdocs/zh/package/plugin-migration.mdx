# Plugin 迁移方案

这页是后续继续重构时可以直接复用的迁移规则。

## 迁移目标

把旧的工作流导向逻辑迁到当前模型：

- session 负责执行
- plugin 负责暴露显式能力
- hooks 统一使用 `pipeline / guard / effect / resolve`
- 托管 plugin 只在确实需要长期运行态时承担 runtime 行为

## 判断规则

### 继续留在执行主轴附近的情况

- 它改变一轮执行如何被装配或执行
- 它依赖 history、prompt 或 model loop
- 它本身就是 turn ownership 的一部分

### 迁到 plugin 的情况

- 它是在对外暴露能力
- 它是在增强一个稳定运行时点
- 它在注入 system 指导
- 它拥有一个托管 runtime 模块

## 四种 hook 语义怎么选

### `pipeline`

用于串行改写值。

### `guard`

用于可阻断的校验。

### `effect`

只做副作用。

### `resolve`

用于单点单实现的值解析。

## 标准迁移步骤

1. 先判断这段逻辑属于执行拥有权，还是能力拥有权。
2. 如果属于能力拥有权，再判断它是 action、hook、system 还是 managed runtime。
3. 把临时扩展行为替换成四种稳定 hook 语义之一。
4. 把依赖检查与 availability 逻辑收回 plugin 边界。
5. 只要公开行为或协作者心智变了，就同步更新文档。

## 推荐参考样板

- chat ingress 周围的 auth
- 语音转写作为 inbound augmentation
- skill lookup / install 作为显式 action
- shell runtime 作为托管 plugin

## 迁移时重点检查什么

- 文档里是否还在描述一层独立的顶层扩展层
- 工作流拥有权和能力暴露权是否又被混在一起
- 某个 plugin 是否在试图替代 session 执行，而不是增强它
