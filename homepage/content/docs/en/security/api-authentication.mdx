# API Authentication

Unified Auth V1 adds the core auth endpoints for Downcity.
The current model is intentionally simple: one admin account, admin password rotation, and multiple Bearer tokens.

- `POST /api/auth/bootstrap-admin`
- `POST /api/auth/login`
- `GET /api/auth/me`
- `POST /api/auth/password/update`
- `GET /api/auth/token/list`
- `POST /api/auth/token/create`
- `POST /api/auth/token/revoke`

## First admin bootstrap

If the console database does not have any user yet, bootstrap the first admin:

```bash
curl -X POST http://127.0.0.1:3000/api/auth/bootstrap-admin \
  -H 'Content-Type: application/json' \
  -d '{
    "username": "admin",
    "password": "pass-123456",
    "displayName": "Admin",
    "tokenName": "bootstrap"
  }'
```

The response returns:

- the user summary
- a one-time plaintext Bearer token

The plaintext token is only returned at issuance time.
If you initialize auth through `city start` or `city start`, the CLI prompts for the first admin password and falls back to `downcity` when left blank. If you call `bootstrap-admin` directly, the password is exactly the `password` you submit.

## Sign in

After bootstrap, create more tokens through login:

```bash
curl -X POST http://127.0.0.1:3000/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{
    "username": "admin",
    "password": "the password you set during bootstrap-admin"
  }'
```

`tokenName` is optional for raw API callers. Console UI sets its own internal default and does not ask the user to provide one.

## Use the Bearer token

Send the token through the `Authorization` header:

```bash
curl http://127.0.0.1:3000/api/auth/me \
  -H 'Authorization: Bearer dc_xxx'
```

If the token is valid, the endpoint returns the current user, roles, and permissions.

## `DC_AUTH_TOKEN` and `DC_AGENT_TOKEN`

Start with the main rule:

- Only external HTTP calls need a Bearer token
- Local `city service`, `city plugin`, `city task`, `city chat`, and similar commands now use IPC by default, so they do not need a token first

For direct API users, the key idea is simple:

- external calls always authenticate through `Authorization: Bearer <token>`

Inside the CLI/runtime, you may still see two env vars:

| Variable | Intended For | Purpose |
| --- | --- | --- |
| `DC_AUTH_TOKEN` | users, scripts, CI | explicitly choose which token this CLI/API invocation should use |
| `DC_AGENT_TOKEN` | Downcity agent runtime | automatically propagate auth when an agent explicitly makes a remote HTTP call |

Recommended mental model:

- if you are a human user or an external script, prefer `--token` or `DC_AUTH_TOKEN`
- if an agent explicitly makes a remote HTTP call internally, runtime can inject `DC_AGENT_TOKEN`

The CLI only uses the following token resolution order for explicit remote HTTP calls:

1. `--token`
2. `DC_AUTH_TOKEN`
3. `DC_AGENT_TOKEN`
4. local CLI auth state

So:

- `DC_AUTH_TOKEN` is explicit override
- `DC_AGENT_TOKEN` is internal propagation
- both eventually become the same Bearer header on the actual HTTP request

If you see `Missing bearer token`:

- do not start by suspecting `sessionId`
- first confirm that this invocation is actually using remote HTTP
- then check whether the current process has any valid token source
- if the error happens inside an agent/Bash/ACP subprocess making a remote HTTP call, inspect the `DC_AGENT_TOKEN` inheritance path first

## Manage tokens

List tokens:

```bash
curl http://127.0.0.1:3000/api/auth/token/list \
  -H 'Authorization: Bearer dc_xxx'
```

Create a token:

```bash
curl -X POST http://127.0.0.1:3000/api/auth/token/create \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "name": "chrome-extension"
  }'
```

Revoke a token:

```bash
curl -X POST http://127.0.0.1:3000/api/auth/token/revoke \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "tokenId": "token_record_id"
  }'
```

## Rotate the admin password

The control plane now keeps only one username/password account: the admin.

```bash
curl -X POST http://127.0.0.1:3000/api/auth/password/update \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "currentPassword": "old-pass",
    "nextPassword": "new-pass"
  }'
```

Notes:

- Rotating the password does not automatically revoke existing tokens
- Other integrations continue to authenticate with Bearer tokens directly
- Plaintext tokens are still returned only once, at issuance time

## Current scope

This rollout covers the V1 auth backbone:

- unified user / role / permission / token / audit tables
- Bearer token issuance and validation
- admin password rotation
- current-admin token management endpoints

Later iterations will move more control-plane APIs behind the same Bearer auth layer.
