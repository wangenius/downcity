# Port and Instance Lifecycle

## Unified Rule

- Agent ports are allocated by Downcity runtime.
- Agent lifecycle operations do not require users to manage ports manually.
- Plugin calls resolve the target agent first, then the endpoint.

## Runtime Roles

### Downcity runtime

- Maintains `~/.downcity/main/agents.json`
- Starts/stops/restarts agent daemons
- Allocates agent ports

### Console

- Provides the browser UI for observing and controlling agents
- Does not own local runtime startup

### agent

- One project maps to one daemon
- Runtime HTTP APIs include `/api/plugins/list`, `/api/plugins/catalog`, and `/api/plugins/*`
- Debug metadata is persisted to `.downcity/.debug`

### plugin endpoints

- plugin calls enter plugin routes
- plugin routes are dispatched in the target runtime

## Agent Port Allocation

1. Validate Downcity runtime is running
2. Validate project files
3. Allocate an available port
4. Start daemon
5. Persist debug metadata
6. Upsert managed agent registry

## Endpoint Resolution

For plugin API calls:

1. explicit CLI `--host` / `--port`
2. `DC_CITY_HOST/DC_CITY_PORT`
4. daemon metadata
5. default `127.0.0.1:5314`

## Recommendations

- In multi-agent scenarios, plugin actions should resolve an explicit target agent. Actions run locally against the selected project path.
- Do not keep stale `DC_CITY_PORT` in your shell profile.
- If runtime status is abnormal, repair daemon metadata and inspect daemon logs.
