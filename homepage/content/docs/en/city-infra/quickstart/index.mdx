This section only does one thing: help you run a complete Downcity call chain as quickly as possible.

What you end up with is not just a demo page. It is the smallest product-shaped system:

- one reusable InfraRuntime
- one model record
- one `product_id`
- one `user_token` that an end user can actually use
- one `UserClient` that successfully calls InfraRuntime

You do not need a separate AI backend for every new product. A new project can start as a client first and connect to an already deployed InfraRuntime.

## Recommended path

1. [Use it now](/en/docs/city-infra/quickstart/use-now): run the shortest end-to-end path with `blocks/node` and `packages/city-terminal`
2. [Create InfraRuntime](/en/docs/city-infra/quickstart/create-infra): create InfraRuntime and let the first startup initialize the default database automatically
3. [Add model](/en/docs/city-infra/quickstart/add-model): register models with Provider and AIService
4. [Issue `user_token`](/en/docs/city-infra/quickstart/issue-token): give a product user a valid call credential
5. [Connect client](/en/docs/city-infra/quickstart/connect-client): call InfraRuntime from the product with `UserClient`

## Assumptions in this quickstart

- the example InfraRuntime uses `sqlite:./data.sqlite`
- models come from runtime registration code, not handwritten model tables
- all products currently share the same AIService model directory
- default models are defined through `Provider.model({ default })`
