# Tasks

Tasks are reusable pieces of scheduled agent work.

They are not just delayed chat messages. A task has its own definition, schedule, run artifacts, and result delivery semantics.

## What A Task Owns

A task definition describes:

- what should run
- when it should run
- which session should receive the result
- whether the task is enabled, paused, or disabled

The task runtime writes run artifacts under the project `.downcity/` state tree. That keeps scheduled work tied to the project agent that owns it.

## Result Delivery

`sessionId` decides where the task result returns.

It is not an authentication identity. It is a delivery route back into an existing session or chat context.

## Time Semantics

Task schedules are evaluated by the runtime process. Cron expressions therefore follow the runtime environment's timezone.

One-shot time triggers should include an explicit timezone so that the stored schedule is not ambiguous.
