# tools and plugins

## tools

Local Agents can receive an explicit tool set:

```ts
const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  tools: {
    my_tool: myTool,
  },
});
```

These tools are available during session execution.

## plugins

Local Agents can also receive explicit plugin instances:

```ts
import { Agent } from "@downcity/agent";
import { SkillPlugin, WebPlugin } from "@downcity/plugins";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  plugins: [new SkillPlugin(), new WebPlugin()],
});
```

Registered plugin actions can be called through `agent.plugins.runAction(...)`, and plugin `system(context)` text is injected into session prompts.

## ImagePlugin

Use `ImagePlugin` when you want an Agent to generate images during a conversation:

```ts
import { Agent, ImagePlugin } from "@downcity/agent";

const agent = new Agent({
  id: "creative-agent",
  path: "/path/to/project",
  model,
  plugins: [
    new ImagePlugin({
      create: (input) => city.ai.imageJobCreate(input),
      status: (input) => city.ai.imageJobStatus(input),
      result: (input) => city.ai.imageJobResult(input),
    }),
  ],
});
```

After registration, the Agent automatically gets the built-in `plugin_call` tool. Image generation is exposed as an observable job workflow:

```ts
plugin_call({
  plugin: "image",
  action: "create",
  payload: {
    prompt: "A cinematic illustration of a rainy city corner at night",
    aspect_ratio: "16:9",
  },
});

plugin_call({
  plugin: "image",
  action: "status",
  payload: { job_id: "img_..." },
});

plugin_call({
  plugin: "image",
  action: "result",
  payload: { job_id: "img_..." },
});
```

`ImagePlugin` is provider-agnostic. It does not know about City, OpenAI, Gemini, 302.ai, or Luchi directly. If you only pass `image(input)`, it wraps that function in a local background job. If your backend already exposes jobs, pass `create/status/result` so the Agent can poll the real upstream state without depending on a synchronous image call.

The legacy `generate` action still exists as a bounded wait shortcut, but agents should prefer `create -> status -> result` so progress, failures, and retries are visible.

Generated images are not left only inside the tool result. The Agent runtime merges generated `file` parts into the final assistant message, so session history stores:

```ts
{
  role: "assistant",
  parts: [
    { type: "text", text: "Image generated." },
    { type: "file", mediaType: "image/png", url: "..." },
  ],
}
```

## plugin HTTP

If Town exposes the Agent HTTP gateway, registered plugins with `plugin.http.server.register(...)` are also mounted onto the same HTTP app automatically.

So:

- plugin existence comes from the `plugins` array you pass into `Agent`
- plugin HTTP exposure happens when Town publishes the Agent HTTP gateway
