# City User Identity

Federation issues user identities, while the City client uses a `user_token` to access Federation or a product backend. One Federation can serve multiple products, isolated by `city_id`.

## Identity boundaries

- **Federation** — stores the Ed25519 private key and issues `user_token` values
- **City** — stores `federation_url + user_token` and makes user requests
- **Product backend** — identifies users with `FedBureau` and Federation public keys
- **FederationAdmin** — provides the separate Federation management control plane

## Authentication flow

1. The user selects a product `city_id` during login
2. Federation signs a `user_token` bound to that `city_id`
3. The user calls Federation through `new City({ federation_url, user_token })`
4. The user sends the same Bearer Token to the product backend
5. The backend's `FedBureau` fetches and caches Federation JWKS, then identifies the user

```ts
const city = new City({
  federation_url: "https://fed.example.com",
  user_token,
});
```

Product backend:

```ts
const bureau = new FedBureau({
  federation_url: "https://fed.example.com",
  city_id: "city_product_a",
});

const identity = await bureau.identify(request);
```

The public key is safe to publish: it can verify tokens but cannot issue or forge them. Federation publishes:

```text
/.well-known/downcity.json
/.well-known/jwks.json
```

The product backend must preconfigure its trusted `federation_url` and `city_id`. It must never select JWKS from an unverified token.

## Channel accounts

Bot credentials for messaging platforms are stored in `~/.downcity/downcity.db` and referenced by ID from the project:

```json
{
  "plugins": {
    "chat": {
      "channels": {
        "telegram": {
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```
