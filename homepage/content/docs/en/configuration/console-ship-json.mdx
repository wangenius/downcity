# Console Config Storage (ship.db)

From current versions, Console no longer uses:

1. `~/.ship/ship.json`
2. `~/.ship/.env`

Console-global config is unified in:

- `~/.ship/ship.db`

## What is stored

`ship.db` contains two groups:

1. Model pool (providers/models)
2. Console-global config (for example `extensions.*`)

Sensitive fields (such as API keys) are stored encrypted.

## Boundary with Agent project config

These files remain agent-project-level config (unchanged):

1. `<agent>/ship.json`
2. `<agent>/.env`

So the split is:

- Console: `~/.ship/ship.db`
- Agent: project-local `ship.json/.env`

## Environment Variable Scope (Important)

Current versions split environment scope into two layers:

1. Console-global (shared): for console-level capabilities such as model pool and extensions.
2. Agent-project (private): from `<agent>/.env`, used by `services.*` (for example chat channel credentials).

Key behavior:

1. An agent `.env` is applied only inside that agent runtime and does not mutate global `process.env`.
2. Switching agents does not reuse `.env` values from the previous agent.
3. `${ENV_KEY}` in project `ship.json` resolves from current agent `.env` only (no fallback to shared `process.env`).
4. Only shared fields from Console global layer are injected into agent config (currently `extensions` only), never `services.*`.
5. So `services.chat` reads the current agent private env, while model/extension in console sidebar remains in shared config scope.

## Common operations

Initialize Console (writes into `ship.db`):

```bash
sma console init
```

Read/update console extension config:

```bash
sma console config get extensions.voice
sma console config set extensions.voice.enabled true
```

Manage model pool:

```bash
sma console model list
sma console model create
```

## Runtime files

These runtime files stay in filesystem (not config):

1. `~/.ship/console/console.pid`
2. `~/.ship/console/ui.pid`
3. `~/.ship/console/console.log`
4. `~/.ship/console/ui.log`
5. `~/.ship/console/agents.json`
