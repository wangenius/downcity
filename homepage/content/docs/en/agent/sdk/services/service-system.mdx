# Service System Text

A service can implement:

```ts
system(context) {
  return "some service-level system text";
}
```

Semantically, that means the service wants to contribute a service-level system block.

## What it is good for

- service-specific usage rules
- behavior boundaries for the service
- stable domain instructions that conceptually belong to that service

## But the current SDK has one very important boundary

The current local SDK session path uses `SdkSessionSystemComposer`.

Today it mainly injects:

- `PROFILE.md`
- `SOUL.md`
- the core prompt
- the runtime clock prompt

It does **not currently auto-inject `service.system()`** into the main local SDK session prompt path.

## What that means in practice

It means:

- `service.system()` is part of the service contract
- but it is not yet a "write this and the model immediately sees it every run" feature in the current local SDK path

If you assume "my custom service has `system()`, so the model must already see it," you will have the wrong expectation.

## So how should you use it today

The more practical mental model for SDK users is:

- keep `system()` as part of a complete service shape
- keep truly active local SDK prompt rules in `PROFILE.md` and `SOUL.md`
- treat services primarily as runtime, state, and lifecycle holders

## When it is still worth implementing

Even though the current local SDK path does not auto-inject it, `system()` still has value:

- it keeps your service shape complete
- it stays aligned with the broader runtime service model
- if the SDK expands this behavior later, your service shape does not need to change

## Safest current recommendation

If a rule must affect the local SDK session prompt right now, prefer:

- `PROFILE.md`
- `SOUL.md`

If a rule conceptually belongs to a service and is more about full contract shape or future expansion, `service.system()` is still a reasonable place.
