# Telegram 集成

> ⚠️ **简化模式（2026-02-03）**：当前发布的 `downcity` 包已暂时移除审批（默认全权限执行）。旧文档里关于“审批提示”的描述可能已过期。

使用 Telegram 和 Downcity 对话（私聊或群聊）。

> 想了解内部怎么跑起来的？看这篇：[/zh/docs/services/chat/telegram-message-flow](/zh/docs/services/chat/telegram-message-flow)

## 配置步骤

### 1）创建 Telegram 机器人

1. 打开 Telegram，搜索 `@BotFather`
2. 发送 `/newbot`
3. 按提示设置名称与用户名
4. 复制 **HTTP API Token**

### 2）创建 Channel Account 并绑定到项目

在 Console UI 的 `Global / Channel Accounts` 创建 Telegram 账户（例如 `telegram-main`）：

1. `channel = telegram`
2. 填入 BotFather 生成的 `botToken`

然后在 `ship.json` 里绑定这个账户：

```json
{
  "services": {
    "chat": {
      "channels": {
        "telegram": {
          "enabled": true,
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

> Downcity 的聊天回复采用 tool-strict：Agent 需要使用 `chat_send` 工具自行决定何时发消息（可以分多条、分阶段发送）。

### 3）启动 Downcity

```bash
city agent start
```

日志里看到以下内容说明启动成功：

- `Starting Telegram Bot...`
- `Bot username: @...`
- `Telegram Bot started`

## 使用方式

### 私聊

给机器人发 `/start`（或任意消息）即可开始对话。

### 发送文件（PDF / 图片 / 音频 / 视频）

你可以直接给机器人发送文件（document/photo/voice/audio/video）。

- 如果你在发送文件时同时输入文字，Telegram 会把这段文字当作 **caption（附件说明）** —— Downcity 会把 caption 当作你的指令来处理。
- Downcity 会把文件保存到 `.ship/.cache/telegram/`，并以 `@attach ...` 的形式传给 Agent。

示例：发送一个 PDF，并在 caption 写：`帮我总结这个 PDF，并列出待办事项。`

### 本地 STT 模型管理

如果你要统一管理本地语音转写模型，请使用独立的 `voice` plugin 命令组：

```bash
# 启用并安装模型
city voice on SenseVoiceSmall

# 查看当前 voice 配置
city voice status
```

当 `voice` plugin 已启用，且 `voice.transcriber` asset 可用时，Telegram 入站 `voice/audio` 附件会以 best-effort 模式触发转写，并把识别文本追加到用户指令中。

### 群聊 / 超级群

- 把机器人拉进群
- 群成员发送普通消息即可触发对话（不需要 `@bot`）

说明：
- 如果群开启了 **Topics（话题）**，机器人会尽量在同一个话题里回复。
- 群里发送文件时，caption 会作为指令文本一并处理，不需要额外 `@bot`。
- Agent 执行期间会周期性发送 `sendChatAction(typing)`，你会看到 Telegram 的“正在输入”提示，直到结果发出。

## 回复关联与贴表情逻辑（重点）

这一段解释两件事：
- 什么时候会走 `reply_to_message`
- 什么时候适合用 `chat react` 贴表情而不是发文本

### 1）`reply_to_message` 是怎么决定的

当 Agent 执行 `city chat send` 时，默认是普通发送，不会自动挂 reply。
只有显式传了 `--reply`，运行时才会尝试附带 `reply_to_message_id`。

`messageId` 的来源优先级：
1. 显式参数（例如你在 `city chat react --message-id <id>` 里指定）
2. 当前 `chatKey` 最近一次入站消息元信息（chat meta 自动维护）

行为规则：
- 能拿到合法数字 `messageId`：发送时附带 `reply_to_message_id`，Telegram 显示为“回复某条消息”
- 拿不到 `messageId`：仍会发送成功，但作为普通消息发送（不挂回复关系）
- 这个规则对文本和附件都生效（附件也会尝试带 `reply_to_message_id`）

### 2）`chat react` 是怎么工作的

`chat react` 适合“轻量确认/表达态度”场景，例如：
- “收到”
- “同意”
- “赞同/鼓励”

示例：

```bash
# 默认 👍
city chat react

# 指定表情
city chat react --emoji '🔥'

# 显式指定目标消息
city chat react --emoji '✅' --message-id 12345 --chat-key telegram-chat--100xxxx
```

行为规则：
- 默认表情是 `👍`
- `--big` 会启用 Telegram 的 `is_big` 效果
- 如果没有显式 `--message-id`，会尝试从当前 `chatKey` 的最新入站消息回填
- 如果最终拿不到合法 `messageId`，`chat react` 会失败并提示需要消息 ID（因为反应必须贴到具体消息）

### 3）建议实践

- 需要明确答复用户（结论/下一步）时：用 `city chat send`
- 只需快速确认态度时：优先用 `city chat react`
- 只有在消息很多、你希望“明确挂在某条消息下”时，再用 `city chat send --reply`
- 跨会话操作时：显式传 `--chat-key`，避免贴错会话
- 刚建新会话还没有入站消息时：先让对方发一条消息，再做 reply/react

## 常见问题排查

- **完全没反应**：确认进程在跑，并查看 `.ship/logs/<YYYY-MM-DD>.jsonl`（可搜索 `TELEGRAM` / `telegram` / error 关键词）。
- **群里收不到消息**：在 BotFather 执行 `/setprivacy` 设为 `Disable`，并确认机器人已被允许读取群消息。
- **轮询收不到（getUpdates 冲突）**：如果之前配置过 webhook，会与轮询冲突。Downcity 启动时会尝试清理 webhook；留意日志里的 webhook/Conflict 报错。
- **群里发不出消息**：检查群权限/管理员设置，机器人可能被禁言或不允许发消息。
- **Agent 口头说“我去执行”但没真正跑工具**：请在消息里明确写执行意图关键词（例如“执行 / run / 用 skill / lookup”）。这些回合运行时会强制触发一次工具调用；若仍失败，优先检查当前模型/网关是否支持工具调用。
