The `downcity` package installs the `city` command for City operators. `city` is not a City runtime SDK.

## When you should open city first

- you want to confirm what is actually in the database
- you are initializing City locally
- you suspect a service did not create or write the expected data
- you do not want to write a one-off script just to inspect models or tables

## Install

```bash
npm i -g downcity
```

If you are iterating on `city manage` inside this repository, run:

```bash
pnpm build:cli
```

That command builds the workspace `city`, `city`, and `downcity` CLIs. It does not automatically rewrite your global commands. To refresh an installed global CLI, use `city update` or reinstall the global package.

If `city` / `city` is already installed, you can also run:

```bash
city update
```

`city update` updates the currently installed global Downcity CLI. During repository development, use the workspace build and local entry points when you only need to verify local source changes.

## First run

```bash
city
```

In an interactive terminal, that opens the full-screen City TUI dashboard, which is often the fastest first inspection entry.

In non-interactive environments, `city` prints command help. Use explicit commands such as `city manage`, `city create .`, `city deploy`, or `city env refresh` when scripting.

If you do not have any connected City yet, the CLI does not inject a default address. It first opens a welcome dashboard flow.

## First connect a City

On first launch, the default primary path is:

- `Connect an existing City`

That step only asks for:

- `base_url`

The CLI runs a public reachability check, then saves the City connection locally.

It does not require `admin_secret_key` during connect.

## What happens after connect

After saving, that City becomes the current workspace. From there, the default path is user-first:

- if a local user session already exists for that City, `city` opens the user workspace directly
- if no local user session exists yet, `city` opens sign in first

So the default mental model is now:

1. connect a City
2. sign in as user
3. open server management only when trusted-side admin work is needed

## How server management works afterward

The home screen now revolves around the current City workspace:

- `Open current City`
- `Switch City`
- `Connect another City`
- `Upgrade CLI`

The user workspace keeps high-frequency user actions such as:

- `Models`
- `Balance`
- `Recharge`
- `My profile`
- `Services`

Low-frequency trusted-side administration is grouped under:

- `Server management`

Inside `Server management`, you can then choose:

- `Configure admin access`
- `Open admin tools`
- `Edit City`
- `Remove City`

## A common first workflow

```bash
npm i -g downcity
city
```

That is often enough for the first round of confirmation:

1. connect a working City
2. save it as the current local workspace
3. continue into user sign in or the user workspace

## What the CLI is best at

### Scenario 1: human inspection

If your real question is whether data actually got written, whether usage was recorded, or whether model configuration is wrong in the DB, the CLI is often faster than writing temporary code.

### Scenario 2: model initialization or repair

```bash
city
```

Those tasks now start from the interactive menu instead of assuming a default server.

### Scenario 3: manual Stripe recharge

If your City enables the Stripe payment provider, the CLI can now:

- read Stripe env requirements exposed by the service
- configure admin access from `Server management`
- let admin tools configure `STRIPE_SECRET_KEY` and `STRIPE_WEBHOOK_SECRET`
- show the Stripe webhook endpoint for the current server in `Payment`
- create a topup as a user
- create a Stripe Checkout Session and open the browser for payment

Stripe success and cancel pages are generated automatically from `DOWNCITY_CITY_BASE_URL`. If it is not configured, the service falls back to the current request origin for local development.

The Stripe webhook endpoint itself is fixed:

- `{base_url}/v1/payment/webhook?provider=stripe`

## What it is not for

- embedding into city code
- replacing backend management workflows
- long-lived automation

Those are better owned by `Admin City` or trusted backend scripts.

## Common entry points

- `city` / `city`
- `city update`
- `Connect an existing City`
- `Server management`
- `Sign in`

## Common misunderstandings

### The CLI is not for end users

It is for City operators, developers, and maintainers.

### The CLI does not replace Admin City

The CLI is for a human at a terminal. `Admin City` is for trusted programmatic workflows.

### The CLI is not only for table inspection

It is also a practical entry for initialization, model maintenance, and debugging.
