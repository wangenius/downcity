# Agent, Town, and City

Three concepts appear constantly in Downcity:

- `agent`: the runtime for one project
- `town`: the local manager for Agents and local shared resources
- `city`: the City layer for model, admin, and platform resources

## The short version

- `agent` owns how one project runs
- `town` owns which local Agents exist and local shared resources such as chat accounts
- `city` owns resources such as AIService models and admin-managed services

If you prefer a control-plane vs execution-plane mental model:

- `town` is the local Agent control plane
- `city` is the City plane
- `agent` is the execution plane

## What Town owns

Town is responsible for local Agent management:

- managing multiple agent projects
- starting, stopping, and inspecting agent runtimes
- maintaining local shared chat accounts for the chat plugin
- connecting the local Agent runtime to City

Town does not own model providers, model credentials, or the City AIService model catalog.

## What City owns

City is responsible for model, admin, and platform resources:

- maintaining the City AIService model catalog
- providing shared platform credentials and service APIs

These are shared resources. They do not naturally belong to one project or to the local Town process.

## What the Agent owns

The single-project agent runtime is responsible for project execution:

- reading the current project config
- creating and running sessions
- starting project plugins
- binding the project to its selected model
- exposing project-level HTTP and CLI entry points
- persisting project messages and logs

These concerns are tightly bound to one project because they depend directly on that project's prompts, config, history, and tools.

## Why the split matters

Without this split, the system quickly becomes muddy:

- multi-project management and single-project execution get coupled
- global config and project config get mixed together
- it becomes much harder to tell whether a failure belongs to the control plane or the project runtime

This split is also what keeps many CLI and HTTP behaviors understandable:

- `town` locates or manages the target project
- the project `agent` actually executes the work

## What this means in practice

For day-to-day use, keep two rules in mind:

1. When you configure the project runtime, you edit project files like `downcity.json`.
2. When you manage local Agents and plugins, you use `town`.
3. When you manage model providers and City base resources, you use `city` and keep Town connected to City.

## Common examples

### Model binding

- the model definition lives in City AIService
- the project chooses its model through `downcity.json.execution.modelId`

So model definition and model selection are separate concerns.

### Chat channels

- bot credentials and chat accounts are Town-level shared resources for the chat plugin
- the project only binds a `channelAccountId`

That is why project config usually contains something like:

```json
{
  "channelAccountId": "telegram-main"
}
```

instead of raw credentials.

### Runtime status

- project runtime status answers whether one agent is running
- the managed-agent registry answers which agents are known globally

## A common misunderstanding

People often ask: "If Town is the entry point, is the agent really separate?"

Yes.

Town is the user-facing entry point. The actual runtime being started, queried, or stopped is the single-project agent runtime.
