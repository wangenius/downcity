title: Deploy to Cloudflare Workers

This page is the operational path for deploying a City project to Cloudflare Workers.

If you only want to understand how City is wired to Workers and D1, read [Cloudflare Workers](/en/city-sdk-docs/guides/cloudflare-workers) first. This page focuses on production steps: `city.json`, local project creation, local deploys, D1, admin key retrieval, env setup, OAuth callbacks, and health checks.

## Prerequisites

- A Cloudflare account
- Dependencies installed in the project

Downcity does not deploy Workers automatically after `git push`. Deployment is explicit:

```bash
city deploy
```

## 1. Create a City project

Developers normally start from an empty directory:

```bash
mkdir my-city
cd my-city
city create .
```

`city create .` interactively asks for the City name and deploy target, then generates the project skeleton:

- `city.json`
- `src/index.ts`
- `package.json`
- `tsconfig.json`
- `.env.example`

The generated `city.json` only stores basic project information:

```json
{
  "type": "city",
  "name": "my-city",
  "target": "cloudflare-workers"
}
```

`type` says this is a City project, and `target` declares where it deploys. The current target is `cloudflare-workers`, which means Cloudflare Edge Workers. The entry file, D1, bindings, and Wrangler config are generated by `city create` and `city deploy` from the target convention instead of being written into `city.json`.

If the City template already lives in a Git repository, use `city create` to bring it local first:

```bash
city create https://github.com/example/my-city.git
cd my-city
```

`city deploy` does not deploy remote URLs directly. It only deploys local City projects, so `.env` and deploy results have a clear project owner.

## 2. Deploy the current directory

Inside the City project directory:

```bash
city deploy
```

When `city.json` uses the `cloudflare-workers` target, `city deploy` checks Cloudflare auth automatically. If Wrangler is not logged in, it opens Wrangler login. If Wrangler is logged in but Cloudflare cannot return an account automatically, City prompts you to refresh login or enter the Cloudflare account id once. The account id is saved to the project `.env` and reused by later deploys.

You can also pass a local directory:

```bash
city deploy ./my-city
```

## 3. Local deploy values

Common local values can live in the project `.env`:

```bash
CITY_D1_DATABASE_NAME=my-city-db
```

Cloudflare account is not part of the City project config. `city deploy` reuses Wrangler login first; only when Cloudflare cannot resolve the account automatically does City prompt once and save it into local City CLI state.

You can also pass the account id for one deploy:

```bash
city deploy --account-id <your-cloudflare-account-id>
```

## 4. Dry-run

Before publishing, check the bundle and Wrangler config:

```bash
city deploy --dry-run
```

For another directory:

```bash
city deploy ./my-city --dry-run
```

## 5. Deploy flow

`city deploy` runs these steps:

- reads `city.json` from the target directory
- loads `.env` from the target directory
- checks the target platform identity from `city.json.target`; Cloudflare Workers opens Wrangler login or guides account selection when needed
- if this is not `--dry-run`, automatically bumps `package.json.version` by one patch version before deployment
- runs `pnpm build` if package.json has a `build` script
- runs `pnpm typecheck` if package.json has a `typecheck` script
- prepares D1 for the `cloudflare-workers` target by looking up `CITY_D1_DATABASE_NAME`, and creates it automatically when missing
- generates Wrangler config temporarily instead of writing it into the project directory
- runs `wrangler deploy --config <generated-wrangler.toml>`
- writes the D1 database name back to the project `.env`
- automatically registers the deployed Worker URL as the current City server
- requests Worker `/health` when `--verify` is passed

`.env` is the local deploy binding file and should not be committed to Git. `city.json` stays as the commit-safe project declaration.

`city deploy --dry-run` and `city deploy --verify-only` do not change the project version.

## 6. Developer edge starter

The Downcity repository keeps `cities/edge` as a developer-friendly starter project:

```bash
city deploy cities/edge --verify
```

This starter is useful for local experiments and custom deployments. Downcity's private official production Worker implementation lives outside the public repository.

## 7. Initialize City

Call `/health` once after deployment. This lets the Worker boot City and create the built-in `env` / `towns` tables.

```bash
curl https://my-city.<your-subdomain>.workers.dev/health
```

Expected shape:

```json
{
  "ok": true,
  "services": ["env", "towns", "accounts", "balance", "payment", "usage", "ai"]
}
```

## 8. Retrieve the admin key

On first boot, City stores `DOWNCITY_CITY_ADMIN_SECRET_KEY` in its own `env` table. For Workers + D1, retrieve it from the remote D1 database:

```bash
npx wrangler d1 execute my-city-db --remote \
  --command "SELECT value FROM env WHERE key='DOWNCITY_CITY_ADMIN_SECRET_KEY'"
```

Keep this key in a trusted place. It is the credential used by `Admin City`, Town CLI, or direct admin HTTP calls.

## 9. Write provider and service env

Provider keys and service secrets should be written into City's env table instead of public clients.

Example:

```bash
curl -X POST https://my-city.<your-subdomain>.workers.dev/v1/env/upsert \
  -H "Authorization: Bearer <admin_secret_key>" \
  -H "Content-Type: application/json" \
  -d '{"key":"DEEPSEEK_API_KEY","value":"sk-..."}'
```

Common keys:

| Key | Purpose |
| --- | --- |
| `DEEPSEEK_API_KEY` | Provider key used by the edge starter project |
| `STRIPE_SECRET_KEY` | Stripe API key for Checkout creation |
| `STRIPE_WEBHOOK_SECRET` | Stripe webhook signing secret |
| `DOWNCITY_CITY_BASE_URL` | Public City base URL used to derive payment result pages |
| `CREEM_API_KEY` | Creem API key for Checkout creation |
| `CREEM_PRODUCT_ID` | Creem product used for hosted Checkout |
| `CREEM_WEBHOOK_SECRET` | Optional Creem webhook signing secret |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | Optional Google OAuth login |
| `WECHAT_CLIENT_ID` / `WECHAT_CLIENT_SECRET` | Optional WeChat website login |

## 10. Configure OAuth callbacks

If you enable OAuth login, configure the provider callback URL to the deployed Worker:

```text
https://my-city.<your-subdomain>.workers.dev/v1/accounts/oauth/callback
```

The callback domain must match the public domain users actually visit. If you later move to a custom domain, update the OAuth provider configuration as well.

## 11. Verify

If a City server is already connected, run only the health check:

```bash
city deploy --verify-only
```

You can also check manually:

```bash
curl https://my-city.<your-subdomain>.workers.dev/
curl https://my-city.<your-subdomain>.workers.dev/health
```

## Updating the deployment

After code changes:

```bash
city deploy --verify
```

Runtime env lives in D1, so redeploying the Worker does not erase provider keys, OAuth secrets, towns, balances, usage records, or Stripe payment records.
