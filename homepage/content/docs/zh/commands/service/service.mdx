# `city service`

`city service` 是 service 的 console 级 catalog 入口。

适合用来：

- 查看当前注册了哪些 service
- 查看某个 service 的静态能力面
- 在少数高级场景下，对指定 agent 做 lifecycle 或 command 转发

## 命令层级

- 一级命令：`city service`
- 具体 service 命令：`city chat` / `city task` / `city memory`

这些具体命令才是日常面向 agent 的运行态入口。

## 用法

```bash
city service <subcommand>
```

## 子命令

- `list`
- `status <serviceName>`
- `start <serviceName>`
- `stop <serviceName>`
- `restart <serviceName>`
- `command <serviceName> <command> --payload <json>`
- `schedule list`
- `schedule info <jobId>`
- `schedule cancel <jobId>`

## 通用定时执行

所有 service action 命令现在都支持通用调度参数：

- `--delay <ms>`：延迟执行
- `--time <time>`：定时执行（Unix 时间戳秒/毫秒或带时区 ISO 时间）

说明：

- 调度任务会持久化到当前 agent 项目的 `.downcity/schedule.sqlite`
- agent 重启后，会继续恢复尚未完成的任务
- `--delay` 与 `--time` 不能同时使用
- 该能力面向 one-shot 延迟/定时执行；cron 与长期工作流仍建议使用 `task` service

更详细的说明、恢复行为与 HTTP 入口示例，见：

- [`Service Action 定时执行`](/zh/docs/commands/service/scheduling)

## 说明

- `city service list` 和 `city service status` 是静态的 console 级 catalog 视图。
- 它们不依赖 `--path`、`--agent`，也不要求 agent 已运行。
- `city service start/stop/restart/command` 才是面向具体 agent 的高级定向命令。
- 日常运行态操作，更推荐直接使用具体命令根：
  - `city chat ...`
  - `city task ...`
  - `city memory ...`
- 如果你需要在 `city service start/stop/restart/command` 中显式指定目标，再传 `--agent <name>` 或 `--path <path>`。

典型流程：

```bash
city service list
city service status memory
city memory status
city memory restart
```
