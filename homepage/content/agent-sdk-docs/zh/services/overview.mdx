# Services 总览

在 Agent SDK 里，`services` 表示一组“长期存在的实例能力”。

典型写法：

```ts
import { Agent, ChatService } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  tools: {},
  services: [
    new ChatService({
      telegram: {
        botToken: process.env.TELEGRAM_BOT_TOKEN!,
      },
    }),
  ],
});
```

## 先记住一句话

- `tools` 更像“一次执行时可调用的函数能力”
- `services` 更像“agent 进程长期持有的能力实例”

## service 适合承载什么

service 特别适合承载这些东西：

- 渠道 bot 状态
- queue worker
- scheduler / cron engine
- shell session 池
- contact / inbox 这类长期状态
- 你自己的业务级长期运行对象

如果一段能力需要跨多个 session 长期存在，通常更适合放进 service，而不是放进 tool。

## service 和 session 的关系

session 是“执行一轮对话或任务”的对象。  
service 是“被 agent 长期持有”的对象。

也就是说：

- 一个 agent 可以有多个 session
- 这些 session 可以共享同一个 service 实例

这也是为什么 SDK service 应该优先使用实例字段保存状态，而不是模块级单例。

## 当前 SDK 公开到什么程度

当前 `@downcity/agent` 对外明确公开的 service 相关入口主要有：

- `BaseService`
- `ChatService`

其中：

- `BaseService` 用于实现 custom service
- `ChatService` 是当前最直接的 SDK 内建 service 用法

## 当前完整 runtime 内建了哪些 service

Downcity 当前内建的 service 注册表包含：

- `chat`
- `contact`
- `task`
- `memory`
- `shell`

但要注意：

- “runtime 内建存在”不等于“当前 SDK 已把这个类作为主要 public API 承诺给你”
- 在显式 SDK 注入场景里，今天最常直接使用的仍然是 `ChatService`

## service 什么时候会启动

这是一个非常重要的现实边界。

当前 SDK 中：

- `new Agent({ services: [...] })` 只做装配，不会在构造阶段启动 I/O
- `agent.http.start()` 与 `agent.rpc.start()` 会先调用 `ensureServicesStarted()`
- 也就是说，`service.lifecycle.start()` 当前最稳定的自动触发时机，是你把 agent 暴露为 HTTP / RPC 端点时

如果你的 custom service 没有后台运行态，只提供 actions 或实例方法，这通常不是问题。  
如果它依赖明确的启动流程，你需要按这个边界去设计。

## system prompt 注入

当前 SDK local session 会把显式注入 service 的 `service.system(context)` 文本注入到 session prompt 中。

SDK 当前会注入：

- `PROFILE.md`
- `SOUL.md`
- core prompt
- `service.system(context)`
- runtime clock prompt

所以 service 在 SDK 里的价值包括：

- 生命周期
- 长期状态
- 渠道与运行时能力
- 属于 service 自己的稳定 system 规则
