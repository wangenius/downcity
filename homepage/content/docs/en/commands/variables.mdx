# Command Environment Requirements

This page answers one question: what environment each `bay` command group needs to run correctly.

## 0) Top-level command boundary

Top-level command groups are:

- `bay console`
- `bay agent`
- `bay plugin`

Commands like `bay chat`, `bay task`, `bay memory`, `bay shell`, `bay skill`, `bay asr`, and `bay tts` are plugin commands, not top-level roots.

## 1) Environment by command group

| Command Group | Required Environment | Optional Environment | Notes |
| --- | --- | --- | --- |
| `bay init` | Writable `~/.downcity/` | None | Writes `~/.downcity/downcity.db` |
| `bay start/stop/restart/status` | Writable `~/.downcity/console/` | None | Manages global console process and registry |
| `bay console` | Console is running; browser can reach UI port | `--host`, `--port` | Single UI instance can switch across multiple running agents |
| `bay agent create` | Writable target directory | None | Generates `downcity.json`, `PROFILE.md`, `.downcity/` |
| `bay agent start` | Console is running; project has `downcity.json` + `PROFILE.md` | `--host` | Agent port is allocated by console and registered into console registry |
| `bay agent restart/status/doctor` | Target agent is registered in console registry | None | Operates through registry resolution |
| `bay config ...` | Valid `downcity.json` exists under target path | `--path` | Reads/writes project config; does not call agent API |
| `bay plugin list/status` | None | `--json` | Static plugin catalog; does not resolve any agent |
| `bay plugin start/stop/restart/command` | Target agent is running | `--agent`, `--path`, `--host`, `--port` | Used for managed plugins |
| `bay plugin action ...` | Project has `downcity.json` | `--path` | Advanced local execution form |
| `bay <managed-plugin> <action>` | Target agent is running | `--path`, `--host`, `--port` | Examples: `bay chat`, `bay task`, `bay memory`, `bay shell` |
| `bay <local-plugin> <action>` | Project has `downcity.json` | `--path` | Examples: `bay skill`, `bay web`, `bay asr`, `bay tts` |

## 2) Runtime transport

Default rule:

1. no `--host` / `--port`: use the local agent daemon HTTP endpoint resolved from the target project
2. `--host` or `--port` provided: override the endpoint and use HTTP with the given host/port

Bearer-token resolution matters only when the target HTTP endpoint currently requires auth.

## 3) `DC_AUTH_TOKEN`

For most local usage, you do not need to care about auth env vars.

Managed-runtime calls default to the local daemon HTTP endpoint. `DC_AUTH_TOKEN` matters only when that HTTP endpoint requires auth.

Priority order:

1. explicit `--token`
2. `DC_AUTH_TOKEN`

## 4) Context env requirements for common commands

### `bay chat ...`

- If `--chat-key` is omitted, it tries `DC_CTX_CHAT_KEY`
- If `--session-id` is omitted, it first tries `DC_SESSION_ID`

### `bay task ...`

- Task creation and run need `sessionId`
- Pass `--session-id` explicitly, or rely on `DC_SESSION_ID`

## 5) Recommended usage

Use plugin roots for day-to-day work:

```bash
bay chat send --chat-key ctx_xxx --text "hi"
bay task list
bay memory search "release decision"
bay plugin list
```
