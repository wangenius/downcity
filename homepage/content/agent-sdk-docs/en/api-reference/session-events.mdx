# Session Events

`session.subscribe()` currently emits these core event types:

- `turn-start`
- `text-delta`
- `reasoning-delta`
- `tool-call`
- `tool-result`
- `assistant-step`
- `operation`
- `turn-finish`
- `error`

## The most common one is `text-delta`

If you only need real-time text output, this is usually enough to handle first:

```ts
if (event.type === "text-delta") {
  process.stdout.write(event.text);
}
```

If you want a fuller experience, you will usually also handle:

- `turn-start`
- `turn-finish`
- `error`
- `assistant-step`
- `operation`
- `tool-call`
- `tool-result`

`assistant-step` is especially useful for transports such as chat bridges that want step-level visible reply chunks without owning Session history persistence themselves.

## `operation`

`operation` represents auxiliary Session work such as context compaction, model switching, and history forking. It is not assistant text.

Successful operation events are also persisted as `role: "operation"` messages in the Session JSONL history. In-progress or failed operations are only delivered through `subscribe()`, so history stays focused on completed state changes. Operation messages remain visible to UI history/timeline readers, but the Agent filters them out before building LLM input.

The built-in operation names currently emitted by Agent are:

- `compacting`
- `model-switching`
- `history-forking`

```ts
session.subscribe((event) => {
  if (event.type === "operation" && event.name === "compacting") {
    console.log(event.status, event.label);
  }
});
```
