# Command Environment Requirements

This page answers one question: what environment each `town` command group needs to run correctly.

## 0) Top-level command boundary

Top-level command groups are:

- `town console`
- `town agent`
- `town plugin`

Commands like `town chat`, `town task`, `town memory`, `town shell`, `town skill`, `town asr`, and `town tts` are plugin commands, not top-level roots.

## 1) Environment by command group

| Command Group | Required Environment | Optional Environment | Notes |
| --- | --- | --- | --- |
| `town init` | Writable `~/.downcity/` | None | Writes `~/.downcity/downcity.db` |
| `town start/stop/restart/status` | Writable `~/.downcity/console/` | None | Manages global console process and registry |
| `town console` | Console is running; browser can reach UI port | `--host`, `--port` | Single UI instance can switch across multiple running agents |
| `town agent create` | Writable target directory | None | Generates `downcity.json`, `PROFILE.md`, `.downcity/` |
| `town agent start` | Console is running; project has `downcity.json` + `PROFILE.md` | `--host` | Agent port is allocated by console and registered into console registry |
| `town agent restart/status/doctor` | Target agent is registered in console registry | None | Operates through registry resolution |
| `town config ...` | Valid `downcity.json` exists under target path | `--path` | Reads/writes project config; does not call agent API |
| `town plugin list/status` | None | `--json` | Static plugin catalog; does not resolve any agent |
| `town plugin start/stop/restart/command` | Target agent is running | `--agent`, `--path`, `--host`, `--port` | Used for managed plugins |
| `town plugin action ...` | Project has `downcity.json` | `--path` | Advanced local execution form |
| `town <managed-plugin> <action>` | Target agent is running | `--path`, `--host`, `--port` | Examples: `town chat`, `town task`, `town memory`, `town shell` |
| `town <local-plugin> <action>` | Project has `downcity.json` | `--path` | Examples: `town skill`, `town web`, `town asr`, `town tts` |

## 2) Runtime transport

Default rule:

1. no `--host` / `--port`: use the local agent daemon HTTP endpoint resolved from the target project
2. `--host` or `--port` provided: override the endpoint and use HTTP with the given host/port

Bearer-token resolution matters only when the target HTTP endpoint currently requires auth.

## 3) `DC_AUTH_TOKEN`

For most local usage, you do not need to care about auth env vars.

Managed-runtime calls default to the local daemon HTTP endpoint. `DC_AUTH_TOKEN` matters only when that HTTP endpoint requires auth.

Priority order:

1. explicit `--token`
2. `DC_AUTH_TOKEN`

## 4) Context env requirements for common commands

### `town chat ...`

- If `--chat-key` is omitted, it tries `DC_CTX_CHAT_KEY`
- If `--session-id` is omitted, it first tries `DC_SESSION_ID`

### `town task ...`

- Task creation and run need `sessionId`
- Pass `--session-id` explicitly, or rely on `DC_SESSION_ID`

## 5) Recommended usage

Use plugin roots for day-to-day work:

```bash
town chat send --chat-key ctx_xxx --text "hi"
town task list
town memory search "release decision"
town plugin list
```
