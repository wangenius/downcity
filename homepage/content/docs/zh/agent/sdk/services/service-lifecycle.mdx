# Service 生命周期

service 生命周期主要是：

- `start`
- `stop`
- `command`

它们都定义在 `lifecycle` 里。

## 一个最小例子

```ts
readonly lifecycle = {
  start: async (context) => {
    // 启动 worker、连接、监听器
  },
  stop: async () => {
    // 清理资源
  },
  command: async ({ context, command, payload }) => {
    // 处理额外命令
    return {
      success: true,
      message: "ok",
    };
  },
};
```

## `start`

适合做：

- 启动 worker
- 建立连接
- 准备长期缓存
- 启动渠道监听

## `stop`

适合做：

- 停掉 worker
- 关闭连接
- 清理定时器
- 回收 in-memory 资源

## `command`

它更像“service 级附加命令入口”。

例如：

- `reload`
- `reschedule`
- `flush`

这类命令不一定属于某一条普通 action，但属于 service 级控制面。

## 当前 SDK 什么时候会自动 `start`

这是最重要的一条现实规则。

当前 SDK 中，自动启动 service 的稳定时机主要是：

- `agent.http.start()`
- `agent.rpc.start()`

原因是这两个入口在真正暴露端点前，会先调用 `ensureServicesStarted()`。

## 当前 SDK 不要假设什么

不要默认假设：

- `new Agent(...)` 就会自动跑 `start`
- `agent.session()` 就会自动跑 `start`
- `session.run()` 就会自动把所有后台 service lifecycle 跑完

当前代码并没有对这几条路径做统一自动启动承诺。

## 当前 SDK 对 `stop` 的边界

当前 SDK 没有对外公开统一的：

- `agent.stopServices()`

这意味着你不应该把所有关键清理都押注在“SDK 之后一定会自动帮我 stop”这件事上。

更稳妥的建议是：

- `start` 做幂等
- `stop` 做幂等
- 资源管理尽量可重复调用

## 对 custom service 的设计建议

### 1. 没有后台运行态时，优先做轻量 service

如果只是一些长期状态和 action，完全可以不依赖 `start`。

### 2. 有后台运行态时，明确把 HTTP / RPC 启动当成 service 启动点

这样边界最清晰。

### 3. 不要把关键资源放到模块级单例里

否则 lifecycle 会和 agent 实例边界打架。
