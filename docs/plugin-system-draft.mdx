# 插件系统重构草案

## 1. 结论先行

当前 `extension` 设计不是最优。

问题不在“能不能工作”，而在于它把几类不同问题揉成了一层：

1. 业务能力
2. 生命周期
3. CLI 暴露
4. HTTP 暴露
5. 跨模块调用
6. 配置持久化

这导致调用链过长、抽象边界模糊、`voice` 被多处特判，整体上有“为了抽象而抽象”的倾向。

更合适的方向不是再造一套复杂的 `PluginManager + HookManager + MiddlewareManager + CapabilityManager` 大框架，而是采用一个更克制的模型：

1. `service` 继续负责显式入口和主业务
2. `plugin` 只负责“增强”和“插入时机”
3. 主动调用走 `capability`
4. 被动增强走 `hook`
5. 插件不再拥有独立 runtime 状态机

一句话：`service` 是主干，`plugin` 是挂件，不要让插件再长成第二套 service 系统。

## 2. 当前设计里哪些地方属于“脱裤子放屁”

### 2.1 插件自己维护独立 runtime 状态机

如果插件只是对运行时做增强，而不是独立对外提供常驻进程能力，那么单独的：

1. `start`
2. `stop`
3. `restart`
4. `status`

多数情况下都没有必要。

对 `voice` 来说，真正需要运行的是“转写能力”和“消息前增强逻辑”，而不是一个独立 runtime。

所以：

- `service` 需要 runtime 管理，因为它确实可能承载 channel、cron、queue
- `plugin` 通常不需要 runtime 管理，因为它只是挂到某些时机上执行

### 2.2 用插件名做业务调用

像下面这种：

```ts
context.extensions.invoke({
  extension: "voice",
  action: "transcribe",
  payload: { audioPath },
});
```

本质上让业务代码依赖了实现名，而不是能力名。

这会导致：

1. chat service 依赖 `voice`
2. 插件重命名会影响业务
3. 后续多实现竞争时无法替换

更合理的是：

```ts
const transcribe = runtime.capabilities.get("audio.transcribe");
const result = await transcribe({ audioPath });
```

### 2.3 把 CLI/API 作为插件核心模型的一部分

CLI 和 API 只是“管理入口”，不是插件的核心抽象。

`voice install`、`voice use`、`voice status` 这些命令本质上是：

1. 资源安装
2. 配置切换
3. 状态查看

它们应该是插件的附属管理面，而不是插件运行模型本身。

### 2.4 同时引入 hook、middleware、action、lifecycle 四套心智

如果一套系统里同时有：

1. action
2. hook
3. middleware
4. lifecycle

但每一套都只用到一点点，就容易变成学习成本远大于收益。

最小化后，应该只保留两种主语义：

1. `capability`：显式调用
2. `hook`：按时机增强

`middleware` 不单独成为一等概念，只作为部分 hook 的执行模式存在。

## 3. 吸收 service 的优点

当前 `service` 体系有三个优点值得保留：

### 3.1 运行时注入对象足够薄

`ServiceRuntime` 只暴露稳定端口：

1. `logger`
2. `config`
3. `env`
4. `context`
5. `invoke`

这类“薄 runtime”是好设计。

插件也应该吃同样风格的 runtime，而不是再重新定义一套很重的上下文对象。

### 3.2 显式入口清晰

`service` 的 action 适合承载显式命令和显式 API。

例如：

1. `chat send`
2. `task run`
3. `memory search`

这些都不是 hook 场景。

所以插件系统不要试图替代 service，而是只解决 service 不擅长的“时机增强”问题。

### 3.3 service 是业务所有者

谁处理聊天消息，谁就应该是 chat service。

插件最多只能说：

1. 我在消息进入前补充点东西
2. 我在消息发出后做点副作用
3. 我提供一个业务可调用的能力

插件不应该反客为主，变成业务编排中心。

## 4. 最小可行新模型

新的插件模型建议只保留五块：

1. `Plugin`
2. `HookRegistry`
3. `CapabilityRegistry`
4. `AssetRegistry`
5. `Asset`

这里最重要的原则是：

1. 不要 `setup()`
2. 不要 `admin`
3. 不要插件 runtime 状态机
4. 要把 `插件行为 / 资产实现 / 配置 / 可用性` 拆开

### 4.1 Plugin 定义

插件应该尽量像 `service` 一样，采用声明式对象，而不是执行式注册函数。

```ts
export interface Plugin {
  name: string;
  config?: PluginConfigDefinition<JsonValue>;
  actions?: PluginActions;
  hooks?: PluginHooks;
  capabilities?: PluginCapabilities;
  system?: (runtime: PluginRuntime) => string | Promise<string>;
  availability?: (
    runtime: PluginRuntime,
  ) => Promise<PluginAvailability> | PluginAvailability;
  requirements?: PluginRequirements;
}
```

这有几个好处：

1. 插件暴露什么，打开文件一眼就能看到
2. 不需要进 `setup()` 里找副作用
3. 更容易做静态注册、CLI 注册、API 注册
4. 更接近 `service.actions` 的心智模型

### 4.2 PluginRequirements 定义

插件不应该直接声明“模型文件、Python 包、命令行工具”这些实现细节。

插件应该只声明自己依赖哪些资产。

```ts
export interface PluginRequirements {
  assets?: string[];
}
```

也就是说：

1. 插件依赖 `asset name`
2. 插件不依赖资产内部结构
3. 插件更不应该直接知道 `modelId / repo / pip package` 这些实现细节

### 4.3 PluginRuntime 定义

插件 runtime 不要重新发明轮子，直接吸收 `ServiceRuntime` 的优点：

```ts
export interface PluginRuntime {
  cwd: string;
  rootPath: string;
  logger: Logger;
  config: ShipConfig;
  env: Record<string, string>;
  systems: string[];
  context: ServiceContext;
  services: ServiceInvokePort;
  capabilities: CapabilityRegistry;
  assets: AssetRegistry;
}
```

注意：

1. 不再保留 `extensions.invoke`
2. 插件之间不直接互调
3. 显式调用统一走 `capability`
4. 资产统一通过 `assets` 访问

## 5. Hook 设计

Hook 不宜过多，也不宜太抽象。

先只定义稳定、明确、有业务意义的几个时机：

```ts
export type HookName =
  | "runtime:boot"
  | "message:received"
  | "message:before-agent"
  | "response:before-send"
  | "response:sent"
  | "runtime:shutdown";
```

为了避免额外造一套 middleware 框架，hook 先只支持两种执行方式：

1. `emit`：通知型，不改值
2. `transform`：串行变换输入输出

```ts
export interface HookRegistry {
  on<T>(name: HookName, handler: PluginEventHook<T>): void;
  transform<T>(name: HookName, handler: PluginTransformHook<T>): void;
  emit<T>(name: HookName, value: T): Promise<void>;
  run<T>(name: HookName, value: T): Promise<T>;
}

export type PluginEventHook<T> = (
  params: { runtime: PluginRuntime; value: T; plugin: string },
) => Promise<void> | void;

export type PluginTransformHook<T> = (
  params: { runtime: PluginRuntime; value: T; plugin: string },
) => Promise<T> | T;
```

不要一开始就上 `next()` 风格 middleware。

原因很简单：

1. `transform` 已经够覆盖绝大多数消息增强场景
2. middleware 适合 HTTP / request pipeline
3. 当前插件更多是“加工消息”，不是“接管控制流”

## 6. Capability 设计

Capability 是插件最核心的主动调用方式。

```ts
export interface CapabilityRegistry {
  register<TInput, TOutput>(
    name: string,
    handler: PluginCapability<TInput, TOutput>,
  ): void;

  get<TInput, TOutput>(
    name: string,
  ): ((input: TInput) => Promise<TOutput>) | null;
}

export type PluginCapability<TInput, TOutput> = (
  params: { runtime: PluginRuntime; payload: TInput; plugin: string },
) => Promise<TOutput> | TOutput;
```

约束：

1. 调用方只依赖 capability 名
2. 不依赖插件名
3. 一个 capability 默认只允许一个实现
4. 未来若出现多实现竞争，再单独加 provider 优先级，不提前设计

## 7. Action 设计

插件也应该像 `service` 一样拥有显式 action。

这些 action 不是插件主模型，而是插件的显式管理/操作入口。

```ts
export type PluginActions = {
  [actionName: string]: PluginAction<JsonValue, JsonValue>;
};

export type PluginAction<
  P extends JsonValue = JsonValue,
  R extends JsonValue = JsonValue,
> = {
  command?: PluginActionCommand<P>;
  api?: PluginActionApi<P>;
  execute: (params: {
    runtime: PluginRuntime;
    payload: P;
    pluginName: string;
    actionName: string;
  }) => Promise<PluginActionResult<R>> | PluginActionResult<R>;
};
```

这样：

1. `voice status`
2. `voice install-model`
3. `voice use-model`
4. `voice doctor`

都可以保留，而且不需要再额外造一个 `admin` 概念。

## 8. Asset 应该是一等对象

你提的这个点是关键修正：

`voice` 不应该知道 `model`。

`model` 只是某种资产实现细节。

更准确地说：

1. `voice plugin` 关心“我能不能拿到一个可用的转写资产”
2. `asset` 关心“我到底用什么模型、怎么安装、怎么检查、怎么暴露句柄”

所以不能让插件里直接出现：

1. `installedModels`
2. `modelId`
3. `repo`
4. `python packages`

这些都应该收进 asset 内部。

### 8.1 Asset 对象定义

资产应该是第一类对象，而不是被动清单项。

```ts
export interface Asset<
  TConfig extends JsonValue = JsonValue,
  THandle = unknown,
  TInstallInput extends JsonValue = JsonValue,
> {
  name: string;
  scope: "global" | "project";
  config?: AssetConfigDefinition<TConfig>;
  check(context: AssetContext): Promise<AssetCheckResult> | AssetCheckResult;
  install(
    context: AssetContext,
    input?: TInstallInput,
  ): Promise<AssetInstallResult> | AssetInstallResult;
  resolve(
    context: AssetContext,
  ): Promise<THandle> | THandle;
}
```

### 8.2 AssetRegistry 定义

系统里统一管理 asset，而不是让每个插件自己手搓安装逻辑。

```ts
export interface AssetRegistry {
  register(asset: Asset): void;
  check(name: string): Promise<AssetCheckResult>;
  install<TInput extends JsonValue = JsonValue>(
    name: string,
    input?: TInput,
  ): Promise<AssetInstallResult>;
  use<THandle = unknown>(name: string): Promise<THandle>;
  getConfig<TConfig extends JsonValue = JsonValue>(
    name: string,
  ): Promise<TConfig | null>;
  setConfig<TConfig extends JsonValue = JsonValue>(
    name: string,
    value: Partial<TConfig>,
  ): Promise<TConfig>;
}
```

### 8.3 AssetContext 定义

```ts
export interface AssetContext {
  cwd: string;
  rootPath: string;
  logger: Logger;
  env: Record<string, string>;
  shipConfig: ShipConfig;
}
```

注意：

1. asset 不应该依赖 `plugin`
2. asset 是底层实现对象
3. plugin 只是消费 asset

## 9. Assets / Dependencies / Config / Availability 的边界

这是插件系统里最容易混乱的一层，必须单独拆开。

### 9.1 先区分四件事

1. 插件代码是否存在
2. 插件配置是否启用
3. 插件资产是否已安装
4. 插件当前是否可用

这四件事不能混成一个“installed/running”状态。

### 9.2 插件代码

对当前仓库来说，优先采用 `builtin plugin` 模型。

也就是：

1. 插件代码跟随 package 一起发布
2. 插件本身不需要单独安装代码
3. 启动时统一静态注册

所以 `voice plugin` 不是“安装一个插件包”，而是“启用一个内建插件”。

### 9.3 插件配置

配置决定的是：

1. 是否启用
2. 是否注入 prompt
3. 是否自动增强消息
4. 用户级行为参数是什么

```ts
export interface PluginConfigDefinition<T> {
  plugin: string;
  scope: "global" | "project";
  defaultValue: T;
}
```

关键点：

插件配置只描述“插件行为”。

不要让插件配置承担这些内容：

1. 模型 ID
2. 模型仓库
3. Python 路径
4. 依赖安装方式
5. runner command

这些都属于 asset 配置。

### 9.4 资产配置

资产配置决定的是“底层实现怎么工作”。

例如某个转写 asset 的配置可能包括：

1. provider
2. model repo
3. revision
4. python binary
5. command template
6. storage path

```ts
export interface AssetConfigDefinition<T> {
  asset: string;
  scope: "global" | "project";
  defaultValue: T;
}
```

### 9.5 插件依赖

依赖是资产中的一类，但语义上更偏“执行条件”。

例如：

1. 系统里是否有 `python3`
2. 某个 pip 包是否存在
3. 某个命令行二进制是否存在

建议实现上把它们放进 `assets` 检查体系里，而不是额外发明一套 `dependencies` 运行模型。

也就是说：

1. 对用户文档可以说“依赖”
2. 对系统抽象统一视为“asset check”

这样更简单。

### 9.6 插件可用性

插件不需要 runtime 状态，但需要可用性判断。

```ts
export interface PluginAvailability {
  enabled: boolean;
  available: boolean;
  reasons: string[];
  missingAssets?: string[];
}
```

对插件来说，可用性不应该自己去理解 model/python/pip 的细节。

插件应该只关心：

1. `enabled=false`
2. 依赖的 asset 不可用

asset 自己再返回更底层的原因。

## 10. 安装到底是什么意思

这里必须统一术语，不然后面实现会持续混乱。

### 10.1 不建议说“安装 voice plugin”

对当前 builtin plugin 体系，这句话是不准确的。

因为代码已经在仓库里了。

### 10.2 应该拆成三种动作

1. `enable plugin`

开启插件

1. `install assets`

安装插件依赖的模型/环境/资源

1. `check availability`

检查插件是否可用

所以 `voice` 的实际动作应该理解为：

1. 启用 `voice`
2. 安装 `voice` 依赖的转写 asset
3. 检查 `voice` 是否可用

## 11. voice 插件在新模型下的样子

### 11.1 voice 插件职责

在新模型下，`voice` 插件做四件事：

1. 提供 `audio.transcribe`
2. 在 `message:before-agent` 增强语音消息
3. 提供配置
4. 提供显式 action 管理启用、诊断、安装资产

`voice` 不做的事情：

1. 不直接理解 model 结构
2. 不直接维护下载逻辑
3. 不直接管理 python 依赖细节

这些都交给 asset。

### 11.2 voice asset 伪代码

```ts
export type AudioTranscriberHandle = {
  transcribe(input: {
    audioPath: string;
    language?: string;
  }): Promise<{
    success: boolean;
    text?: string;
    error?: string;
  }>;
};

export const voiceTranscriberAsset: Asset<
  {
    provider: "local" | "command";
    command?: string;
    modelSource?: string;
    installDir?: string;
    pythonBin?: string;
  },
  AudioTranscriberHandle,
  {
    force?: boolean;
  }
> = {
  name: "voice.transcriber",
  scope: "global",

  config: {
    asset: "voice.transcriber",
    scope: "global",
    defaultValue: {
      provider: "local",
      command: undefined,
      modelSource: undefined,
      installDir: undefined,
      pythonBin: undefined,
    },
  },

  async check(context) {
    const config = await loadAssetConfig(context, "voice.transcriber");
    const reasons: string[] = [];

    if (!hasRuntimeForProvider(config.provider, config.pythonBin)) {
      reasons.push("runtime missing");
    }

    if (!hasInstalledSource(config.provider, config.modelSource, config.installDir)) {
      reasons.push("asset payload missing");
    }

    if (!isProviderReady(config.provider, config)) {
      reasons.push("asset not ready");
    }

    return {
      available: reasons.length === 0,
      reasons,
    };
  },

  async install(context, input) {
    const config = await loadAssetConfig(context, "voice.transcriber");
    await installTranscriberPayload({
      context,
      config,
      force: input?.force === true,
    });
    return {
      success: true,
      message: "voice.transcriber installed",
    };
  },

  async resolve(context) {
    const config = await loadAssetConfig(context, "voice.transcriber");
    return {
      async transcribe(input) {
        return await runTranscriberWithResolvedConfig({
          context,
          config,
          audioPath: input.audioPath,
          language: input.language,
        });
      },
    };
  },
};
```

### 11.3 voice 插件伪代码

```ts
export const voicePlugin: Plugin = {
  name: "voice",

  config: {
    plugin: "voice",
    scope: "global",
    defaultValue: {
      enabled: false,
      injectPrompt: true,
      augmentMessage: true,
    },
  },

  requirements: {
    assets: ["voice.transcriber"],
  },

  async availability(runtime) {
    const config = runtime.config.plugins?.voice;
    const assetStatus = await runtime.assets.check("voice.transcriber");

    if (!config?.enabled) {
      return {
        enabled: false,
        available: false,
        reasons: ["voice plugin disabled"],
        missingAssets: ["voice.transcriber"],
      };
    }

    return {
      enabled: true,
      available: assetStatus.available === true,
      reasons: assetStatus.reasons || [],
      missingAssets:
        assetStatus.available === true ? [] : ["voice.transcriber"],
    };
  },

  capabilities: {
    "audio.transcribe": async ({ runtime, payload }) => {
      const status = await voicePlugin.availability!(runtime);
      if (!status.enabled || !status.available) {
        return {
          success: false,
          error: status.reasons.join("; "),
        };
      }

      const transcriber =
        await runtime.assets.use<AudioTranscriberHandle>("voice.transcriber");

      return await transcriber.transcribe({
        audioPath: payload.audioPath,
        language: payload.language,
      });
    },
  },

  hooks: {
    "message:before-agent": [
      async ({ runtime, value }) => {
        const transcribe = runtime.capabilities.get<
          { audioPath: string; language?: string },
          { success: boolean; text?: string; error?: string }
        >("audio.transcribe");

        if (!transcribe) return value;

        const voiceAttachments = value.attachments.filter((item) => {
          return item.type === "voice" || item.type === "audio";
        });
        if (voiceAttachments.length === 0) return value;

        const blocks: string[] = [];
        for (const item of voiceAttachments) {
          const result = await transcribe({
            audioPath: item.path,
          });
          if (!result.success || !result.text) continue;
          blocks.push(`【语音转写 ${item.type}: ${item.path}】\n${result.text}`);
        }

        if (blocks.length === 0) return value;

        return {
          ...value,
          promptAugments: [...value.promptAugments, blocks.join("\n\n")],
        };
      },
    ],
  },

  actions: {
    status: voiceStatusAction,
    doctor: voiceDoctorAction,
    install: voiceInstallAssetAction,
    configure: voiceConfigurePluginAction,
    off: voiceDisableAction,
    on: voiceEnableAction,
  },

  system(runtime) {
    const config = runtime.config.plugins?.voice;
    if (!config?.enabled || config.injectPrompt !== true) return "";
    return [
      "# Voice Plugin",
      "Audio attachments may be transcribed before agent execution.",
    ].join("\n");
  },
};
```

### 11.4 voice action 伪代码

```ts
export const voiceInstallAssetAction: PluginAction<
  { force?: boolean },
  JsonValue
> = {
  command: {
    description: "安装 voice 所依赖的转写资产",
    configure(command) {
      command.option("--force", "强制覆盖已存在资源");
    },
    mapInput({ opts }) {
      return {
        force: opts.force === true,
      };
    },
  },
  async execute({ runtime, payload }) {
    return await runtime.assets.install("voice.transcriber", payload);
  },
};
```

`voice` action 只是转发给 asset，不关心内部是 model、command 还是 python。

## 12. 调用路径应该有多简单

目标是让调用链回到最短路径。

### 11.1 被动增强场景

不要让 chat service 手动找 `voice` 插件。

而是：

```ts
const prepared = await hooks.run("message:before-agent", messageState);
```

插件自己在这个阶段完成增强。

### 11.2 主动调用场景

如果业务真的需要主动调用，就直接拿 capability：

```ts
const transcribe = runtime.capabilities.get("audio.transcribe");
if (transcribe) {
  const result = await transcribe({ audioPath });
}
```

而不是：

```ts
runtime.extensions.invoke({
  extension: "voice",
  action: "transcribe",
  payload: { audioPath },
});
```

## 13. 是否最优

如果“最优”指理论上最通用，那不是。

但如果“最优”指当前仓库阶段下：

1. 足够简单
2. 强解耦
3. 调用短
4. 资产边界清楚
5. 迁移成本可控

那我认为这版更优。

核心原因是它避免了三种过度设计：

1. 不把插件做成第二套 service
2. 不引入 `setup()` 执行式注册
3. 不把“安装插件代码”和“安装插件资产”混为一谈
4. 不让插件直接理解底层实现名词

它选择的是一个更克制的平衡：

1. 显式业务继续归 `service`
2. 运行时增强归 `plugin`
3. 主动能力调用走 `capability`
4. 被动插桩走 `hook`
5. 外部资源统一归 `asset`

## 14. 迁移建议

建议分四步迁移：

### 第一步

先引入：

1. `CapabilityRegistry`
2. `HookRegistry`
3. 新的声明式 `Plugin` 契约
4. `AssetRegistry`
5. `PluginAvailability`

但保留旧 `extension`。

### 第二步

把 `voice.transcribe` 改造成 `audio.transcribe` capability。

同时让 chat message pipeline 接入：

1. `message:before-agent`

### 第三步

把 `voice` 的模型、依赖、命令模板、诊断逻辑迁移进：

1. `voice.transcriber asset`
2. `asset config`
3. `asset check/install/resolve`

### 第四步

删除：

1. `extensions.invoke`
2. extension runtime state machine
3. extension command 主调度模型

## 15. 最终建议

新的插件系统不应该问：

“插件有没有启动？”

而应该问：

1. “插件是否启用？”
2. “插件是否可用？”
3. “插件提供哪些能力？”
4. “插件在哪些时机插入？”
5. “插件依赖哪些 asset？”
6. “asset 自己如何安装、检查、解析？”

当这几个问题拆开以后，系统会自然比当前 `extension` 模型更简单，也更不容易耦合失控。
