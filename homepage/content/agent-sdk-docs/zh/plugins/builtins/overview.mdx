# 内建 Plugin 总览

当前 Downcity 的内建 plugin 清单固定为：

- `auth`
- `skill`
- `web`
- `asr`
- `tts`
- `workboard`

它们来自同一份源码清单，并不是文档侧手工维护的概念列表。

## 在 SDK 里怎么接入内建 Plugin

这些 plugin 都可以直接从 `@downcity/agent` 导入，再显式挂到当前本地 `Agent` 上：

```ts
import {
  Agent,
  AuthPlugin,
  SkillPlugin,
  WebPlugin,
  AsrPlugin,
  TtsPlugin,
  WorkboardPlugin,
} from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  tools: {},
  plugins: [
    new AuthPlugin(),
    new SkillPlugin(),
    new WebPlugin(),
    new AsrPlugin(),
    new TtsPlugin(),
    new WorkboardPlugin(),
  ],
});

const installed = agent.plugins.list();
```

这里最重要的语义是：

- 内建 plugin class 已经公开导出
- SDK 不会默认自动注册全部内建 plugin
- 当前 `Agent` 真正可用哪些 plugin，取决于你是否显式注册

## 六个内建 Plugin 的角色分工

| Plugin | 主要职责 | 主要能力面 | 更像什么 |
| --- | --- | --- | --- |
| `auth` | chat 授权、观测、角色解析 | `guard` / `effect` / `resolve` / `actions` | 平台规则层 |
| `skill` | skill 发现、安装、读取 | `config` / `actions` / `system` | 能力目录层 |
| `web` | 联网 provider 选择与提示词适配 | `config` / `setup` / `usage` / `availability` / `actions` / `system` | 外部能力适配层 |
| `asr` | 语音转写与消息增强 | `config` / `setup` / `usage` / `availability` / `actions` / `pipeline` / `system` | 入站增强层 |
| `tts` | 文本转语音 | `config` / `setup` / `usage` / `availability` / `actions` / `system` | 显式生成能力层 |
| `workboard` | 运行时工作快照 | `availability` / `http.runtime` | 观测与控制面层 |

## 最推荐的理解顺序

### 1. 先看 `skill`

`skill` 是最容易建立直觉的 plugin。

它主要做四件事：

- 找技能
- 安装技能
- 列出已学会技能
- 读取某个技能的 `SKILL.md`

它能帮助你先理解“plugin 可以是一组显式 action + system prompt 注入”的最基础形态。

### 2. 再看 `web`

`web` 很适合理解完整 plugin 能力面。

它同时用到了：

- `config`
- `setup`
- `usage`
- `availability`
- `actions`
- `system`

所以它几乎覆盖了“一个插件如何被安装、配置、启用、切换 provider、注入 system prompt”的完整链路。

### 3. 然后看 `asr`

`asr` 让你看到另一种很重要的 plugin 形态：

- 一部分能力通过 action 显式调用
- 另一部分能力通过 pipeline hook 自动嵌进运行链路

如果你想设计“消息先增强，再交给 agent”的场景，它是最值得参考的内建实现。

### 4. 再看 `tts`

`tts` 和 `asr` 都属于语音能力，但设计取向不一样。

`tts` 主要是显式调用型插件：你明确要求它生成音频，然后它返回产物路径和文件标签。

### 5. 接着看 `workboard`

`workboard` 能帮助你理解：

- plugin 不一定主要靠 `runAction(...)` 使用
- plugin 也可以主要通过 `http.runtime` 暴露能力

它是当前最典型的 runtime HTTP 插件。

### 6. 最后回头看 `auth`

`auth` 很重要，但它不是最容易上手的参考样本。

因为它的角色不是“给用户一个新功能”，而是：

- 决定谁能进入 chat 链路
- 记录用户与会话观察态
- 为下游系统提供角色解析结果

它更偏平台规则，因此更适合在你已经理解 hooks 与 resolve 之后再看。

## 它们分别适合什么场景

### `auth`

适合你需要：

- 管理谁能给 agent 发消息
- 记录 Telegram / Feishu / QQ 的用户与群聊观测信息
- 根据用户角色控制下游能力

### `skill`

适合你需要：

- 把外部 skill 安装进本地技能目录
- 让 agent 先知道自己已经学会了哪些技能
- 在实际使用 skill 前读取 `SKILL.md`

### `web`

适合你需要：

- 在 `web-access` 和 `agent-browser` 之间切换
- 为当前项目统一配置联网策略
- 让 system prompt 自动带上联网约束

### `asr`

适合你需要：

- 处理语音附件
- 自动把语音变成文本后再交给 agent
- 单独转写一个本地音频文件

### `tts`

适合你需要：

- 把文字回复转成音频
- 在渠道里发送可听版本的输出
- 统一配置模型、格式和语速

### `workboard`

适合你需要：

- 给 Console 或 UI 暴露结构化工作快照
- 观察 agent 现在在做什么
- 把运行时状态做成一个受控的 HTTP 端点

## 这六个 Plugin 不是同一种形态

理解内建 plugin 时，最容易踩的坑是把它们都当成“只是 action 容器”。

其实它们分成三类会更好理解：

### 以 action 为主

- `skill`
- `tts`

### 以 action + 配置 + system 为主

- `web`

### 以运行链路增强为主

- `auth`
- `asr`

### 以 runtime HTTP 为主

- `workboard`
