# Project Configuration (downcity.json)

`downcity.json` is the project runtime entrypoint.

```json
{
  "name": "my-agent",
  "execution": {
    "type": "api",
    "modelId": "quality"
  },
  "start": {
    "host": "127.0.0.1",
    "port": 5314
  }
}
```

## Cautions

1. A project now must declare `execution`, and the mode is fixed to `api`.
2. `execution.modelId` must exist in the connected Federation AIService.
3. `start.host/start.port` affects default CLI routing.
4. Keep project-level config in `downcity.json`, including `plugins` and execution behavior.
5. Local shell / CLI commands now always run inside a sandbox. Safe Sandbox boundaries are generated by the runtime, not configured through `downcity.json`.

## Execution Modes

If you want to use a Federation AIService model:

```json
{
  "execution": {
    "type": "api",
    "modelId": "default"
  }
}
```

## Shell Sandbox

In the current version, `shell` / CLI commands run in the **Safe Sandbox** by default. Safe Sandbox boundaries are generated by the runtime and are not adjusted through `downcity.json`.

Current behavior:

- Affects both `@downcity/shell` command execution and `kind=script` task execution
- Current backends support macOS and Linux: macOS uses built-in `sandbox-exec`; Linux uses `bubblewrap` (`bwrap`)
- Safe Sandbox cannot be disabled and shell execution does not silently fall back to an unrestricted host process
- Host paths not mounted into the sandbox are not visible
- The project root and `.downcity/sandbox` are writable by default
- Sandbox child processes share `.downcity/sandbox` as their persistent `HOME`, with `.downcity/sandbox/tmp` for common temporary-directory variables and the zsh `TMPPREFIX` target, and `.downcity/sandbox/.cache` for `XDG_CACHE_HOME`
- Project-local installs such as `.venv`, `node_modules`, and downloaded model/cache files under the project are supported
- For host-level actions such as `brew install`, `npm install -g`, `pip install --user`, `gh auth login`, or host service setup, the agent may explicitly request `sandbox="unrestricted"` with a `reason`
- Unrestricted Sandbox is managed by Downcity Runtime; every execution requires user approval, and denied or expired requests do not run
- For unrestricted `shell_start`, approval only authorizes starting that process. Each later `shell_write` to the same unrestricted session also requires its own `reason` and user approval.
- `sudo`, Xcode tool installation, SSH/keychain/shell profile changes, and clearly destructive commands are rejected by default
- Safe Sandbox allows network access by default

On Linux, install `bubblewrap` first and make sure the system allows unprivileged user namespaces. If `bwrap` is missing or user namespaces are disabled, shell commands fail instead of falling back to an unrestricted host process.

## Where to edit it

Shell Safe Sandbox boundaries are not currently editable through project config. For host-level access, the agent must explicitly request `sandbox="unrestricted"` and wait for user approval.
