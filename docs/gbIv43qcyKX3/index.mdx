# ShipMyAgent 当前提示词注入逻辑（2026-03）

## 1. 总览

当前实现里，“提示词注入”是一个分层组装流程，不是单点拼接：

1. 启动时加载静态 prompts（`PROFILE.md` / `SOUL.md` / `USER.md` + 默认 core prompt）
2. 每次 run 时根据场景（`chat` / `task`）构建本轮 system messages
3. 动态收集 service 级 prompts（chat/skill/task + memory）
4. 执行模板变量替换
5. 在 `streamText` 调用时作为 `system` 传给模型

核心入口：`package/src/main/prompts/system/SystemDomain.ts`

---

## 2. 启动阶段：静态 Prompt 基座

### 2.1 静态文件来源

- `PROFILE.md`（必选，缺失时有 fallback）
- `SOUL.md`（可选）
- `USER.md`（可选）

加载逻辑：
- `package/src/main/prompts/PromptRuntime.ts`
- `package/src/main/server/env/Paths.ts`

`PromptRuntime.buildStaticSystems(...)` 会把静态文件文本与 `DEFAULT_SHIP_PROMPTS` 合并：

```text
[PROFILE, SOUL, USER, DEFAULT_SHIP_PROMPTS]
```

其中 `DEFAULT_SHIP_PROMPTS` 来自：
- `package/src/main/prompts/system/assets/core.prompt.txt`

### 2.2 热更新

运行中监听项目根目录文件变化（`PROFILE.md/SOUL.md/USER.md`），变更后自动重载 `runtime.systems`：

- `PromptRuntime.start()`
- `RuntimeState.startRuntimeHotReload()`

核心文件：
- `package/src/main/context/manager/RuntimeState.ts`
- `package/src/main/prompts/PromptRuntime.ts`

---

## 3. 每次请求时：System 解析主链路

### 3.1 调用链

1. `ContextManager.run(...)` 绑定 `RequestContext(contextId)`  
   文件：`package/src/main/context/manager/ContextManager.ts`
2. `Agent.prepareExecuteInput(...)` 调用 `this.system.resolve(...)`  
   文件：`package/src/main/agent/Agent.ts`
3. `PromptSystem.resolve(...)` 调用 `resolveAgentSystemMessages(...)`  
   文件：`package/src/main/prompts/system/PromptSystem.ts`
4. `SystemDomain` 完成 profile 判定 + prompts 组装  
   文件：`package/src/main/prompts/system/SystemDomain.ts`

### 3.2 profile 判定（chat / task）

`resolveSystemContextProfile(...)` 规则：

- `chat`：
  - `mode = chat`
  - 不替换 core prompt
  - 不禁用 service
- `task`：
  - `mode = task`
  - 用 `task.prompt.txt` 替换默认 core prompt
  - 禁用 `chat` service prompt 注入

相关文件：
- `package/src/main/prompts/system/assets/task.prompt.txt`
- `package/src/main/prompts/system/SystemDomain.ts`

---

## 4. 最终组装顺序（非常关键）

`buildAgentSystemMessages(...)` 的输出顺序固定为：

1. `staticSystemMessages`（PROFILE/SOUL/USER + core 或 task-core）
2. `serviceSystemMessages`（service 统一说明 + 各 service + memory）
3. `runtimeSystemMessages`（仅 task 模式有，放尾部）
4. `runtimeClockTail`（每轮动态时钟尾部，放最后）

即：

```text
static -> service -> runtime rules -> runtime clock tail
```

对应代码：
- `package/src/main/prompts/system/SystemDomain.ts`

---

## 5. runtime prompt 注入规则

`buildContextSystemPrompt(...)`：

- `chat` 模式：返回空字符串（不注入 runtime context 块）
- `task` 模式：注入
  - `Project root`
  - task 输出规则（禁止默认外发 chat 等）

另外每轮都会追加一个单独的动态时钟尾部（`Runtime clock`）system：
- Current time
- Timezone
- Location

验证测试：
- `package/test/core/context-system-prompt.test.mjs`

---

## 6. service prompt 注入规则

`loadServiceSystemPrompts(...)` 的拼接顺序：

1. `assets/service.prompt.txt`（主 service 规则）
2. 遍历 `SERVICES` 的 `service.system(...)` 输出
3. memory system（`buildMemorySystemText(...)`）

并且是 fail-open：某个 service/system 读取失败不阻断主流程。

核心文件：
- `package/src/main/prompts/system/SystemDomain.ts`
- `package/src/main/service/Services.ts`

### 6.1 chat service

`chat.system(context)` 会按 `services.chat.method` 选择：

- `cmd` -> `PROMPT.txt`
- `direct` -> `PROMPT.direct.txt`

并叠加已启用渠道（telegram/feishu/qq）的渠道 prompt。

文件：
- `package/src/services/chat/Index.ts`
- `package/src/services/chat/runtime/ChatMethod.ts`

### 6.2 skill service

`skill.system(context)` = `skills/PROMPT.txt` + 动态 skills 概览：

- 扫描本地可用 skills
- 生成 `# Skills System` 段落
- 当前实现不做 pinned skill 注入（`lookup` 为无状态）

文件：
- `package/src/services/skills/Index.ts`
- `package/src/services/skills/runtime/SystemProvider.ts`
- `package/src/services/skills/runtime/Prompt.ts`

### 6.3 task service

`task.system()` 恒定返回 `task/PROMPT.txt`。

文件：
- `package/src/services/task/Index.ts`

### 6.4 memory system

额外注入三段可选记忆：

- `.ship/profile/Primary.md`
- `.ship/profile/other.md`
- `.ship/context/<contextId>/memory/Primary.md`

文件：
- `package/src/services/memory/runtime/SystemProvider.ts`

---

## 7. 模板变量替换

静态和 service prompts 在转成 system message 时会经过**稳定变量替换**（`stable` 模式）：

- `{{project_path}}`
- `{{project_root}}`

`current_time/location/context_id/request_id` 不再作为前缀 prompt 的每轮动态替换项，避免破坏缓存前缀。
“当前时间/地点”由最后一条 `Runtime clock` system 提供。

链路：

1. `transformPromptsIntoSystemMessages(...)`
2. `replaceVariablesInPrompts(...)`
3. `resolvePromptGeoContext(...)`（IP 服务 + 30 分钟缓存，失败回退本地时区）

文件：
- `package/src/main/prompts/common/PromptRenderer.ts`
- `package/src/main/prompts/variables/VariableReplacer.ts`
- `package/src/main/prompts/variables/GeoContext.ts`

---

## 8. task 运行的特殊注入

task 执行不复用普通 chat agent，而是创建 task 专用 `PromptSystem(profile: "task")`：

- 使用独立 run contextId（`task-run:<taskId>:<timestamp>`）
- 自动套用 task profile 注入规则
- chat service prompt 在该模式下被禁用

文件：
- `package/src/services/task/runtime/Runner.ts`

---

## 9. 最终注入到模型的位置

组装完成的 `system` 在 `Agent.executePreparedRun(...)` 中传给：

- `streamText({ system, messages, tools, ... })`

文件：
- `package/src/main/agent/Agent.ts`

这一步就是实际“提示词注入到 LLM 请求”的落点。

---

## 10. 一句话总结

当前实现是“运行时分层注入”：

- 静态基座（PROFILE/SOUL/USER/core）由 `PromptRuntime` 维护
- 每轮按 `chat/task` profile 决定注入策略
- service（chat/skill/task）与 memory 动态追加
- 统一变量替换后在 `streamText` 统一下发给模型
