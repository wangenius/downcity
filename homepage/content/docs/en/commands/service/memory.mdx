# `city memory`

Memory service command group.

> Hierarchy note: `city memory` is a service action command, not a top-level command. Top-level entry is `city service`.

## Usage

```bash
city memory <subcommand> [options]
```

## Subcommands

- `status`
- `index`
- `search <query>`
- `get <memoryPath> [--from <line>] [--lines <count>]`
- `store --content <text> [--target <longterm|daily|working>] [--session-id <sessionId>]`
- `flush --session-id <sessionId> [--max-messages <count>]`

## Examples

```bash
city memory status
city memory index --force
city memory search "release decision"
city memory get .downcity/memory/MEMORY.md --from 10 --lines 20
city memory store --target longterm --content "User prefers concise release notes."
city memory flush --session-id telegram-chat-123 --max-messages 40
city memory flush --session-id telegram-chat-123 --time "2026-03-25T23:00:00+08:00"
```

## Notes

- Memory is enabled by default (zero config). Disable only when needed: `context.memory.enabled=false`.
- The default local session execution path already does automatic recall / capture; `city memory` is mainly for manual search, store, rebuild, and debugging.
- Current automatic capture rules:
- Every successful local turn writes `working` and `daily`
- If the user message clearly expresses a stable preference or long-lived rule (for example: "remember", "default", "from now on"), it also writes `longterm`
- Before writing `longterm`, the runtime checks `MEMORY.md` and skips duplicate entries
- Current `longterm` entries also go through a deterministic normalization step and are classified as `preference / fact / decision / rule`
- All memory actions also support common one-shot scheduling via `--delay` / `--time`.
- For detailed behavior, see [`Service Action Scheduling`](/en/docs/commands/service/scheduling).
- Start the agent first:

```bash
city agent start
```
