# ChatService

`ChatService` 是当前最适合直接从 `@downcity/agent` 使用的内建 service。

## 最常见用途

- 让 SDK 进程自己持有 Telegram / Feishu / QQ 渠道
- 把聊天渠道生命周期和你的应用代码绑定在一起
- 让本地 agent 对外暴露聊天入口，而不完全依赖项目级自动装配

## 最小示例

```ts
import { Agent, ChatService } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  tools: {},
  services: [
    new ChatService({
      telegram: {
        botToken: process.env.TELEGRAM_BOT_TOKEN!,
      },
    }),
  ],
});
```

## 它内部长期持有什么

当前 `ChatService` 实例内部长期持有的核心状态包括：

- `channelState`
- `queueWorker`
- `queueStore`

这也是它非常典型的 service 例子：  
它不只是一个“执行函数”，而是一组会持续存在的运行时对象。

## 你可以配置什么

`ChatServiceOptions` 当前主要包括：

- `queue`
- `telegram`
- `feishu`
- `qq`
- `channelAccounts`

### `queue`

用于覆盖 chat queue worker 的运行配置。

它适合做这些定制：

- 并发数
- 消息合并防抖
- 最大等待时间

如果你显式传了 `queue`，它的优先级高于 `downcity.json.services.chat.queue`。

### `telegram` / `feishu` / `qq`

这些是显式渠道配置。

例如 Telegram：

```ts
new ChatService({
  telegram: {
    botToken: process.env.TELEGRAM_BOT_TOKEN!,
    channelAccountId: "telegram-main",
    name: "telegram",
  },
});
```

显式凭据最适合这些场景：

- 你的应用自己管理密钥
- 你不想把这批凭据依赖回项目 `downcity.json`
- 你希望 SDK service 的能力完全由调用方控制

### `channelAccounts`

这是一个账户解析 provider。

它适合这些场景：

- 你的上层产品已经有自己的账号池
- 你想把凭据解析逻辑放在应用层
- 你不希望 ChatService 自己去依赖固定的全局存储实现

## 渠道账户解析优先级

从当前实现理解，优先级大致可以这样记：

1. 显式渠道凭据
2. `channelAccounts` provider
3. 项目配置中的 `services.chat.channels.*`

也就是说，如果你已经显式传入 `botToken` / `appId` / `appSecret`，通常会优先用这批显式配置。

## 它什么时候启动

当前 SDK 下，`ChatService.lifecycle.start()` 最稳定的自动触发点是：

- `agent.http.start()`
- `agent.rpc.start()`

这是因为 SDK 在暴露 HTTP / RPC 端点前，会先调用 `ensureServicesStarted()`。

## 一个很重要的现实边界

如果你只是：

- `new Agent(...)`
- `agent.session()`
- `session.run()`

那并不等于 ChatService 的后台生命周期已经自动完整跑起来。

所以如果你希望：

- bot 开始接收消息
- queue worker 明确进入运行态

更稳妥的方式是把 agent 暴露为 HTTP / RPC 服务，或者围绕这一点来设计你的应用启动流程。
