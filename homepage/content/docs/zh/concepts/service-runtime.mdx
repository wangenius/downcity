# Service Runtime

Service 是 Downcity 里的核心业务工作流模块。

典型内置 service：

- `chat`
- `task`
- `memory`
- `shell`

## Service 应该负责什么

- 用户主流程
- 运行时编排
- 上下文感知执行
- 稳定的 service actions
- 定义允许 plugin 接入的扩展点

## Service 不应该负责什么

- 底层依赖的安装、检查、切换
- plugin 私有实现细节
- 可以抽离出去的增强逻辑

## 推荐协作模型

- service -> 触发自己的 plugin 点
- plugin -> 实现这些点
- plugin -> 在内部处理自己的依赖

在运行时接口上，这三者也已经分开：

- `ServiceRuntime` 与 `PluginRuntime` 现在共享同一套执行上下文
- runtime 只负责注入上下文，不再对外暴露独立依赖端口
- plugin 依赖由 plugin 自己封装

是否真的执行某个 plugin，不由 service 临时判断，而由项目配置里的 `plugins.<name>.enabled` 决定。

- enabled = `true`：允许该 plugin 进入 hook / `plugin.system` 注入链路
- enabled = `false`：该 plugin 仍然可以被管理和查看状态，但不会参与主执行流程
- 未显式配置 `enabled`：回退到 plugin 自己声明的默认值

例如 `chat` 不需要知道具体转写模型细节。  
它只负责在处理语音附件时触发自己的语音相关扩展点，真正的实现由 `voice` plugin 提供，转写依赖与配置也由该 plugin 自己处理。

## 这样设计的好处

- 主流程不会散落到多个插件里
- 插件不会长成第二套 runtime
- 依赖细节不会泄漏进 service 编排层

对于命令执行，agent 现在也不再直接持有 shell 进程表，而是通过 `shellService` 管理 `shell_id` 会话。
