`@downcity/city` is the City itself. If you are deploying a real City that will serve one or more studios, this is the package everything rests on.

This is not a small helper around an AI SDK. It is the core server-side City where your AI services, custom services, official services, and shared `/v1/*` route surface come together.

The best reading order is:

1. Start with [City and bootstrapping](/en/docs/city/packages/city/city-runtime) to understand what a `City` instance actually owns.
2. Continue with [Service and Action design](/en/docs/city/packages/city/service-and-action) to see why AI services, custom services, and services all end up behind one route model.

## What this package actually owns

- creating the `City` instance
- registering `Service`, official services, and `AIService`
- studio, env, and token-related built-in capabilities
- the unified `/v1/*` route boundary

## What it does not own

- It does not own the studio-side SDK. That belongs to [@downcity/gate](/en/docs/city/packages/gate).
- It does not create the concrete database connection. Create a Drizzle ORM db and pass it to `new City({ db })`.
- It does not own ready-made auth, usage, or payment business logic. Those come from official services.

## When this should be your first stop

- you are building a real City deployment
- you need your own AI services or custom services
- you want shared auth, hooks, and model catalogs behind one backend

## Most common package combinations

| Goal | Package combination |
| --- | --- |
| Local or standard Node deployment | `@downcity/city` + Drizzle SQLite / Postgres |
| Studio frontend or app calls | `@downcity/city` + `@downcity/gate` |
| Accounts, usage, or payments | `@downcity/city` + the matching official service |

## Install

```bash
pnpm add @downcity/city drizzle-orm better-sqlite3
```

## Minimal example

```ts
import { City, AIService } from "@downcity/city";
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";

const sqlite = new Database("./data.sqlite");
sqlite.pragma("journal_mode = WAL");

const db = Object.assign(drizzle(sqlite), {
  $client: { exec: (sql: string) => sqlite.exec(sql) },
});

const base = new City({ db, dialect: "sqlite", raw: sqlite });

const ai = new AIService();

ai.use({
  id: "local-echo",
  name: "Local Echo",
  default: ["text"],
  actions: {
    text: async (ctx) => ({
      id: crypto.randomUUID(),
      role: "assistant",
      parts: [
        {
          type: "text",
          text: String(ctx.input.prompt ?? ""),
          state: "done",
        },
      ],
    }),
  },
});

base.use(ai);
```

This example shows the real role of `@downcity/city`: it is where the database, services, and services are mounted into one City. It does not decide how the frontend logs in or how studios call services.
