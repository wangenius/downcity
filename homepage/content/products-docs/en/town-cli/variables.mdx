# Command Environment Requirements

This page answers one question: what environment each `town` command group needs to run correctly.

## 0) Top-level command boundary

Top-level command groups are:

- `town start/stop/restart/status`
- `town agent`
- `town plugin`

Commands like `town chat`, `town task`, `town memory`, `town contact`, and `town skill` are plugin shortcut commands.

## 1) Environment by command group

| Command Group | Required Environment | Optional Environment | Notes |
| --- | --- | --- | --- |
| `town init` | Writable `~/.downcity/` | None | Writes `~/.downcity/downcity.db` |
| `town start/stop/restart/status` | Writable `~/.downcity/main/` | None | Manages Town runtime and managed agent registry |
| `town agent create` | Writable target directory | None | Generates `downcity.json`, `PROFILE.md`, `.downcity/` |
| `town agent start` | Town runtime is running; project has `downcity.json` + `PROFILE.md` | `--host` | Agent port is allocated by Town runtime and registered into the managed agent registry |
| `town agent restart/status/doctor` | Target agent is registered in the managed agent registry | None | Operates through registry resolution |
| `town config ...` | Valid `downcity.json` exists under target path | `--path` | Reads/writes project config; does not call agent API |
| `town plugin list/info` | None | `--json` | Static plugin catalog; does not resolve any agent |
| `town plugin command ...` | Target agent is running | `--agent`, `--path`, `--host`, `--port` | Used for managed plugins |
| `town plugin action ...` | Project has `downcity.json` | `--path` | Advanced local execution form |
| `town <managed-plugin> <action>` | Target agent is running | `--path`, `--host`, `--port` | Examples: `town chat`, `town task`, `town memory`, `town contact` |
| `town <local-plugin> <action>` | Project has `downcity.json` | `--path` | Examples: `town skill` |

## 2) Runtime transport

Default rule:

1. no `--host` / `--port`: resolve the target agent's local RPC endpoint from the registry
2. `--host` or `--port` provided: override the RPC endpoint and connect to that host/port

Bearer-token resolution matters only for Town Agent HTTP gateway calls whose endpoint requires auth.

## 3) `DC_AUTH_TOKEN`

For most local usage, you do not need to care about auth env vars.

Managed-runtime calls use the local Agent RPC endpoint or the Town Agent HTTP gateway, depending on the command. `DC_AUTH_TOKEN` only applies to HTTP gateway calls whose endpoint requires auth.

Priority order:

1. explicit `--token`
2. `DC_AUTH_TOKEN`

## 4) Context env requirements for common commands

### `town chat ...`

- If `--chat-key` is omitted, it tries `DC_CTX_CHAT_KEY`
- If `--session-id` is omitted, it first tries `DC_SESSION_ID`

### `town task ...`

- Task creation and run need `sessionId`
- Pass `--session-id` explicitly, or rely on `DC_SESSION_ID`

## 5) Recommended usage

Use plugin roots for day-to-day work:

```bash
town chat send --chat-key ctx_xxx --text "hi"
town task list
town memory search "release decision"
town plugin list
```
