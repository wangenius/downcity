如果你已经知道自己要看哪个 package，这一组文档会比场景指南更直接。

Downcity 的 package 设计很克制：先用 `@downcity/infra` 搭出 InfraRuntime，再选 `@downcity/conduit` 作为产品调用入口；数据库由你的运行环境创建 Drizzle db 后直接传给 InfraRuntime，最后按产品需要接官方服务。

这套顺序背后的意思是，Downcity 其实有 4 个清晰角色：

- `core` 是 Kernel，负责 InfraRuntime 本体
- `services` 是 Capabilities，负责官方能力
- `client` / `city terminal` 是 Interfaces，负责接入与操作
- `products/*` 是 Composition，负责把这些东西组装成可运行实例

如果你只从 npm package 角度理解，最值得先记住的是前 3 个：`core`、`services`、`client`。

## 推荐理解顺序

1. 先读 [@downcity/infra](/docs/city-infra/packages/infra)，理解 InfraRuntime 本体负责什么
2. 再读 [@downcity/conduit](/docs/city-infra/packages/conduit)，理解产品怎么调用它
3. 需要运维工具时读 [@downcity/city](/docs/city-infra/packages/city-terminal)
4. 需要账号、余额、usage 或 Stripe 支付能力时，再看 `@downcity/services`

## 如果你只想先搞清接入关系

建议把 package 层先理解成这条链：

```txt
@downcity/infra
  -> 提供 InfraRuntime Kernel
@downcity/services
  -> 把官方 capabilities 装进 InfraRuntime
@downcity/conduit
  -> 用 UserClient / AdminClient 调这些能力
```

也就是说，`@downcity/conduit` 不只是调 AI。

它统一负责：

- AI service 调用
- custom service 调用
- 官方 service 调用
- 管理侧 product / token / env 调用

## 一张图理解职责

```txt
Kernel        -> @downcity/infra
Capabilities  -> @downcity/services
Interfaces    -> @downcity/conduit + @downcity/city
Composition   -> blocks/node + blocks/edge
```

## 典型组合

### 最小 AI 产品

- `@downcity/infra`
- `@downcity/conduit`

### 要最小注册登录

- `@downcity/infra`
- `@downcity/conduit`
- `@downcity/services`

### 要 usage 计量

- `@downcity/infra`
- `@downcity/conduit`
- `@downcity/services`

### 要全局余额与充值单

- `@downcity/infra`
- `@downcity/conduit`
- `@downcity/services`

### 要支付权益

- `@downcity/infra`
- `@downcity/conduit`
- `@downcity/services`
