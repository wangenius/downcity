# 0) 这项目的“可控性设计”到底是什么

autoresearch 之所以看起来只有 630 行/很优雅，是因为它把研究自动化拆成了两层控制面：

- **人类控制面：`program.md`**

  你改的是“研究组织的制度/流程/评价标准/记录格式/保留策略”。

- **代理执行面：`train.py`**

  代理只改这一份代码：模型结构、优化器、超参、batch、训练 loop……都在这里。

同时有几个强约束保证它不会演化成无法评估的混乱系统：

- 不能改 `prepare.py`（评估、数据、tokenizer、时间预算等都被锁死）
- 不能加依赖（不能靠装包作弊）
- 固定 5 分钟 wall clock 训练预算（保证实验节奏与可比较）
- 统一 metric：`val_bpb`（越低越好）

# 1) `program.md`：Karpathy 其实给了一个“自动科研 SOP”

这份文件是整个项目最值得学的“制度化设计”。我按章节拆：

## 1.1 Setup：怎么开一个新的实验 run

它要求 agent 和用户先做几件非常工程化的事：

1) **约定 run tag**：例如 `mar5`  

2) **新建分支**：`autoresearch/<tag>`，必须是新 run（不能已存在）  

3) **读全量上下文**：只允许读三个文件（README/prepare/train）  

4) **确认数据缓存**：要求本地已有 `~/.cache/autoresearch/`（否则让人类先跑 `uv run prepare.py`）  

5) **初始化 results.tsv**：只写 header  

6) **确认后开始**（之后进入永不停止循环）

这套设计的含义：

- 用 git 分支把一次“通宵实验”封装成一个可回溯的研究产物
- 用 `results.tsv` 把评估结果结构化，便于第二天人类复盘
- 强制 agent 先跑 baseline，避免“没有参照系”的优化

## 1.2 Experimentation：允许改什么、不允许改什么

**CAN：只改 `train.py`**  

“Everything is fair game”：架构/优化器/超参/loop/batch/model size…

**CANNOT：**

- 不改 `prepare.py`（评估函数 `evaluate_bpb` 是 ground truth）
- 不加依赖（只能用 `pyproject.toml` 已有的）
- 不改评估 harness（禁止“指标投机”）

这就是“研究诚信”的规则：优化只能发生在被允许的变量上。

## 1.3 目标函数 + 软约束

- **主目标**：最低 `val_bpb`
- **软约束**：VRAM 不要炸（可接受小幅上涨换取显著收益）
- **复杂度惩罚（simplicity criterion）**：这是关键

  > 0.001 的提升如果带来 20 行 hack 可能不值；  

  > 0.001 的提升如果来自删代码，必留；  

  > 几乎不变但更简洁，也可留。

这条其实是在对抗 agent 的典型坏习惯：为了一点点指标堆屎山。

## 1.4 Output format：从日志中抽取关键指标

训练结束会打印摘要（例子里有）：

- `val_bpb`
- `peak_vram_mb`
- `mfu_percent`（模型 FLOPs 利用率）
- `total_tokens_M`、`num_steps`、`num_params_M`、`depth`

而且要求 agent 用 grep 提取，避免把整段 log 塞进上下文：

```bash
grep "^val_bpb:" run.log
```

## 1.5 results.tsv：结构化记录规范

TSV 字段：

`commit  val_bpb  memory_gb  status  description`

注意：

- **results.tsv 不要提交到 git**（保持 untracked）
- crash 用 `val_bpb=0.000000, memory=0.0, status=crash`
- description 用短句说明“这次试了什么”

这保证你第二天可以很快扫描 100 次实验，而不是读 100 篇小作文。

## 1.6 真正的“自动研究循环”（LOOP FOREVER）

这是核心算法（接近你截图文章里那句）：

1. 看当前分支/commit
2. 改 `train.py`
3. `git commit`
4. 跑：`uv run train.py > run.log 2>&1`（强制重定向）
5. `grep "^val_bpb:|^peak_vram_mb:" run.log`
6. 如果 grep 为空：crash → 看 `tail -n 50 run.log` 修几次，修不好放弃
7. 记录到 results.tsv（不提交）
8. **如果 val_bpb 变小：保留这个 commit（advance branch）**
9. **否则 reset 回起点（discard）**

加了两个“防失控”的政策：

- **Timeout**：超过 10 分钟 kill，算失败
- **NEVER STOP**：开始后别问人类要不要继续（默认人类睡了）

> 这一整套就是 Karpathy 给 agent 写的“科研流程操作系统”。

# 2) `train.py`：有哪些“可以动刀的杠杆”（按模块拆）

`train.py` 本身其实就是一个“小型 GPT 训练框架”，我按你最关心的可改点列。

## 2.1 模型配置（GPTConfig）

开头有 `GPTConfig`：

```py
sequence_len=2048
vocab_size=32768
n_layer=12
n_head=6
n_kv_head=6
n_embd=768
window_pattern="SSSL"
```

代理可以改这些（注意：`MAX_SEQ_LEN` 在 `prepare.py` 是锁死的，但模型内部 sequence_len 仍有配置意义，尤其是 rotary buffer、attention window 等）。

## 2.2 注意力：FlashAttention3 + windowed attention

它用了 `kernels` 来拿 FA3：

- Hopper (cap == (9,0)) 用 `varunneal/flash-attention-3`
- 否则用 `kernels-community/flash-attn3`

注意力 forward：

- Q/K 做 rotary
- Q/K 做 RMSNorm
- 调用 `fa3.flash_attn_func(... causal=True, window_size=window_size)`

并且支持 `window_pattern` 控制每层的 window size（S/L pattern）：

- L：长窗口（全长）
- S：短窗口（半长）

`window_pattern="SSSL"` 表示层间交替短短短长（大概是计算/效果折中）

**可研究方向：**

- window pattern 改成全 L / 全 S / 其他组合
- n_kv_head（MQA/GQA）对效果/速度的影响
- q/k 的 norm/rotary 的变体（但不要动评估）

## 2.3 Value Embedding / ResFormer 风格的 value residual

这里有一个非常具体的结构：部分层有 value embedding（`has_ve` 函数决定哪些层有）：

- `has_ve`：交替 + 最后一层必含
- 每层 attention 里如果 `ve` 存在，会通过一个 gate 混到 v 上：

```py
gate = 2 * sigmoid(ve_gate(x[..., :32]))
v = v + gate * ve
```

这是一块“非标准 GPT”的特色点。

**可研究方向：**

- ve 的开关策略（哪些层有）
- gate channels 从 32 调整
- gate 初值与尺度（当前 init=0 → sigmoid(0)=0.5 → *2=1.0 neutral）

## 2.4 MLP：ReLU^2（squared ReLU）

MLP 是：

```py
x = relu(x).square()
```

而不是 GeLU / SwiGLU。

这也是一个可换的杠杆点（但要注意复杂度惩罚）。

## 2.5 Residual mixing：resid_lambdas / x0_lambdas

模型里有两组 per-layer 可学习标量：

- `resid_lambdas` 初始化全 1
- `x0_lambdas` 初始化 0.1

每层：

```py
x = resid_lambda[i]*x + x0_lambda[i]*x0
```

这像是把输入 embedding 的“直通”混入每层，增强梯度/稳定性。

**可研究方向：**

- 初始化策略
- 是否需要约束（clamp / softplus）
- 是否改为更简单结构

## 2.6 Logits “softcap”

输出 logits 之后做了：

```py
softcap = 15
logits = softcap * tanh(logits / softcap)
```

这是一个稳定训练的 trick（限制 logit 幅度）。

**可研究方向：**

- softcap 的值
- 是否需要（但删除要谨慎，可能稳定性崩）

## 2.7 Optimizer：Muon + AdamW 的组合（重头戏）

`setup_optimizer` 很关键：

- embedding / lm_head / value_embeds / 标量 → **AdamW**
- transformer 里各种矩阵参数按 shape 分组 → **Muon**

并且 embedding lr 非常大（例如 embedding_lr=0.2），还会按 `1/sqrt(dmodel/768)` 缩放。

这说明这个训练配方强依赖“特定优化器组合 + 特定 lr 分配”。

**可研究方向（非常多）：**

- matrix_lr、embedding_lr、unembedding_lr 的比例
- weight_decay（默认 0）
- betas（默认 (0.8,0.95)）
- Muon 的 ns_steps、momentum、beta2
- 把某些矩阵从 muon 挪到 adamw（或反过来）

# 3) 你要怎么“用它”（不是跑 ML，而是学框架）

如果你的目标不是训练模型，而是把框架迁移到别的领域，你应该学的是：

- **只允许改一个文件/一个模块**（强约束减少搜索空间）
- **固定预算**（时间/成本固定才能自动比较）
- **统一指标**（可比性 > 指标是否完美）
- **结构化日志**（results.tsv）
- **自动保留/回滚**（branch forward / reset）
- **复杂度惩罚**（防止 agent 堆 hack）

这就是 autoresearch 的“最小科研自动化范式”。
