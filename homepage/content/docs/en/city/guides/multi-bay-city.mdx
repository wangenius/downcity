---
title: Many Bays, One City
description: How one City can serve multiple bays without mixing bay differences with shared infrastructure.
---

When many bays share one City, the goal is not “save one server.” The goal is to separate **shared infrastructure** from **bay-specific differences**.

What should be shared:

- provider env
- model directory
- service routing
- hooks, usage, and logs
- `user_token` verification

What should stay in each bay:

- pages and interaction
- the user system
- plan, order, and balance rules
- bay-specific brand, positioning, and pricing

## How to think about `bay_id`

`bay_id` is a bay boundary, not a model boundary.

It is useful for:

- identifying which bay sent the call
- deciding whether a bay is still `active`
- telling hooks where usage should be recorded
- separating one user’s activity across different bays

A common pattern is one stable `bay_id` per bay:

```ts
const bay = await admin.bays.create({
  name: "Chrome Extension",
});

const issued = await admin.bays.tokens.apply({
  bay_id: bay.bay_id,
  user_id: "user_123",
  metadata: {
    plan: "starter",
    channel: "extension",
  },
});
```

## What belongs in token metadata

`bay_id` says *which bay this is*. Token metadata is better for *the user’s business state inside that bay*.

Common fields:

- `plan`
- `organization_id`
- `channel`
- `feature_flag`
- `team_id`

That lets hooks differentiate behavior with `ctx.bay` and `ctx.user.metadata` instead of cloning City for each bay.

## Recommended layering

### Bay side

- keeps routes, UI, and business experience
- uses `User Gate`
- only holds `bay_id + user_token`

### Business backend

- owns login, orders, balance, subscriptions, and business data
- uses `Admin Gate` in a trusted environment
- issues `user_token` after a successful login

### City

- owns the model directory
- owns provider calls
- owns usage and hooks
- owns the reusable AI service layer

## When to create a new bay

Usually worth creating a new bay when:

- it is a separate bay or market
- it needs to be paused or activated independently
- it needs independent usage reporting or business strategy
- you need to separate web app, extension, client demo, and other bay surfaces

Usually not worth creating a new bay when:

- it is only another page in the same bay
- it is only a feature flag inside the same bay
- it is just another conversation/session for one user

Those are better kept inside your own business system or token metadata.
