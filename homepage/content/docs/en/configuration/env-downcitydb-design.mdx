# Env and downcity.db Design

Use one rule:

- shared secrets and shared infrastructure live in `~/.downcity/downcity.db`
- project behavior lives in `downcity.json`

## `downcity.db` owns

- `env_entries`
- `channel_accounts`
- `chat_auth_roles`
- `chat_auth_role_permissions`
- `chat_auth_channel_defaults`
- `chat_auth_user_roles`
- `model_providers`
- `models`
- other console-owned encrypted settings

## Project `downcity.json` owns

- `model.primary`
- `plugins.*`

## Why

- secrets should not be duplicated into project config
- plugin enablement and plugin-owned dependency config are project behavior
- console storage should not hide project-specific plugin state

## Runtime load order

1. the host may read Console-managed env from `downcity.db`
2. the host injects runtime-needed values into `process.env`
3. Agent reads host `process.env`
4. Agent reads project `.env`
5. optional `new Agent({ env })` overrides both
6. Agent reads project `downcity.json`
7. resolve `plugins.*`
8. resolve channel credentials, chat authorization rules, and model pool from `downcity.db`
