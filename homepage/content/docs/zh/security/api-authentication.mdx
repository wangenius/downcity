# API 认证

从统一账户 V1 开始，Downcity 提供一组基础认证接口，用来初始化唯一管理员、登录、修改管理员密码，以及管理访问 token。

当前已经落地的接口是：

- `POST /api/auth/bootstrap-admin`
- `POST /api/auth/login`
- `GET /api/auth/me`
- `POST /api/auth/password/update`
- `GET /api/auth/token/list`
- `POST /api/auth/token/create`
- `POST /api/auth/token/revoke`

## 第一次使用

如果当前 console 数据库里还没有任何用户，可以先初始化第一个管理员：

```bash
curl -X POST http://127.0.0.1:3000/api/auth/bootstrap-admin \
  -H 'Content-Type: application/json' \
  -d '{
    "username": "admin",
    "password": "pass-123456",
    "displayName": "Admin",
    "tokenName": "bootstrap"
  }'
```

成功后会返回：

- 用户信息
- 一次性明文 Bearer Token

注意：明文 token 只会在签发时返回一次，后续列表接口不会再返回。
如果你是通过 CLI 首次执行 `city start` / `city console start` 完成初始化，CLI 会提示你设置管理员密码；留空时默认使用 `downcity`。如果你是直接调用 `bootstrap-admin` API，则密码完全以你提交的 `password` 为准。

## 登录

管理员初始化完成后，后续通过登录获取新的 Bearer Token：

```bash
curl -X POST http://127.0.0.1:3000/api/auth/login \
  -H 'Content-Type: application/json' \
  -d '{
    "username": "admin",
    "password": "你在 bootstrap-admin 时设置的密码"
  }'
```

`tokenName` 对 API 调用方是可选字段；Console UI 会在内部固定成自己的默认值，不要求用户填写。

## 带 Token 调用接口

拿到 token 后，统一放进 `Authorization` 头：

```bash
curl http://127.0.0.1:3000/api/auth/me \
  -H 'Authorization: Bearer dc_xxx'
```

如果 token 有效，接口会返回当前用户、角色和权限。

## `DC_AUTH_TOKEN` 与 `DC_AGENT_TOKEN`

对于直接调用 API 的用户，只需要记住一条：

- 对外调用统一使用 Bearer Token，本质都落到 `Authorization: Bearer <token>`

但在 CLI/runtime 里，会看到两个环境变量：

| 变量 | 面向谁 | 作用 |
| --- | --- | --- |
| `DC_AUTH_TOKEN` | 用户、脚本、CI | 显式指定“这次 CLI/API 调用用哪个 token” |
| `DC_AGENT_TOKEN` | Downcity agent runtime | agent 内部再次调用 `city ...` 时自动传播身份 |

推荐理解方式：

- 如果你是人或外部脚本，优先用 `--token` 或 `DC_AUTH_TOKEN`
- 如果是 agent 内部二次调用 `city task`、`city chat`，通常由 runtime 自动注入 `DC_AGENT_TOKEN`

最终 CLI 解析顺序是：

1. `--token`
2. `DC_AUTH_TOKEN`
3. `DC_AGENT_TOKEN`
4. 本地 CLI 登录态

所以：

- `DC_AUTH_TOKEN` 是显式覆盖
- `DC_AGENT_TOKEN` 是内部传播
- 两者最终都会归一化为 Bearer Token 请求头

如果你看到 `Missing bearer token`：

- 不要先怀疑 `sessionId`
- 先检查当前进程有没有可用 token 来源
- 如果是在 agent/Bash/ACP 子进程里，再重点检查 `DC_AGENT_TOKEN` 是否被正确继承

## 管理 Token

查看当前用户的 token 列表：

```bash
curl http://127.0.0.1:3000/api/auth/token/list \
  -H 'Authorization: Bearer dc_xxx'
```

创建一个新的 token：

```bash
curl -X POST http://127.0.0.1:3000/api/auth/token/create \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "name": "chrome-extension"
  }'
```

吊销 token：

```bash
curl -X POST http://127.0.0.1:3000/api/auth/token/revoke \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "tokenId": "token_record_id"
  }'
```

## 修改管理员密码

当前模型只保留一个管理员账户；控制面中的用户名密码登录只对这个管理员开放。

```bash
curl -X POST http://127.0.0.1:3000/api/auth/password/update \
  -H 'Content-Type: application/json' \
  -H 'Authorization: Bearer dc_xxx' \
  -d '{
    "currentPassword": "old-pass",
    "nextPassword": "new-pass"
  }'
```

注意：

- 修改密码不会自动吊销已存在 token
- 其他接入方如果已经拿到 token，仍然继续通过 Bearer Token 访问
- 明文 token 依旧只会在签发当次返回一次，后续列表里只能看到摘要信息

## 当前边界

这次落地的是统一账户 V1 的认证主干：

- 已有统一用户表、角色表、权限表、token 表、审计日志表
- 已有 Bearer Token 签发与校验
- 已有单管理员密码更新接口
- 已有当前管理员 token 管理接口

后续版本会继续把更多控制面接口切到统一 Bearer 鉴权。
