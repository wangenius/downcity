# Metadata and Storage

SDK Sessions are saved under the project directory. Persistence exists to recover conversation and execution state, not for an application to write these files as its own database.

```text
<projectRoot>/.downcity/agents/<agentId>/sessions/<sessionId>/
└── messages/
    ├── messages.jsonl
    ├── assistant_message.json
    └── meta.json
```

`agentId` is part of the storage partition. Multiple Agents in one project can use the same `sessionId` without conflict, but do not treat `agentId` as a casual display label.

## `messages.jsonl`: completed history

Each line is a complete `SessionMessage` snapshot, not a Mutation and never an object with `variant`. User, Error, completed Assistant, and Action Messages are stored here.

Some Messages have multiple revisions, such as an Action changing from `running` to `completed`. The reader keeps the greatest revision for a `message_id` and then returns linear history ordered by `sequence`.

This is why applications should call `session.messages()` rather than parse JSONL themselves: the SDK already handles revisions, visibility, and pagination.

## `assistant_message.json`: streaming draft

Before it completes, a streaming Assistant does not append every Delta to history. Instead, it atomically replaces the current complete draft:

```text
text delta / reasoning delta / tool state update
-> update complete Assistant Message draft
-> publish live Mutation
```

On completion, the final Assistant Message is appended to `messages.jsonl` and the draft is removed. This lets every Delta reach the UI while keeping history from growing once per token.

Each Session has at most one streaming Assistant draft at a time. Tool calls live in that draft's `parts` and retain real `text -> tool -> text` order.

## Interrupted-process recovery

When a Session starts or reopens, the SDK checks for unfinished drafts:

- A streaming Assistant is finalized into a history Message with `status: "stopped"`.
- A still-`running` Action is finalized as `failed`.
- Existing text, reasoning, and tool Parts are retained and never reordered.

A recovered client should still load `messages()` before subscribing to future Mutations. Do not infer an interrupted tool's final result from local UI cache.

## `meta.json`

`meta.json` stores lightweight fields for lists and detail views, including:

- `sessionId`, `agentId`
- `title`
- `modelId`, `modelLabel`
- `createdAt`, `updatedAt`
- `timezone`

Session property changes such as model and title update metadata. A title change also publishes a `session/title` Mutation.

## Usage boundary

- Read state through `messages()`, `get_info()`, `system()`, and the Session collection API.
- Change state through `prompt()`, `set()`, `fork()`, and `archive()`.
- Do not manually edit, delete, or concurrently write these files. Doing so breaks revision handling, draft atomicity, and recovery.
- For troubleshooting, inspect files read-only, but treat SDK API results as the source of truth for an external UI.

The Session collection manages archived directories. Use `archive()` and `clean_archive()` instead of moving directories yourself.
