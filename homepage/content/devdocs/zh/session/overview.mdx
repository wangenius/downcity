# Session 总览

这页只回答一个问题：

- 当前 Downcity 里，什么才算一个 Session

## 先说结论

- `Session` 是以 `sessionId` 为标识的一段长期执行会话
- `Executor` 是单次 turn 的执行引擎
- `SessionRunScope` 是单次 run 的临时作用域
- `Context` 不是当前代码里的主对象名

一句话：

```text
Session 是长期会话容器，Executor 推进这一轮，SessionRunScope 只属于本轮执行。
```

## Session 的真实组成

当前代码里，一个 Session 主要由这些部分构成：

- `sessionId`
- `Session`
- `Executor`
- history store
- `messages.jsonl`
- `meta.json`
- archive / step / summary 等持久化数据

## 为什么主语义是 `sessionId`

当前实现里，真正稳定的长期标识就是：

- `sessionId`

会话读写、执行缓存、历史持久化、task run session，都是围绕 `sessionId` 运转。

## Session、Executor、RunScope 的关系

### Session

长期存在，跨多次 run。

它负责沉淀：

- 历史消息
- metadata
- archive
- 会话级配置和状态

### Executor

只负责推进当前这一轮。

它负责：

- 读取 system 与 history
- 装配 tools
- 执行模型 / tool loop
- 收敛 assistant 输出

### SessionRunScope

只存在于单次请求链路里。

它透传：

- 当前 `sessionId`
- 当前 `requestId`
- step callback
- 本轮执行过程中的运行期信号

## Session 和模型输入的区别

Session 不是“这次送给模型的完整文本”。

更准确地说：

- Session 持有长期历史与执行组件
- 本轮模型输入是 Executor 从 Session 历史里裁剪、压缩、拼装出来的结果

## 常见入口怎么复用 Session

### chat managed plugin

- 渠道消息进入 chat runtime
- 路由到某个 `sessionId`
- 追加消息后执行 `session.prompt`

### api

- 请求先归到某个 `sessionId`
- 再调用 session 执行链

### task managed plugin

- task 可以生成自己的 task run session
- 但底层仍然复用同一套 Session 执行内核

## 一句话定义

```text
Session 是以 sessionId 标识、由 Session 对象、Executor 与 history store 共同维护的长期执行会话。
```
