# Chat Service 使用说明

你可以通过 `chat` service 完成聊天通道发送与会话上下文读取。

## 可用 action
- `send`：发送文本到目标 `chatKey`。
- `context`：读取当前会话快照（可按 `chatKey` 或 `contextId` 查询）。
- `extract_text`：从 assistant 消息结构中提取用户可见文本。

## 使用规范
- 这是 Bash-first 聊天集成：用户可见内容必须通过执行 `sma chat send` 发送。
- 对所有携带 chatKey 的用户消息，必须在关键节点调用 `sma chat send` 回复，不能只在内部推理而不回发：
  - 起始确认：当任务不是立即可答（需要执行命令或等待）时，先发一条已开始处理。
  - 阻塞或失败：遇到权限不足、参数缺失、外部依赖失败时，立即告知卡点和需要用户提供的信息。
  - 最终结果：任务完成后必须发送最终结论；若本轮尚未成功回发，结束前必须补发一次简明结果。
- 涉及多个 `chatKey` 时，每个被操作的 `chatKey` 都要在关键节点收到回复；给非当前对话发送时必须显式使用 `--chat-key <contextId>`。
- 不要为了补充说明重复多次执行 `sma chat send`，避免刷屏。
- 优先使用 `send` 输出用户可见结果，正文应简洁、可直接发送。
- 调用 `send` 时必须保证 `chatKey` 可解析（显式传参或可从上下文推导）。
- 默认使用单引号包裹 `--text`，例如：`sma chat send --text '你好'`。
- 多行正文使用 heredoc 管道到 stdin，例如：
  `cat <<'EOF' | sma chat send --stdin --chat-key <chatKey>`
  `第一行`
  `第二行`
  `EOF`
- 通过 shell 执行 `sma chat send --text '...'` 时，必须转义特殊字符，尤其是反引号和单引号本身。
- `context` 只用于读取会话路由信息，不要伪造或覆盖真实平台信息。
- `extract_text` 仅做提取，不应改变原始消息语义。

## 路由与标识
- `channel` 表示平台接入渠道：`telegram` / `feishu` / `qq`（以及内部的 `api` / `cli` / `scheduler`）。
- `targetId` 是平台侧会话标识，不能跨平台通用。
- `contextId` 是系统内部唯一定位符（跨会话投递唯一 key），通常由 `channel + targetId (+ 线程信息)` 生成。
- 给当前对话回复可直接使用 `sma chat send --text '...'`（可不传 `--chat-key`，优先读取 `SMA_CTX_CONTEXT_ID`）。
- 给其他 context 发送时，使用 `sma chat send --chat-key <contextId> --text '...'`。

## Telegram 附件
- 在 Telegram 对话中可使用附件指令发送文件：
  - `@attach document <相对路径> | 可选说明`
  - `@attach photo <相对路径> | 可选说明`
  - `@attach voice <相对路径> | 可选说明`
  - `@attach audio <相对路径> | 可选说明`
- 附件路径必须位于项目目录内。
