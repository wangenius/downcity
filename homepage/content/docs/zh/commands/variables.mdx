# 命令环境要求总览

这页用于集中回答一个问题：每类 `city` 命令运行时，依赖哪些环境。

## 0) 一级命令边界

一级命令组有：

- `city env`
- `city console`
- `city agent`
- `city service`
- `city plugin`

像 `city chat`、`city task`、`city memory`、`city skill`、`city asr`、`city tts` 都是挂载在 service/plugin 体系下的动作命令，不是一级命令。

## 1) 各命令组所需环境

| 命令组 | 必需环境 | 可选环境 | 备注 |
| --- | --- | --- | --- |
| `city init` | 可写 `~/.downcity/` 目录 | 预先设置 `LLM_API_KEY`（可作为初始化默认值） | 会写 `~/.downcity/downcity.db`（加密存储敏感字段） |
| `city start/stop/restart/status` | 可写 `~/.downcity/console/` | 无 | 管理全局 console 进程与 registry |
| `city console` | console 已启动；浏览器可访问 UI 端口 | `--host`、`--port` | 单 UI 实例可切换查看多个运行中的 agent |
| `city agent create` | 目标目录可写 | 无 | 会生成 `downcity.json`、`PROFILE.md`、`.downcity/` |
| `city agent start` | console 已启动；项目含 `downcity.json` + `PROFILE.md` | `--host` | agent 端口始终由 console 自动分配，并登记到 console registry |
| `city agent restart/status/doctor` | 目标 agent 已在 console registry 注册 | 无 | 不是“任意路径直接操作”，先走 registry 解析 |
| `city config ...` | 目标路径下 `downcity.json` 存在且合法 | `--path` | 读写的是 agent 项目配置，不是 runtime API |
| `city config alias` | 可写 `~/.zshrc` 或 `~/.bashrc` | `--shell zsh|bash` | 仅写 alias，不依赖 agent |
| `city service list/status` | 无 | `--json` | 静态 console 级 service catalog，不解析任何 agent |
| `city service start/stop/restart/command` | 目标 agent 已运行；项目已在 registry；项目含 `downcity.json` + `PROFILE.md` | `--agent`、`--path`、`--host`、`--port` | 高级定向形式；日常更推荐 `city <service> ...` |
| `city plugin list/status` | 无 | `--json` | 静态 console 级 plugin catalog，不解析任何 agent |
| `city plugin action ...` | 项目含 `downcity.json` | `--agent`、`--path` | 高级定向形式；日常更推荐 `city <plugin> ...` |
| `city <service> <action>`（如 `city chat ...`） | 目标 agent 已运行 | `--path`、`--host`、`--port` | 默认通过本地 IPC 转发；显式远程时走 `/api/services/command` |
| `city <plugin> <action>`（如 `city asr ...`） | 项目含 `downcity.json` | `--path` | 直接在本地对目标项目执行 |

补充：

- `city service list/status` 和 `city plugin list/status` 是静态的 console catalog 读取。
- 本地 `city service start/stop/restart/command`、`city task`、`city chat` 等运行时命令默认不走 HTTP，而是直接走本地 IPC。
- 本地 `city plugin action` 和 `city <plugin> <action>` 会直接读取目标项目，并在当前 CLI 进程内执行。
- 只有你显式传了 `--host` 或 `--port`，这次调用才会被视为远程 HTTP 调用，并进入 Bearer Token 解析。
- 远程 HTTP 调用时，token 解析优先级：`--token` > `DC_AUTH_TOKEN` > `DC_AGENT_TOKEN` > 本地 CLI 登录态（保存在 console 加密存储中）。
- `sessionId` / `chatKey` 只是执行上下文，不是认证 token。

## 1.1) `DC_AUTH_TOKEN` 和 `DC_AGENT_TOKEN` 的区别

先说最重要的一条：

- 对大多数用户来说，本地运行 `city ...` 时通常不需要关心这两个变量

因为本地默认已经改成 IPC 调用，只有显式远程 HTTP 调用时，token 变量才会参与认证。

这两个变量都和 Bearer Token 有关，但职责不同。

| 变量 | 主要用途 | 谁来设置 | 典型场景 |
| --- | --- | --- | --- |
| `DC_AUTH_TOKEN` | 用户显式覆盖本次 CLI/API 调用身份 | 用户、脚本、CI | 你自己拿一个 token，手动指定本次 `city ...` 用哪个身份 |
| `DC_AGENT_TOKEN` | agent 运行时内部传播身份 | Downcity runtime 自动注入 | agent 内部显式发起远程 HTTP 调用 |

可以把它们理解成两层：

- `DC_AUTH_TOKEN` 是“用户覆盖层”
- `DC_AGENT_TOKEN` 是“agent 内部运行层”

当前推荐规则：

- 你手动在终端、CI、脚本里发起远程 HTTP 调用时，优先使用 `--token` 或 `DC_AUTH_TOKEN`
- 你在 agent 内部、Bash tool、ACP/Codex/Claude/Kimi 子进程里显式发起 HTTP 调用时，runtime 才会自动注入 `DC_AGENT_TOKEN`
- 不要把 `sessionId`、`chatKey` 当成 token；它们只负责“在哪个上下文里执行”，不负责“谁在调用”

### 为什么内部不用 `DC_AUTH_TOKEN`

因为 `DC_AUTH_TOKEN` 代表“外部显式覆盖”更清晰：

- 用户手动传 `--token` 或设置 `DC_AUTH_TOKEN` 时，应该永远优先
- agent 内部自动传播身份时，应该和用户显式覆盖区分开

所以当前实现里：

- `DC_AGENT_TOKEN` 是内部标准传播变量
- `DC_AUTH_TOKEN` 保留给用户手动覆盖
- CLI 真正发请求前，再按统一优先级解析成最终 Bearer Token

### 统一优先级

当一次 `city ...` 调用被明确视为远程 HTTP 调用时，最终 token 解析顺序是：

1. 显式 `--token`
2. `DC_AUTH_TOKEN`
3. `DC_AGENT_TOKEN`
4. 本机 CLI 当前 token 状态

这意味着：

- 你显式传了 `--token`，一定覆盖其他来源
- 你设置了 `DC_AUTH_TOKEN`，一定覆盖 agent 自动注入的 `DC_AGENT_TOKEN`
- 如果两者都没有，agent 内部远程 HTTP 调用会继续回退到 `DC_AGENT_TOKEN`
- 如果既没有显式 token，也不在 agent 内部，就回退到本机 CLI 当前 token 状态

### 常见误区

- 误区 1：`sessionId` 可以代替 token
  不是。`sessionId` 只决定输出回到哪个 chat/session。
- 误区 2：本地 `city task` / `city chat` 一定要先拿 token
  不是。默认本地调用已经走 IPC，不需要 Bearer Token。
- 误区 3：agent 内部必须手动 `export DC_AUTH_TOKEN=...`
  通常不需要。只有显式远程 HTTP 路径才需要 token，正常情况下 runtime 会自动注入 `DC_AGENT_TOKEN`。
- 误区 4：`DC_AGENT_TOKEN` 适合拿到外部脚本里长期复用
  不建议。对外集成请优先使用 CLI 显式签发的 Bearer Token，并通过 `DC_AUTH_TOKEN` 或 `--token` 传入。

### `Missing bearer token` 怎么看

如果你执行显式远程 HTTP 调用，例如：

- `city service list --host 127.0.0.1 --port 5314`
- `city task create --host 127.0.0.1 --port 5314 ...`

并收到：

- `Missing bearer token`

优先按这个顺序排查：

1. 你是否显式传了 `--token`
2. 当前 shell 是否设置了 `DC_AUTH_TOKEN`
3. 如果是在 agent/Bash tool/ACP 子进程里，当前进程是否继承了 `DC_AGENT_TOKEN`
4. 如果都没有，当前机器是否有可用的本机 CLI 当前 token 状态

如果是 agent 内部再次发起远程 HTTP 调用：

- 首先怀疑的是 `DC_AGENT_TOKEN` 注入链路
- 而不是 `sessionId` 或 task 参数本身

## 2) 常见命令的上下文变量要求

### `city chat ...`

- `--chat-key` 未传时，会尝试读取 `DC_CTX_CHAT_KEY`
- `--session-id` 未传时，会优先尝试读取 `DC_SESSION_ID`
- 在 agent shell 内执行时，这些 `DC_CTX_*` 通常由运行时自动注入

### `city task ...`

- 创建/运行任务需要 `sessionId`
- 可显式传 `--session-id`
- 未传时可优先回退读取 `DC_SESSION_ID`
- 本地默认走 IPC，不需要 Bearer Token
- 只有显式远程 HTTP 调用时，才会按 `--token` / `DC_AUTH_TOKEN` / `DC_AGENT_TOKEN` / 本机 CLI 当前 token 状态 的顺序解析 token

### chat channel 凭据

- 唯一方式：在 `downcity.json` 中绑定 `services.chat.channels.<channel>.channelAccountId`。
- 真实密钥（`botToken`/`appId`/`appSecret` 等）统一存放在 `~/.downcity/downcity.db` 的 `channel_accounts`。
- `downcity.json` 与 agent `.env` 不再承载这些凭据。

## 3) `city service` 目标 agent 解析顺序

`city service start/stop/restart/command`：

1. 显式 `--agent <name>`（最高优先级）
2. 若未传 `--agent` 且 `--path` 为默认 `.`，先尝试 `DC_AGENT_NAME`
3. 回退到 `--path` 或默认 `.`
4. 若默认 `.` 且存在 `DC_AGENT_PATH`，优先解析为该路径

最终仍要求：

- 目标路径已登记在 console registry
- 路径内存在 `downcity.json` 与 `PROFILE.md`

## 4) Runtime 连接方式与地址解析顺序（`host/port`）

默认规则：

1. 没传 `--host` / `--port`：走本地 IPC
2. 传了 `--host` 或 `--port`：走远程 HTTP

只有进入远程 HTTP 模式后，下面的地址解析顺序才生效。

调用 runtime API（service/plugin/action 命令）时，地址优先级：

1. CLI 显式参数：`--host` / `--port`
2. 环境变量：`DC_SERVER_HOST/PORT`
3. 环境变量：`DC_CTX_SERVER_HOST/PORT`
4. daemon meta 参数（`.downcity/.debug/downcity.daemon.json`）
5. 默认值：`127.0.0.1:5314`

补充规则：

- `0.0.0.0` 或 `::` 会被归一化为 `127.0.0.1`

## 5) 变量注入来源

### runtime 启动后注入

- `DC_AGENT_PATH`
- `DC_AGENT_NAME`
- `DC_SERVER_HOST`
- `DC_SERVER_PORT`

### agent shell 子进程注入

- `DC_SESSION_ID`
- `DC_CTX_REQUEST_ID`
- `DC_CTX_SERVER_HOST`
- `DC_CTX_SERVER_PORT`

## 6) 推荐实践

多 agent 并行场景，建议只在高级定向命令中显式指定目标：

```bash
city service restart task --agent my-agent
# 或
city service restart task --path /abs/path/to/agent
```

查看 catalog 时，不需要指定目标：

```bash
city service list
city plugin list
```
