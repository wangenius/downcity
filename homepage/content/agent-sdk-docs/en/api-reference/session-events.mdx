# Session Events

`session.subscribe()` currently emits these core event types:

- `turn-start`
- `text-delta`
- `reasoning-delta`
- `tool-call`
- `tool-result`
- `assistant-step`
- `action`
- `turn-finish`
- `error`

## The most common one is `text-delta`

If you only need real-time text output, this is usually enough to handle first:

```ts
if (event.type === "text-delta") {
  process.stdout.write(event.text);
}
```

If you want a fuller experience, you will usually also handle:

- `turn-start`
- `turn-finish`
- `error`
- `assistant-step`
- `action`
- `tool-call`
- `tool-result`

`assistant-step` is especially useful for transports such as chat bridges that want step-level visible reply chunks without owning Session history persistence themselves.

## `action`

`action` represents auxiliary Session work such as context compaction, model switching, and history forking. It is not assistant text.

The `title` of a model-switch action includes the previous and next model names directly, for example `Session model switched from GPT-5 to Claude Sonnet`, so a UI can display it without composing a separate label.

Action events are also persisted as `type: "action"` records in the Session JSONL file. The same action keeps one record and updates its `state` as it moves through `running`, `completed`, or `failed`. Action records remain visible to UI history/timeline readers, but the Agent filters them out before building LLM input.

The built-in auxiliary actions currently emitted by Agent are:

- `compacting`
- `model-switching`
- `history-forking`

```ts
session.subscribe((event) => {
  if (event.type === "action") {
    console.log(event.state, event.title, event.description);
  }
});
```
