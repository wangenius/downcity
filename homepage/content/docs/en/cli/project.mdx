# Project Structure

An Agent project is a directory that the CLI recognizes and can run. The project directory mainly stores prompts, environment variables, and runtime data; agent ID, model binding, startup options, and plugin options live in the CLI global DB.

## Directory layout

```text
my-project/
├── PROFILE.md         # agent role definition
├── SOUL.md            # long-term principles
├── .env               # environment variables
├── .downcity/         # runtime data
│   ├── sessions/      # persisted conversation history
│   ├── logs/          # runtime logs
│   └── cache/         # plugin cache and temp files
└── src/               # your code (optional)
```

## Key files

### PROFILE.md

Defines the agent's visible role. This is included in the system context for every session. See [Profile](/en/docs/cli/profile).

### SOUL.md

Defines deeper, more stable operating principles. Also included in the system context. See [Profile](/en/docs/cli/profile).

### .env

Project-specific environment variables. These overlay the host environment. The Agent SDK reads this file automatically.

### .downcity/

The runtime data directory. Everything here is managed by the Agent SDK and plugins. You should not edit files inside it directly, but you may inspect logs and sessions for debugging.

## Multiple projects

You can have multiple projects on the same machine. Each project is independent: its own global DB config, its own sessions, its own runtime. The CLI operates on the project in the current working directory.

## Where Config Lives

`city agent create` writes agent config to the CLI global DB, not to a project-local `downcity.json`. Existing `downcity.json` files can still be read once as a legacy migration source; new writes go to the global DB.
