If you already know which package you want to inspect, this section is the fastest route.

Downcity keeps the package graph intentionally small: start with `@downcity/city` for the City itself and its City access layer, create a Drizzle db in your host runtime, and add official services only when the town needs them.

That order maps to four clear roles:

- `city` is the Kernel for the City itself
- `services` are Capabilities that add official behavior
- City and `cli` are Interfaces for town and operator entry
- `cities/*` is the Composition layer that assembles runnable examples

If you only care about npm packages, the two most important pieces to remember first are `city` and `services`.

## Recommended order

1. Start with [@downcity/city](/en/docs/city/packages/city) to understand the City itself
2. Then read [City](/en/docs/city/packages/city/city) to see how towns call it
3. If you want terminal tooling, read [City CLI](/en/docs/city/packages/cli)
4. Add `@downcity/services` when your town needs auth, wallet, usage, or Stripe payment flows

## If you only want the integration picture first

Read the package layer as this chain:

```txt
@downcity/city
  -> provides the City Kernel
@downcity/services
  -> adds official capabilities into City
City from @downcity/city
  -> uses User City and Admin City to call those capabilities
```

So City is not only for AI calls.

It also unifies:

- AI service calls
- custom service calls
- official service calls
- trusted-side town, token, and env management calls

## One responsibility map

```txt
Kernel        -> @downcity/city
Capabilities  -> @downcity/services
Interfaces    -> @downcity/city + downcity
Composition   -> cities/node + cities/edge
```
