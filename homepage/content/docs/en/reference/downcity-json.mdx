# `downcity.json` Reference

Project `downcity.json` is the local agent contract. It declares the agent identity, execution binding, and plugin configuration. Shared model, service details, and Shell Safe Sandbox boundaries live outside the project file.

## Agent project minimal shape

```json
{
  "$schema": "./.downcity/schema/downcity.schema.json",
  "name": "my-agent",
  "version": "1.0.0",
  "execution": {
    "type": "api",
    "modelId": "quality"
  },
  "plugins": {}
}
```

`execution.type` is fixed to `api`, and `execution.modelId` must reference a model exposed by the connected City AIService.

Project `downcity.json` keeps plugin configuration under `plugins.*`.

## Shared model binding

The project only stores the selected model ID:

Use City-side administration to choose the model ID, then keep the project binding in `downcity.json`.

Concrete workflows live in [Town City docs](/en/products-docs/town-cli/city), [Town config docs](/en/products-docs/town-cli/config), and [Agent restart docs](/en/products-docs/town-cli/agent/restart).

## Agent startup resolution

When an agent starts:

1. Check whether the project declares `execution`
2. If `execution.type = "api"`, resolve the selected model ID through the configured City AIService
3. Inject the resolved model into the agent session runtime

`execution` must be valid, otherwise startup fails fast.

## Plugin scope

Plugin behavior can be configured in project `downcity.json` under `plugins.*`. Not every plugin reads runtime options from this file. For example, capability plugins such as `asr` receive their real transcription function through the SDK constructor.

Example chat channel binding:

```json
{
  "plugins": {
    "chat": {
      "channels": {
        "telegram": {
          "enabled": true,
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

## Chat channel binding

Chat channels in `downcity.json` should bind to a global channel account id:

```json
{
  "plugins": {
    "chat": {
      "channels": {
        "telegram": {
          "enabled": true,
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

`channelAccountId` resolves to `channel_accounts.id` in `~/.downcity/downcity.db`.

## Sandbox

`downcity.json` does not currently expose a sandbox configuration block. Shell Safe Sandbox boundaries are generated by the runtime and are not adjusted through project config.

Current boundary:

- Current backends support macOS and Linux: macOS uses `sandbox-exec`; Linux uses `bubblewrap` (`bwrap`)
- Shell commands are required to go through sandbox and do not fall back to an unrestricted host process
- `kind=script` task bodies also run through the same sandbox backend
- Host paths not mounted into the sandbox are not visible
- The agent sandbox persists at `.downcity/sandbox/`
- Sandbox child processes use `.downcity/sandbox/` as `HOME`, `.downcity/sandbox/tmp` as common temporary-directory variables and the zsh `TMPPREFIX` target, and `.downcity/sandbox/.cache` as `XDG_CACHE_HOME`
- Project files, project-local `.venv`, `node_modules`, and downloaded model/cache files under the project are writable by default
- Real user home and system directories such as `/usr/local`, `/opt/homebrew`, and `/System` are not writable from sandbox commands
- Linux requires `bubblewrap` to be installed and unprivileged user namespaces to be enabled
