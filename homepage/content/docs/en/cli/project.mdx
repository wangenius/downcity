# Project Structure

Agent management config lives in one global database. A project directory only contains project env, Skills, and runtime data; it does not keep a second config copy.

## Directory layout

```text
my-project/
├── .env                 # project environment variables
├── .agents/
│   └── skills/          # user-managed, versioned Skills
├── .downcity/           # Agent and Plugin runtime data
│   ├── agents/          # sessions and messages
│   ├── chat/            # Chat Access, routing, and history
│   ├── logs/            # runtime logs
│   ├── memory/          # memory data
│   └── task/            # task runtime data
└── src/                 # application code (optional)
```

## Global config

The default database is `~/.downcity/downcity.db`. With `DC_PLATFORM_ROOT`, it becomes `$DC_PLATFORM_ROOT/downcity.db`. It stores:

- Agent IDs, project paths, and startup options
- Model bindings and Plugin config
- Chat channel accounts and Agent channel bindings
- City sessions, global env, and other platform management state

The CLI, interactive manager, and Agent daemon all read this database. There is no project config copy.

## Project assets

Project `.env` overrides platform global env and the current CLI process env, and is added to `.gitignore` by default. `.agents/skills` is the only user-managed Agent capability asset intended for version control.

## Runtime data

`.downcity/` only contains sessions, Chat Access, tasks, memory, logs, and caches. Management config does not belong there.
