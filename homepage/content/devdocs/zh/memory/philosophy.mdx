# Memory 设计哲学

这一页不讲模块名，先讲思想。

因为 Memory 最容易出问题的地方，不是代码写不出来，而是方向一开始就歪了。

说明：

- 本文中的 `History` 依然指“历史层/原始记录层”
- 它不是专指 chat service 的某一个 `history.jsonl`
- 具体定义见 [History 总览](/zh/devdocs/history/overview)

## 先给一句总纲

```text
Memory 的本质不是“多存一点信息”，而是“治理长期状态，让系统在时间里越来越稳”。
```

## Memory 真正要解决的，不是存储，而是四个判断

一套成熟的 Memory，核心其实不是数据库选型，而是下面四个判断：

1. 什么值得留下
2. 什么值得升级
3. 什么该在这一轮被召回
4. 什么该退出活跃状态

如果这四个判断做不好，再多的存储都只是在堆材料。

## 第一原则：Memory 是状态治理，不是内容堆积

最容易把 Memory 做坏的方法，就是把它理解成：

- 多存
- 多搜
- 多总结

这样做的结果通常不是系统更聪明，而是系统更嘈杂。

Downcity 更应该把 Memory 理解成：

- 一套长期状态治理系统

所以它最重要的问题不是：

- 怎么把东西塞进去

而是：

- 什么该留下
- 什么该合并
- 什么该覆盖
- 什么该退场

## 第二原则：当前执行和长期整理必须分开

这是整个设计里最重要的结构原则。

### 当前执行的要求

- 快
- 轻
- 不要被重整理拖慢

### 长期整理的要求

- 慢
- 稳
- 允许做归纳、合并、改写、清理

所以最合理的设计是：

- 热路径只做轻量读写
- 冷路径才做重整理

也就是大家最容易记住的那句话：

```text
在线先做事，离线再整理。
```

## 第三原则：History 不等于 Memory

这是所有 Memory 设计里最容易混掉的一层。

### `History`

是：

- 所有发生过的内容
- 原始消息
- 原始事件
- 原始任务结果

### `Memory`

是：

- 被筛过的
- 被组织过的
- 以后还会继续依赖的状态

这意味着：

- 不是所有历史都值得留下
- 不是所有留下的东西都值得进入长期正典

## 第四原则：记住和忘掉同样重要

很多系统只讲“记住”，不讲“忘掉”。

但真正好的 Memory，一定同时会做两件事：

- 保留重要的
- 淘汰过时的

遗忘不是 bug。

遗忘是质量控制。

如果没有 Forget，Memory 一定会逐渐出现这些问题：

- 噪音积累
- 旧事实污染新决策
- 临时结论长期滞留
- recall 结果越来越不准

## 第五原则：Memory 改变的不是“谁来思考”，而是“模型看到了什么”

这是一个非常关键的设计哲学。

Memory 不应该替代主执行 Agent 思考。

Memory 真正改变的是：

- 当前模型窗口里被放进了什么信息
- 系统以后默认会参考什么状态

也就是说，Memory 的力量主要来自：

- 改变上下文分布
- 改变默认前提
- 改变后续决策质量

而不是：

- 自己变成另一个全能执行体

## 第六原则：文件适合做 Projection，不适合永远做唯一事实源

Markdown 文件很好，因为：

- 人能读
- 人能改
- 能审计
- 易导出

但文件也天然有局限：

- 更新语义弱
- 状态关系弱
- 覆盖和失效关系难表达
- 越写越容易变成“日志堆”

所以更稳妥的哲学应该是：

- 文件继续保留
- 但它们逐渐退到 projection 层

也就是：

- `working.md` 更像 `Scratchpad` 的视图
- `daily/*.md` 更像 `Journal` 的视图
- `MEMORY.md` 更像 `Canon` 的视图

## 第七原则：LLM 是整理器，不是 Memory 本体

LLM 可以帮 Memory 做很多事，比如：

- 聚合相近事件
- 改写长期表述
- 识别冲突内容
- 生成更干净的 Canon 文本

但它不应该成为默认中心。

更稳妥的顺序应该是：

1. 先有清楚的状态模型
2. 再有清楚的动作协议
3. 最后才把 LLM 放进最适合它的位置

一句话：

```text
先设计 Memory 的骨架，再决定哪里需要模型的血肉。
```

## Downcity Memory 的核心逻辑

如果把整个 Memory 压到最少，本质上就是下面 6 个动作。

| 动作 | 直觉解释 | 主要发生在哪 |
| --- | --- | --- |
| `Capture` | 收进来 | `Scratchpad` / `Journal` |
| `Curate` | 理一理 | `Journal` |
| `Consolidate` | 定版本 | `Canon` |
| `Recall` | 拿回来用 | `Scratchpad` / `Canon` 优先 |
| `Forget` | 让旧的退下去 | `Journal` / `Canon` |
| `Project` | 映射到文件视图 | `working` / `daily` / `longterm` |

## 如果换成最通俗的话，就是六个字

```text
收 -> 理 -> 定 -> 用 -> 退 -> 映
```

### 收

把今天发生的内容收进系统。

### 理

把零散事件按主题、重要性、稳定性整理。

### 定

把真正长期有效的内容固化成当前有效版本。

### 用

在需要的时候拿回来，而不是一直塞进模型。

### 退

让失效、重复、低价值内容退出活跃状态。

### 映

把内部状态重新投影成可读文件和可检索索引。

## 三层状态为什么刚好够用

Downcity 我依然建议坚持只保留三个核心状态：

- `Scratchpad`
- `Journal`
- `Canon`

原因很简单：

### `Scratchpad`

解决“现在在处理什么”。

### `Journal`

解决“到底发生过什么”。

### `Canon`

解决“以后默认按什么来”。

这三个问题已经覆盖了大部分 Context Memory 的本质。

再加太多层，很容易把系统做得精致但不有力。

## 一套性感的 Memory，应该有怎样的感觉

我认为性感的 Memory，不是功能很多，而是“轻轻一拨，整个系统都顺了”。

它应该有这种感觉：

- Session 不再反复重复旧判断
- 当前上下文明显更短，但更有信息密度
- 系统不会把旧结论死抱住不放
- Memory 在后台静悄悄工作，而不是占满主流程

真正的一两拨千斤，不是让系统“记更多”，而是让系统：

- 更少遗忘真正重要的东西
- 更快拿到真正相关的东西
- 更自然地淘汰已经过时的东西

## 最后要守住的底线

无论后面实现细节怎么变，这几条底线最好不要丢：

- 不要把 Memory 做成聊天记录镜像
- 不要把热路径做成总结流水线
- 不要把一切判断都交给模型临场发挥
- 不要只会记，不会退
- 不要让文档设计脱离 package 的真实结构

## 最后一行

```text
Downcity 的 Memory 哲学，不是“尽量记住一切”，而是“把真正重要的状态慢慢整理出来，并在正确的时候拿来用”。
```
