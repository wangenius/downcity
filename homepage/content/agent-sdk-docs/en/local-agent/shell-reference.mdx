# Shell & Sandbox

The Agent uses `@downcity/shell` for local command execution, sandboxed runners, shell session management, and interactive approval flows.

## Install

Shell is a peer dependency of `@downcity/agent`. When you install the agent, you can add the shell package:

```bash
pnpm add @downcity/shell
```

## Shell

The `Shell` class manages command execution and the approval runtime. Application-level approvals belong to a specific Session.

### Minimal usage

```ts
import { Shell } from "@downcity/shell";
import { Agent } from "@downcity/agent";

const shell = new Shell();

const agent = new Agent({
  id: "demo",
  path: process.cwd(),
  shell,
});
```

When Shell is mounted through `Agent`, no path option is needed; the Agent configures its project root and runtime context. Pass `root_path` only when using Shell independently.

### Methods

- `shell.dispose()` — Release shell resources.
- `shell.set_safe_read_only_paths(paths)` — Replace host-approved read-only directories.

Approval requests appear as an `approval-required` Tool Part in `session.subscribe((mutation) => {})`. Read the complete request from `mutation.part.approval`, then submit the user's decision with `session.resolve_approval(...)`. A reconnected client can recover pending requests with `session.approvals()`. Do not call internal Shell Runtime approval methods directly.

## Model tools

A Shell mounted on an Agent provides:

| Tool | Purpose |
|------|---------|
| `shell_exec` | Run one non-interactive command |
| `shell_session` | Manage long-running or interactive PTY commands |
| `grep` | Search project file contents directly with ripgrep |
| `find` | Discover project files with POSIX glob patterns |
| `read` | Read paginated text and attach images to the model |
| `write` | Create UTF-8 files or atomically replace them with `overwrite: true` |
| `edit` | Apply unique, non-overlapping exact-text replacements to one file |

`read` returns up to 500 lines by default, with a maximum of 2,000 lines or 256KB per call. Continue with `truncated` and `next_offset`. `write` accepts up to 1MB and creates parent directories automatically.

For PNG, JPEG, GIF, WebP, and BMP files, `read` returns image metadata and a data URL. The Agent injects that data URL into the next model step as a temporary user image part. Other binary files return metadata only.

`edit` accepts up to ten operations. Every operation is matched against the original file, and the file is written only when all operations are valid. Pass `read.sha256` as `edit.expected_sha256` to reject stale concurrent edits.

`grep` runs `rg --json` directly without a shell. It searches literal text case-insensitively by default; set `literal: false` for regular expressions or `case_sensitive: true` for exact casing. Use `glob` to limit candidate files.

`find` accepts a POSIX glob `pattern`, includes dotfiles, respects `.gitignore`, and does not follow symbolic links. Both search tools return up to 200 results by default and accept `max_results` up to 2,000.

All file and search tools are fixed to the project root and reject symbolic-link escapes. They never request unrestricted access.

## Sandbox

Shell commands use the safe sandbox by default. The backend is selected automatically based on the host platform:

- `macos-seatbelt` — macOS sandbox
- `linux-bubblewrap` — Linux sandbox
- `unrestricted-host` — No sandboxing

The internal `Sandbox` entry resolves an access policy and invokes the platform backend. Shell Session remains the only owner of process state, output, and stdin.

### Additional read-only directories

A host can expose a version-pinned CLI or runtime directory to the Safe Sandbox as read-only:

```ts
const shell = new Shell({
  safe_read_only_paths: [
    "/Users/user/.vibecape/tools/officecli/v1.0.136",
  ],
});
```

Each entry must be an existing absolute directory outside the project and must not be group/world writable. macOS emits a Seatbelt `file-read*` rule, while Linux uses Bubblewrap `--ro-bind`. This never expands project-external write access.

On macOS, Shell resolves `xcode-select -p`, adds the selected Xcode runtime as a system read-only path, and sets `DEVELOPER_DIR` explicitly. This lets the system Git command load its complete runtime without project-external writes.

### Policy

`SandboxPolicy` combines system read-only paths, the fixed writable workspace, and host read-only paths, then validates realpaths, overlaps, and directory permissions.

`SandboxPreflight` runs pre-execution checks to validate the sandbox setup.
## Approval

`ShellApprovalRuntime` manages the interactive approval loop. Unrestricted requests use the current Session approval mode; the default `ask` mode requires an explicit approve or deny decision.

## ShellTools

`ShellTools`, `FileTools`, `SearchTools`, and their schemas provide the AI-ready definitions used by the Agent. You do not normally need to use them directly; they are wired automatically when you pass `shell` to the Agent constructor.
