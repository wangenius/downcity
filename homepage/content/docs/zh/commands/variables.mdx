# 命令环境要求总览

这页用于集中回答一个问题：每类 `city` 命令运行时，依赖哪些环境。

## 0) 一级命令边界

一级命令组有：

- `city console`
- `city agent`
- `city service`
- `city plugin`

像 `city chat`、`city task`、`city memory`、`city skill`、`city voice` 都是挂载在 service/plugin 体系下的动作命令，不是一级命令。

## 1) 各命令组所需环境

| 命令组 | 必需环境 | 可选环境 | 备注 |
| --- | --- | --- | --- |
| `city console init` | 可写 `~/.ship/` 目录 | 预先设置 `LLM_API_KEY`（可作为初始化默认值） | 会写 `~/.ship/ship.db`（加密存储敏感字段） |
| `city console start/stop/restart/status/agents` | 可写 `~/.ship/console/` | 无 | 管理全局 console 进程与 registry |
| `city console ui` | console 已启动；浏览器可访问 UI 端口 | `--host`、`--port` | 单 UI 实例可切换查看多个运行中的 agent |
| `city agent create` | 目标目录可写 | 无 | 会生成 `ship.json`、`PROFILE.md`、`.ship/` |
| `city agent start` | console 已启动；项目含 `ship.json` + `PROFILE.md` | `--host` | agent 端口始终由 console 自动分配，并登记到 console registry |
| `city agent restart/status/doctor` | 目标 agent 已在 console registry 注册 | 无 | 不是“任意路径直接操作”，先走 registry 解析 |
| `city console config ...` | 目标路径下 `ship.json` 存在且合法 | `--path` | 读写的是 agent 项目配置，不是 runtime API |
| `city console config alias` | 可写 `~/.zshrc` 或 `~/.bashrc` | `--shell zsh|bash` | 仅写 alias，不依赖 agent runtime |
| `city service ...` | 目标 agent runtime 可访问；项目已在 registry；项目含 `ship.json` + `PROFILE.md` | `--agent`、`--path`、`--host`、`--port` | `--agent` 优先于 `--path` |
| `city plugin ...` | 目标 agent runtime 可访问；项目已注册；项目含 `ship.json` + `PROFILE.md` | `--agent`、`--path`、`--host`、`--port` | `--agent` 优先于 `--path` |
| `city <service> <action>`（如 `city chat ...`） | 目标 agent runtime 可访问 | `--path`、`--host`、`--port` | 通过 `/api/services/command` 转发 |
| `city <plugin> <action>`（如 `city voice ...`） | 目标 agent runtime 可访问 | `--path`、`--host`、`--port` | 通过 `/api/plugins/action` 转发 |

## 2) 常见命令的上下文变量要求

### `city chat ...`

- `--chat-key` 未传时，会尝试读取 `DC_CTX_CHAT_KEY`
- `--context-id` 未传时，会尝试读取 `DC_CTX_CONTEXT_ID`
- 在 agent shell 内执行时，这些 `DC_CTX_*` 通常由运行时自动注入

### `city task ...`

- 创建/运行任务需要 `contextId`
- 可显式传 `--context-id`
- 未传时可回退读取 `DC_CTX_CONTEXT_ID`

### chat channel 凭据

- 唯一方式：在 `ship.json` 中绑定 `services.chat.channels.<channel>.channelAccountId`。
- 真实密钥（`botToken`/`appId`/`appSecret` 等）统一存放在 `~/.ship/ship.db` 的 `channel_accounts`。
- `ship.json` 与 agent `.env` 不再承载这些凭据。

## 3) `city service` 目标 agent 解析顺序

`city service list/status/start/stop/restart/command`：

1. 显式 `--agent <name>`（最高优先级）
2. 若未传 `--agent` 且 `--path` 为默认 `.`，先尝试 `DC_AGENT_NAME`
3. 回退到 `--path` 或默认 `.`
4. 若默认 `.` 且存在 `DC_AGENT_PATH`，优先解析为该路径

最终仍要求：

- 目标路径已登记在 console registry
- 路径内存在 `ship.json` 与 `PROFILE.md`

## 4) Runtime API 地址解析顺序（`host/port`）

调用 runtime API（service/plugin/action 命令）时，地址优先级：

1. CLI 显式参数：`--host` / `--port`
2. 环境变量：`DC_SERVER_HOST/PORT`
3. 环境变量：`DC_CTX_SERVER_HOST/PORT`
4. daemon meta 参数（`.ship/.debug/downcity.daemon.json`）
5. 默认值：`127.0.0.1:5314`

补充规则：

- `0.0.0.0` 或 `::` 会被归一化为 `127.0.0.1`

## 5) 变量注入来源

### runtime 启动后注入

- `DC_AGENT_PATH`
- `DC_AGENT_NAME`
- `DC_SERVER_HOST`
- `DC_SERVER_PORT`

### agent shell 子进程注入

- `DC_CTX_CONTEXT_ID`
- `DC_CTX_REQUEST_ID`
- `DC_CTX_SERVER_HOST`
- `DC_CTX_SERVER_PORT`

## 6) 推荐实践

多 agent 并行场景，建议显式指定目标：

```bash
city service list --agent my-agent
# 或
city service list --path /abs/path/to/agent
```

在 agent shell 内，默认形式通常足够：

```bash
city service list
```
