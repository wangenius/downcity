# 统一账户系统架构

这一页描述的是面向服务器部署的目标认证模型。

这里刻意不把认证设计写成多套并行方案，而是统一收敛成一个单一体系。

## 核心立场

Downcity 应该使用一个账户系统作为唯一身份源。

也就是说：

- 用户通过同一个登录入口登录
- 服务端签发统一的 Bearer Token
- 所有受保护 API 都通过这个 token 识别用户
- 浏览器 UI、CLI、Chrome Extension、自动化脚本都只是这套系统的客户端

## 为什么这是更合适的方向

- 可以直接收口当前“本地控制面缺少正式鉴权”的风险
- 可以避免再拆成“本地一套、服务器一套”的双系统
- 可以让所有客户端共享同一个权限模型
- 现在实现成本可控，后续也有扩展空间

## V1 范围

V1 只需要：

- 一个管理员 bootstrap 流程
- 一个登录 API
- 一套 Bearer Token
- 一套 RBAC + Scope
- 一条审计日志链路

V1 不需要：

- 客户端自己维护公钥/私钥
- 把浏览器 Session 和 PAT 当成两套独立产品
- 并行维护多套认证系统

## 请求模型

所有受保护请求统一收敛为：

```http
Authorization: Bearer <token>
```

服务端流程：

1. 校验 token hash
2. 解析所属用户
3. 执行角色与 scope 判断
4. 允许或拒绝请求
5. 对敏感操作写审计日志

## 部署边界

推荐边界如下：

- Gateway 是唯一对外入口
- Runtime 位于 Gateway 后面
- Runtime 不直接暴露到公网

## 覆盖的客户端

这一个模型应该统一覆盖：

- Console UI
- CLI
- Chrome Extension
- 自动化脚本

它们不是不同的认证系统，只是同一账户系统的不同客户端。

## 建议的数据模型

最小表集合：

- `auth_users`
- `auth_roles`
- `auth_permissions`
- `auth_user_roles`
- `auth_role_permissions`
- `auth_tokens`
- `auth_audit_logs`

## 计划中的模块区域

建议实现根目录：

- `packages/downcity/src/main/auth/`

建议职责包括：

- 登录与管理员初始化
- Bearer Token 校验
- 路由保护策略
- 权限判断
- 认证存储
- 审计日志

## 最终规则

长期来看，系统不应该演进成：

- 一套本地 token 系统
- 一套服务器账户系统
- 一套浏览器专属 session 系统

而应该始终保持：

- 一个账户系统
- 一个登录模型
- 一套 Bearer Token 访问模型
- 一套统一权限模型
