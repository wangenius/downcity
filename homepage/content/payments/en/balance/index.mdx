---
title: "Balance Service"
description: "Maintain one global user balance, ledger, topup orders, and redeem_code flows while keeping charging policy in business hooks."
---

`@downcity/services` gives you a global user wallet, not a hard-coded pricing engine.

It is responsible for:

- balance accounts
- balance ledger
- topup orders
- redeem_code flows

It does not decide:

- which actions should charge
- how much each call should cost
- which products are free or paid

Those rules should stay inside your services and their hooks.

## Reading order

1. Start with [Global wallet model](/en/payments/balance/global-wallet) to understand why balance is not bound to downcity.
2. Continue with [Charge directly from hooks](/en/payments/balance/hook-charging) for the recommended integration pattern.
3. Continue with [Topups and ledger](/en/payments/balance/topups-and-ledger) for create-first, confirm-later recharge flows.
4. Finish with [redeem_code flows](/en/payments/balance/redeem-codes) for one-time gift code credits.

## Minimal setup

```ts
import { BalanceService } from "@downcity/services";

const balance = new BalanceService({
  init_credits: 100_000_000,
});

base.use(balance);
```

## The most important boundary

### Balance is one global user wallet

One user shares the same balance across multiple products.

### Charging policy stays in business code

You should write:

- a pre-call `balance_precheck()`
- a post-success debit

inside the service that owns the action, instead of putting pricing rules into service config or next to `base.use(service)`.

## Common combinations

- pair it with [@downcity/services](/en/payments) for one wallet after login
- pair it with [@downcity/services](/en/payments) to separate "what happened" from "what was charged"
- pair it with [@downcity/services](/en/payments) to translate payment results into wallet topups
- use `redeem_code` on its own for gift credits, compensation codes, and campaign codes
