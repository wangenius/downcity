This page does one job: it pulls the calling model of `@downcity/city` into one reference overview.

If you want the integration story first, start with [@downcity/city](/en/docs/city/packages/city/gate). If you need concrete method details, continue to [User Gate](/en/docs/city/reference/gate-user) and [Admin Gate](/en/docs/city/reference/gate-admin).

## Two roles

| | User Gate | Admin Gate |
|---|---|---|
| Used by | frontend, app, extension, desktop client | studio backend, ops scripts, internal tools |
| Owns | AI services, custom services, public services | studios, tokens, runtime env, admin-side services |
| Common auth | `user_token` | `admin_secret_key` |
| Safe in browser | yes | no |

## Four common call types

```ts
// Studio side
const client = new Gate({
  role: "user",
  city_url,
  studio_id,
  user_token,
});

// AI Service
await client.ai.text({ prompt: "hello", model: "deepseek-v4-flash" });
await client.ai.stream({ prompt: "hello" });

// Custom Service
await client.service("translate").action("zh2en").invoke({ text: "hello" });
await client.service("accounts").action("login").invoke({ email: "a@b.com", password: "..." });

// GET action
const items = await client.service("studios").get("list");

// Payment methods
const methods = await client.payment.methods();
const checkout = await client.payment.method("stripe").invoke({
  topup_id: "topup_demo",
});

// Admin side
const admin = new Gate({
  role: "admin",
  city_url,
  admin_secret_key,
});
await admin.service("studios").action("create").invoke({ name: "My App" });
```

## Relationship between service, service, and AI service

- AI service: called through `client.ai.*`
- custom service: called through `client.service(name)...`
- service: still called through `client.service(name)...`, but the service comes from a service

Examples:

- `rewrite`, `translate`: custom services you registered yourself
- `accounts`, `usage`, `payment`, `payment.stripe`: services exposed by official packages

From the caller's point of view, they all live in the same City service route space.

## Where registration and login fit in the SDK model

The two most common paths are:

1. your backend finishes login first, then uses `Admin Gate` to request `user_token`
2. the frontend uses a guest `User Gate` to call `accounts.login/register`, and the accounts service returns `user_token`

Both paths end in the same user context:

```txt
city_url + studio_id + user_token
  -> new Gate({ role: "user", ... })
  -> client.ai.* or client.service(...)
```
