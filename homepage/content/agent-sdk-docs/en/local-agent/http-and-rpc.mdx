# Local Agent RPC and Town HTTP

A local Agent can be called directly in-process, and it can expose a local RPC endpoint for another local process.

HTTP is not an `Agent` SDK server capability. In Downcity, HTTP exposure is owned by Town.

## What `agent.start()` does

`agent.start()` handles:

- starting plugin lifecycles
- optionally starting the local RPC server

Example:

```ts
const started = await agent.start({
  rpc: {
    host: "127.0.0.1",
    port: 15314,
  },
});

console.log(started.rpc?.url);
```

The return value includes:

- `started.rpc?.url`
- `started.pluginsStarted`

## What `agent.start()` does not do

`agent.start()` does not start an HTTP server.

That means this is not the current real API:

```ts
await agent.start({
  http: {
    host: "127.0.0.1",
    port: 15314,
  },
});
```

## How to expose HTTP through Town

If you want HTTP, start the Agent with Town and let Town publish the Agent HTTP gateway:

```bash
town agent start
```

The responsibility split is:

- `Agent`: local execution shell
- `agent.start()`: long-lived plugins and optional local RPC
- `town agent start`: Agent process host and HTTP gateway publisher

## `start()` vs not calling `start()`

If you do not call `agent.start()`:

- you can still call `await agent.createSession()` and then use `session.prompt()` / `session.subscribe()`
- this is the right shape when you use the Agent as a local in-process library
- no background plugin lifecycle starts
- no RPC server starts

After you call `agent.start()`:

- plugins start by default
- ActionSchedule and other background loops can keep running
- if you pass `rpc`, the Agent also exposes a `rpc://...` endpoint

## When to use RPC

- another local process should consume Agent session capability directly
- you do not want to mount HTTP yourself
- you want a lighter local transport

## When to use HTTP

- an external client should call the Agent through Town over HTTP
- you want a standard HTTP entry for frontend/backend or third-party consumers
- you want Town to host SDK routes and plugin HTTP routes on the same gateway

## One plugin-related detail that matters

When Town exposes the Agent HTTP gateway, registered plugins with `plugin.http.server.register(...)` are mounted onto the same HTTP app automatically.

So:

- `agent.plugins` owns the plugin registry and plugin calls
- Town is what actually exposes plugin HTTP routes
