# Runtime 演进方向

这页不描述“已经是什么”，而是回答：

- 基于现在的代码，后续最稳的演进方向是什么

## 先说结论

- `console` 应该越来越像纯控制面
- `agent host` 应该越来越像纯执行面
- `session` 应继续保持统一执行主轴
- `plugin` 应继续承担能力暴露与运行时增强
- 托管 plugin runtime 应只在确实需要长期状态时存在

## 为什么还需要继续演进

当前实现已经完成了很多关键收敛：

- plugin hooks 统一到 `pipeline / guard / effect / resolve`
- `AgentCore` 已经收束成实例级内核
- `skill`、`web`、`auth` 这类能力已经清楚回到 plugin 体系

但还有几个方向值得继续强化。

## 1. 控制面和执行面继续分清

更稳的方向是：

- console 只管理 registry、模型池、共享配置、可见性和控制入口
- agent host 只负责项目执行、runtime 装配与会话推进

## 2. session 始终保持执行主轴

后续演进时，不要把执行权重新打散到额外层里。

最稳定的链路仍然应该是：

- `AgentCore`
- `Session`
- `Executor`

## 3. plugin runtime 继续收口

更稳的方向是：

- 显式能力继续做 action
- 运行时增强继续走 hook
- 长期模块继续做托管 runtime

不要再引入一套和这些平行的旧式能力层表达。

## 4. 文档和代码命名继续统一

最容易拖慢协作的不是功能缺失，而是名字和心智对不上。

所以后续继续重构时，优先级应该是：

1. 先统一执行主轴的叫法
2. 再统一 plugin runtime 的叫法
3. 最后再处理历史遗留类名或注释

## 一句话总结

```text
后续最稳的方向，不是重新发明一套 runtime，而是继续强化 console 作为控制面、agent host 作为执行面、session 作为执行主轴、plugin 作为能力层的边界。
```
