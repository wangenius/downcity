# task 内建 Service

`task` 是负责调度与后台执行的内建 service。

它拥有 task 定义、重复触发、runtime reload 行为，以及可审计的执行流程。

## 它持有什么

从当前实现看，它的核心长期持有内容包括：

- per-service-instance 的 cron engine
- start、stop、reload 对应的 service lifecycle
- task 定义变更后会触发 reschedule 的 action surface
- 通过 task prompt 注入的 service 级 system 语义

## 为什么它是 service，而不是 tool

调度不是一次调用就结束的能力。

它需要：

- 长期存活的 engine
- 启动边界
- 停止边界
- 变更后的 reload 逻辑

这正是 service 最适合承载的运行时形态。

## 为什么 SDK 使用者应该关注它

`task` 是理解“一个 service 如何在前台 session 之外持续工作”的最好参考之一。

如果你正在设计：

- cron 风格自动化
- 重复性后台任务
- 需要自己持有 scheduler 的 service

这页会很有参考价值。

## 最适合的理解方式

可以把 `task` 理解成后台自动化运行时。

agent 可以去定义和修改任务，但真正把这些工作长期维持起来的是 `task` service 自己持有的 scheduler。
