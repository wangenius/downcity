# Local Agent HTTP / RPC

A local Agent can be called directly in-process, but it can also expose service endpoints.

In the current implementation, `RPC` and `HTTP` do not come from the same entry point.

## What `agent.start()` currently does

`agent.start()` currently handles:

- starting plugin lifecycles
- optionally starting the local RPC server

Example:

```ts
const started = await agent.start({
  rpc: {
    host: "127.0.0.1",
    port: 15314,
  },
});

console.log(started.rpc?.url);
```

The return value includes:

- `started.rpc?.url`
- `started.pluginsStarted`

## What `agent.start()` does not do

`agent.start()` does not currently start the HTTP server.

That means this is not the current real API:

```ts
await agent.start({
  http: {
    host: "127.0.0.1",
    port: 15314,
  },
});
```

## How to expose HTTP

If you want to expose the local Agent over HTTP, use the host-side HTTP assembly entry:

```ts
import { Agent, startServer } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  model,
});

await agent.start({
  plugins: true,
});

const server = await startServer({
  host: "127.0.0.1",
  port: 15314,
  getAgentRuntime: () => agent.getRuntime(),
  getAgentContext: () => agent.getContext(),
  sessionCollection: agent.getSessionCollection(),
});
```

The responsibility split is:

- `Agent`: local execution shell
- `agent.start()`: long-lived plugins / optional RPC
- `startServer(...)`: mount runtime, context, and session collection as HTTP

## `start()` vs not calling `start()`

If you do not call `agent.start()`:

- you can still call `await agent.createSession()` and then use `session.prompt()` / `session.subscribe()`
- this is the right shape when you use the Agent as a local in-process library
- no background plugin lifecycle starts
- no RPC server starts

After you call `agent.start()`:

- plugins start by default
- ActionSchedule and other background loops can keep running
- if you pass `rpc`, the Agent also exposes a `rpc://...` endpoint

## When to use RPC

- another local process should consume Agent session capability directly
- you do not want to mount HTTP yourself
- you want a lighter local transport

## When to use HTTP

- another process should call the agent through `RemoteAgent({ url: "http://..." })`
- you want a standard HTTP entry for frontend/backend or third-party consumers
- you want SDK routes and plugin HTTP routes on the same host

## One plugin-related detail that matters

When you expose HTTP through `startServer(...)`, registered plugins with `plugin.http.server.register(...)` are also mounted onto the same HTTP app automatically.

So:

- `agent.plugins` owns the plugin registry and plugin calls
- `startServer(...)` is what actually exposes plugin HTTP routes
