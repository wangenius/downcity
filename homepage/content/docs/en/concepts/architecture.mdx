# Architecture Overview

Downcity is easiest to understand through two top-level trunks:

- `console`: the global control and operations plane
- `agent`: the per-project execution plane

Inside an agent runtime, the execution layer is built from:

- `service`: workflow owners
- `plugin`: passive extensions
- `asset`: plugin dependency resources

## Layered Structure

1. **Console Layer**
- owns registry, model pool, global config, and the UI gateway

2. **Agent Layer**
- loads one project, owns one runtime, and persists traces into `.downcity/`

3. **Service Layer**
- domains like `shell`, `chat`, `task`, and `memory`
- own lifecycle, workflow, and stable actions

4. **Plugin Layer**
- modules like `skill`, `voice`, and `auth`
- attach through extension points defined by services
- augment, validate, transform, or resolve behavior without owning a lifecycle

5. **Asset Layer**
- installable or configurable dependencies behind plugins
- should be understood as plugin dependency infrastructure, not a separate business workflow layer

## The Most Important Boundary

- `service` actively participates in the agent execution cycle
- `plugin` does not run on its own and only participates when a service triggers it

So:

- main workflows belong in services
- extensions belong in plugins
- plugin dependencies belong in assets

## A Concrete Example

In chat flows:

- the `chat` service owns inbound messages, queueing, execution, and replies
- the `voice` plugin adds transcription behavior at voice-related points
- the `auth` plugin adds authorization and role resolution
- the `voice.transcriber` asset provides the real dependency
