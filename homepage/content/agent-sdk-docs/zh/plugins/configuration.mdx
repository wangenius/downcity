# Plugin 配置

plugin 配置要分成 3 层理解：

- 项目级启用态
- 项目级 plugin 选项
- Console 的 `setup / usage` 协议

## 1. 项目级启用态

plugin 是否启用，写在项目 `downcity.json`。

它属于项目级事实。

这意味着：

- `plugins.<name>.enabled` 控制当前项目是否启用这个 plugin
- 未显式配置 `enabled` 时，内建 plugin 通常按启用处理
- `enabled: false` 会明确关闭当前项目里的该 plugin

## 2. 项目级 plugin 选项

plugin 的运行参数通常落到项目：

```json
{
  "plugins": {
    "chat": {
      "enabled": true,
      "channels": {
        "telegram": {
          "enabled": true,
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

这类配置表达的是：

- 当前 agent 怎么使用这个 plugin
- 当前项目是否明确关闭了它

## 3. `config.scope`

plugin 类型里支持：

- `global`
- `project`

这表示的是 plugin 配置字段的作用域。当前内建项目插件通常用项目配置同时保存启用态和运行参数。

当前常见内建 plugin，很多都把运行参数放在 `project`。

## `setup` 和 `usage` 是什么

### `setup`

`setup` 面向：

- 安装
- 修复
- 初次依赖准备

典型内容：

- 安装依赖
- 下载模型
- 安装 Python 依赖

### `usage`

`usage` 面向：

- 当前 agent 如何使用这个 plugin
- 行为选项怎么保存

典型内容：

- 是否 inject prompt
- 默认模型
- 默认格式 / 语言 / 语速

## 为什么要分 `setup` 和 `usage`

因为这两件事不是一回事：

- `setup` 负责把能力装起来
- `usage` 负责告诉 agent 现在怎么用它

把它们分开后，Console 可以更清楚地呈现：

- `安装/修复`
- `选项`

## plugin 配置通常怎么持久化

当前 project 级 plugin 参数一般会写回：

- `downcity.json.plugins.*`

对应的持久化工具会只写 `plugins` 配置块，而不会把整份执行期合并态直接落盘。

## 推荐理解方式

把 plugin 配置记成这三句：

- 开关在项目 `plugins.<name>.enabled`
- 参数通常也在项目 `plugins.<name>`
- setup 和 usage 是 UI 协议层
