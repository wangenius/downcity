# Unified Auth V1 Implementation

This page turns the unified auth architecture into a concrete first-stage implementation plan.

## V1 Goal

V1 does one thing:

```text
Build one account system and move all protected APIs to unified bearer-token authentication.
```

## V1 Includes

- admin bootstrap
- login
- bearer token issue and verification
- auth middleware
- route protection policy
- basic audit logging

## Suggested Tables

- `auth_users`
- `auth_roles`
- `auth_permissions`
- `auth_user_roles`
- `auth_role_permissions`
- `auth_tokens`
- `auth_audit_logs`

## Suggested New Module Area

- `packages/downcity/src/main/auth/`

Suggested first modules:

- `AuthSchema`
- `AuthStore`
- `AuthService`
- `TokenService`
- `AuthMiddleware`
- `RoutePolicy`
- `PermissionPolicy`
- `AuthRoutes`
- `TokenRoutes`

## First Protected Surfaces

These should all require bearer token auth:

- `/api/execute`
- `/api/services/*`
- `/api/plugins/*`
- `/api/dashboard/*`
- `/api/ui/*`

## First Rollout Order

1. create schema and auth store
2. add admin bootstrap and login
3. add token issue and revoke
4. add auth middleware
5. protect runtime and console routes
6. update Console UI, CLI, and Chrome Extension to send bearer token

## Done Criteria

V1 is done when:

- the first admin can be bootstrapped
- login returns a bearer token
- protected APIs reject unauthenticated requests
- high-risk actions can be permission-gated
- Console UI and Extension can use the token successfully
