# 本地 Agent 的 HTTP / RPC

本地 Agent 不只可以在当前进程里直接调用，也可以对外暴露服务端点。

但当前实现里，`RPC` 和 `HTTP` 不是同一个入口。

## `agent.start()` 当前负责什么

当前 `agent.start()` 负责：

- 启动 plugin lifecycle
- 可选启动本机 RPC 服务

例如：

```ts
const started = await agent.start({
  rpc: {
    host: "127.0.0.1",
    port: 15314,
  },
});

console.log(started.rpc?.url);
```

返回里会有：

- `started.rpc?.url`
- `started.pluginsStarted`

## `agent.start()` 不负责什么

当前 `agent.start()` 不负责启动 HTTP server。

也就是说，这样的写法并不是当前真实 API：

```ts
await agent.start({
  http: {
    host: "127.0.0.1",
    port: 15314,
  },
});
```

## 如何启动 HTTP

如果你要把本地 Agent 暴露成 HTTP 服务，应使用宿主 HTTP 装配入口：

```ts
import { Agent, startServer } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  model,
});

await agent.start({
  plugins: true,
});

const server = await startServer({
  host: "127.0.0.1",
  port: 15314,
  getAgentRuntime: () => agent.getRuntime(),
  getAgentContext: () => agent.getContext(),
  sessionCollection: agent.getSessionCollection(),
});
```

这里的职责划分是：

- `Agent`：本地执行壳
- `agent.start()`：长期运行能力中的 plugins / rpc
- `startServer(...)`：把 runtime、context、sessionCollection 挂成 HTTP 服务

## `start()` 和不 `start()` 的区别

不调用 `agent.start()` 时：

- 你仍然可以直接 `await agent.createSession()`，再用 `session.prompt()` / `session.subscribe()`
- 适合把 Agent 当作当前进程里的本地库
- 不会启动 plugin 的后台生命周期
- 不会启动 RPC

调用 `agent.start()` 之后：

- plugins 默认会启动
- ActionSchedule 和其他后台 loop 才会真正持续运行
- 如果传了 `rpc`，还会暴露 `rpc://...` 地址

## 什么时候用 RPC

- 另一个本地进程想直接消费 Agent session 能力
- 你不想自己额外挂 HTTP
- 你想走更轻量的本机 transport

## 什么时候用 HTTP

- 你想让另一个进程通过 `RemoteAgent({ url: "http://..." })` 调它
- 你想给前后端或第三方程序一个标准 HTTP 入口
- 你需要同时挂 SDK 路由和 plugin HTTP 路由

## 一个和 plugin 相关的重要点

当前通过 `startServer(...)` 创建 HTTP 服务时，已注册 plugin 的 `plugin.http.server.register(...)` 也会自动挂到同一个 HTTP app 上。

所以：

- `agent.plugins` 负责插件注册表与调用
- `startServer(...)` 负责把插件的 HTTP 路由真正挂出去
