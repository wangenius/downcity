# web Plugin

`web` is the methodology layer for web tasks. It injects general strategies for search, source verification, page reading, and browser interaction, but it does not provide concrete web execution capability.

It exposes an `install` action that only returns instructions. The action does not execute commands, access the network, install dependencies, or change files. The Agent reads `data.prompt` and follows it with the currently available Plugin and Shell capabilities.

## Registration

```ts
import { Agent } from "@downcity/agent";
import { WebPlugin } from "@downcity/plugins/web";

const agent = new Agent({
  id: "research-agent",
  path: "/path/to/project",
  plugins: [new WebPlugin()],
});
```

After registration, the methodology returned by `WebPlugin.system()` is automatically included in the Agent's system prompt.

## install Action

```ts
const result = await agent.plugins.runAction({
  plugin: "web",
  action: "install",
  payload: {
    target: "agent-browser",
    scope: "project",
  },
});
```

The core result shape is:

```ts
{
  kind: "instructions",
  target: "agent-browser",
  scope: "project",
  prompt: "...installation steps for the Agent...",
}
```

Available `target` values:

- `web-access`: return the `web-access` Skill installation workflow; this is the default
- `agent-browser`: return the `agent-browser` Skill and CLI installation workflow
- `all`: return the workflows for both capabilities

The optional `scope` controls only the `agent-browser` CLI guidance:

- `user`: direct the Agent to use a global npm installation; this is the default
- `project`: direct the Agent to inspect the lockfile and use the project's existing package manager to add a devDependency

`scope` does not choose where a Skill is installed. Skill placement comes from the active `SkillPlugin` `use / paths` scan configuration.

## Installation Workflow

`web.install` directs the Agent through this workflow:

1. Call `skill.install` for every required Skill.
2. Follow the scan-aware Shell guidance returned by `skill.install`.
3. When `agent-browser` also needs its CLI, install and verify it with the commands for the requested `scope`.
4. Call `skill.list` to confirm that every new Skill is discoverable.
5. Call `skill.lookup` to read `SKILL.md` before using the capability.

For example, the Skill Plugin remains responsible for the `agent-browser` Skill installation guidance:

```ts
plugin_call({
  plugin: "skill",
  action: "install",
  payload: {
    spec: "agent-browser",
  },
});
```

This keeps the active SkillPlugin scan roots as the single source of truth. WebPlugin does not duplicate project, home, or custom-root installation rules.

## Capability Boundary

`web` does not:

- choose a web provider
- decide which web implementation the Agent must use at runtime
- perform actual search, fetching, clicking, or browser interaction
- execute Skill or CLI installation commands
- read or write project-level web configuration

Concrete web capability comes from the tools, Skills, or other Plugins in the current Agent environment. Search and fetch tools can handle public discovery, while browser, Chrome, agent-browser, or computer-use capabilities can handle dynamic pages and logged-in interaction.

If a required capability is missing, the Agent should state that directly instead of pretending it accessed the web.

## System Prompt

The WebPlugin methodology directs the Agent to:

- inspect the tools and Skills that are actually available
- choose a path for search, reading, structured extraction, or browser interaction
- verify conclusions against primary sources such as official pages, original documentation, and source code
- switch methods when the current path stops making progress
- stop when the goal is complete instead of adding unnecessary steps
