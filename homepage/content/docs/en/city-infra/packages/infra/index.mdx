`@downcity/infra` is the InfraRuntime itself. If you are deploying a real InfraRuntime that will serve one or more products, this is the package everything rests on.

This is not a small helper around an AI SDK. It is the core server-side InfraRuntime where your AI services, custom services, official services, and shared `/v1/*` route surface come together.

## How to read this section

The best reading order is:

1. Start with [InfraRuntime and bootstrapping](/en/docs/city-infra/packages/infra/infra-runtime) to understand what a `InfraRuntime` instance actually owns.
2. Continue with [Service and Action design](/en/docs/city-infra/packages/infra/service-and-action) to see why AI services, custom services, and services all end up behind one route model.
3. Read [Hooks and service integration](/en/docs/city-infra/packages/infra/hooks-and-service) when you need extension points.
4. Read [Store and table data model](/en/docs/city-infra/packages/infra/store-and-table) when you need to understand persistence and table layout.

If you only want the big picture first, finish this page and then follow the sequence above.

## What this package actually owns

- creating the `InfraRuntime` instance
- registering `Service`, official services, and `AIService`
- product, env, and token-related built-in capabilities
- the unified `/v1/*` route boundary

## What it does not own

- It does not own the product-side SDK. That belongs to [@downcity/conduit](/en/docs/city-infra/packages/conduit).
- It does not create the concrete database connection. Create a Drizzle ORM db and pass it to `new InfraRuntime({ db })`.
- It does not own ready-made auth, usage, or payment business logic. Those come from official services.

## When this should be your first stop

- you are building a real InfraRuntime deployment
- you need your own AI services or custom services
- you want shared auth, hooks, and model catalogs behind one backend

## Most common package combinations

| Goal | Package combination |
| --- | --- |
| Local or standard Node deployment | `@downcity/infra` + Drizzle SQLite / Postgres |
| Product frontend or app calls | `@downcity/infra` + `@downcity/conduit` |
| Accounts, usage, or payments | `@downcity/infra` + the matching official service |

## Install

```bash
pnpm add @downcity/infra drizzle-orm better-sqlite3
```

## Minimal example

```ts
import { InfraRuntime, AIService } from "@downcity/infra";
import Database from "better-sqlite3";
import { drizzle } from "drizzle-orm/better-sqlite3";

const sqlite = new Database("./data.sqlite");
sqlite.pragma("journal_mode = WAL");

const db = Object.assign(drizzle(sqlite), {
  $client: { exec: (sql: string) => sqlite.exec(sql) },
});

const base = new InfraRuntime({ db, dialect: "sqlite", raw: sqlite });

const ai = new AIService();

ai.use({
  id: "local-echo",
  name: "Local Echo",
  default: ["text"],
  actions: {
    text: async (ctx) => ({
      id: crypto.randomUUID(),
      role: "assistant",
      parts: [
        {
          type: "text",
          text: String(ctx.input.prompt ?? ""),
          state: "done",
        },
      ],
    }),
  },
});

base.use(ai);
```

This example shows the real role of `@downcity/infra`: it is where the database, services, and services are mounted into one InfraRuntime. It does not decide how the frontend logs in or how products call services.
