---
title: Global wallet model
description: "Why @downcity/services stores balance by user instead of creating one wallet per downcity."
---

`@downcity/services` keeps balance by user, not by downcity.

By default, credits are USD-linked:

- `1 USD = 1,000,000 credits`
- credits are the only accounting unit
- user-facing `GET /v1/balance/me` returns ``credits`, plus `usd`, `display`, and `conversion`
- admin and internal accounting fields also use credits

The wallet can go below zero. A debit is accounting truth; access control should happen before an action starts by calling `precheck()`.

That means:

- user A spends inside downcity X
- user A spends inside downcity Y
- both consume the same shared balance

## Why this fits Downcity better

The typical Downcity shape is:

- one Federation serving multiple products
- the same user moving across those products
- one shared AI account layer behind them

If balance is split per product, both product UX and operations become fragmented.

## Does downcity still matter

Yes, but it belongs in ledger metadata rather than in the wallet primary key.

```ts
await balance.sub(ctx.user!.user_id, 10_000, {
  note: "agent chat",
  meta: {
    city_id: ctx.downcity?.city_id,
    model_id: ctx.variant?.id,
  },
});
```

That keeps the wallet global while still preserving where the spend came from.

## Negative balance and precheck

Use `precheck(user_id)` before actions that should stop already-overdrawn users:

```ts
await balance.precheck(ctx.user!.user_id);
```

Use `precheck(user_id, needed_credits)` when an action needs a known minimum balance before it starts:

```ts
await balance.precheck(ctx.user!.user_id, 10_000);
```

The debit itself still goes through `sub()` or `charge()`, and either method can move the wallet below zero.
