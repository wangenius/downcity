---
title: 全局钱包模型
description: "为什么 @downcity/services 按 user 维护余额，而不是按 downcity 拆很多份余额。"
---

`@downcity/services` 的核心是：余额按用户维护，不按 downcity 维护。

默认情况下，credits 和 USD 1:1 挂钩：

- `1 credit = 1 USD`
- `1 credit = 1,000,000 microcredits`
- 用户侧 `GET /v1/balance/me` 会返回以 credits 为单位的 `balance`，并补充 `microcredits`、`display`、`conversion`
- 管理端与内部账务字段仍然统一使用 microcredits

也就是说：

- user A 在 downcity X 消费
- user A 在 downcity Y 消费
- 扣的都是同一份全局余额

## 为什么这样更适合 Downcity

Downcity 的典型场景是：

- 一套 Federation 服务多个产品
- 同一个用户可能在多个产品之间切换
- 你希望这些产品复用同一套 AI 账户能力

如果余额再按 downcity 拆成很多份，用户体验和运营逻辑都会更碎。

## downcity 还重要吗

重要，但它更适合出现在流水 metadata 里，而不是余额账户主键里。

比如一次扣费可以记录：

```ts
await balance.sub(ctx.user!.user_id, 10, {
  note: "agent chat",
  meta: {
    city_id: ctx.downcity?.city_id,
    model_id: ctx.variant?.id,
  },
});
```

这样你同时得到两件事：

- 余额仍然是全局的
- 历史记录里仍然知道这笔钱花在哪个 downcity

## 账户里保存什么

最小账户只需要：

- `user_id`
- `balance`
- `unit`

这里的 `unit` 可以是：

- `credits`
- `points`
- `tokens`

## 这种模型最适合什么场景

- 多个产品共用一套 AI 能力
- 你想做统一账户余额
- 你不想把钱包切碎到每个 downcity 各算一份
