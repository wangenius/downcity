# City Config Storage (downcity.db)

From current versions, shared City-side config no longer uses:

1. `~/.downcity/downcity.json`
2. `~/.downcity/.env`

City-side config is unified in:

- `~/.downcity/downcity.db`

## What is stored

`downcity.db` contains these major groups:

1. Federation AIService model catalog (providers/models)
2. City-managed env records (`env_entries`)
4. Bot account vault (`channel_accounts`)
5. Chat user authorization (`chat_auth_roles`, `chat_auth_role_permissions`, `chat_auth_channel_defaults`, `chat_auth_user_roles`)

Sensitive fields (such as API keys) are stored encrypted.

## Boundary with Agent project config

These files remain agent-project-level config (unchanged):

1. `<agent>/downcity.json`
2. `<agent>/.env`

So the split is:

- City: `~/.downcity/downcity.db`
- Agent: project-local `downcity.json/.env`

## Environment Variable Scope (Important)

Current versions use explicit layers:

1. Host-injected env: `new Agent({ env })`
2. Agent-local env overlay: `<agent>/.env` (user-managed)
3. Chat channel credentials: `channel_accounts` (bound by `downcity.json` `channelAccountId`)
4. Chat user authorization: `chat_auth_*` structured tables (city-global)

Key behavior:

1. Agent runtime env map is `new Agent({ env }) + <agent>/.env` (`.env` wins).
2. When the host is `downcity`, City-side env is read from `downcity.db` and passed explicitly to `Agent`.
3. `downcity.json` stores channel binding (`channelAccountId`) instead of plaintext secrets.
4. Bot credentials are resolved from `channel_accounts`.
5. Chat user permissions are resolved from `chat_auth_*` tables, not from an agent project.
6. Plugin parameters are read from project `downcity.json`; plugin runtime state belongs to the Agent process.

## Management entry points

Local Downcity state, project plugin parameters, plugin runtime operations, and City model binding all have product-level command surfaces.

Use [Downcity CLI](/en/docs/reference/cli), [Downcity config docs](/en/docs/reference/cli), [Plugin CLI docs](/en/docs/reference/cli), and [Downcity City docs](/en/docs/reference/cli) for concrete workflows.

## Process files

These process-state files stay in filesystem (not config):

1. `~/.downcity/main/downcity.pid`
2. `~/.downcity/main/downcity.daemon.log`
3. `~/.downcity/main/agents.json`

Legacy `gateway.pid` and `gateway.json` files can be removed manually after upgrading if they exist from older installs.
