The concepts section is not extra theory. It exists to keep the Downcity boundary clear before you wire it into real towns.

If you start by treating Downcity as “just another backend framework” or “a generic provider gateway,” the service boundary, town split, and trust model will all drift in the wrong direction.

## Start with this picture

<div className="dc-doc-flow" aria-label="Downcity conceptual overview">
  <div className="dc-doc-lane">
    <div className="dc-doc-node">

      <span>Web apps, extensions, apps, desktop tools, and client demos can all be separate towns.</span>
      <code>town_id + user_token + User City</code>
    </div>
  </div>
  <div className="dc-doc-arrow" aria-hidden="true">→</div>

    <div className="dc-doc-node">

      <span>Centralizes the model directory, provider keys, service routing, usage, and hooks.</span>
    </div>
    <div className="dc-doc-node">

    </div>
  </div>
</div>

## When this section is the right starting point

- you want multiple AI towns to share the same AI infrastructure
- you need to understand the role of `town_id`, `user_token`, and the model directory
- you are still deciding where City, your business backend, and an AI Gateway should stop

If you only want one working call chain first, go straight to [Quickstart](/en/city-sdk-docs/quickstart).
