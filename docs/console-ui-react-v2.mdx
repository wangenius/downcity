# 1. 结论

这个方向正确，建议执行：

- Console UI 从 packages/downcity/public 静态文件彻底解耦。
- 在仓库根目录新增独立前端包（Vite + React）。
- city console ui 只负责启动 UI Gateway + 加载该前端包构建产物。
- UI 能力继续通过 /api/* 访问现有 console/agent API，不把业务逻辑塞回后端模板。

这会显著提升以下能力：

- 可维护性（前后端边界清晰）。
- UI 设计空间（组件体系、主题、交互状态管理）。
- 工程效率（热更新、类型检查、测试与构建分离）。

# 2. 目标与非目标

## 2.1 目标

- 使用 Vite + React + TypeScript 重建 Console UI。
- 引入“自定义 UI 包”（Design Tokens + 组件库）。
- UI 作为独立 workspace 包存在，独立开发、构建、发布。
- city console ui start 统一加载该包的 dist。

## 2.2 非目标

- 不在本阶段改动 agent runtime 的核心服务编排逻辑。
- 不在本阶段引入 SSR（先走纯 SPA + API）。
- 不做“保留旧 UI 双栈长期并行”。

# 4. 架构设计

# 4.1 分层

- 运行层（package）ConsoleUIGateway：只做静态资源托管与 API 代理。不再承载 UI 页面代码。
- 应用层（console-ui）React 页面、路由、状态管理、交互逻辑。
- 组件层（console-ui-kit）设计 Token、通用组件、布局原语、主题系统。

# 4.2 运行时加载协议

city console ui start 时：

1. 启动 ConsoleUIGateway。
2. Gateway 从约定路径读取 products/console/dist：index.htmlassets/*
3. 浏览器访问 UI 端口时，返回 React 构建产物。
4. React 前端通过 /api/*（同域）调用当前已有接口。

> 关键约束：Gateway 只负责“静态资源 + API 代理”，不再维护 UI DOM/样式逻辑。

---

# 5. 前端技术栈建议

- 构建：Vite
- 框架：React 19 + TypeScript
- 路由：React Router（或 TanStack Router，二选一）
- 数据层：TanStack Query
- 组件：@downcity/console-ui-kit
- 样式：Token 使用 CSS Variables（--city-*）组件层可选 Tailwind / CSS Modules（二选一，避免混搭）

---

# 6. UI 包职责拆分

# 6.1 console-ui（应用包）

- 页面与信息架构
- Agent 切换、Dashboard 组装
- Chat Channels 操作（test/reconnect）
- Prompt / local_ui 面板
- API 错误态和空态

# 6.2 console-ui-kit（组件包）

- 设计 Token（颜色、间距、字体、圆角、阴影）
- 基础组件（Button, Badge, Table, Panel, Tabs, Toast）
- 复合组件（StatusPill, RuntimeCard, ChannelRow）
- 动画与交互规范（统一动效参数）

# 7. Console 侧改造点

# 7.1 ConsoleUIGateway

- 移除对 packages/downcity/public 的硬编码依赖。
- 改为读取配置化路径（默认 ./products/console/dist）。
- 支持 SPA fallback（任意前端路由返回 index.html）。

# 7.2 CLI 命令

保留现有命令语义：

- city console ui start|stop|status

新增建议（可选）：

- city console ui build：触发 console-ui 构建（本地开发便利）
- city console ui dev：前端开发模式（Vite dev server + API 代理）

# 8. 开发与发布流程

# 8.1 本地开发

1. 启动 console + agent runtime。
2. console-ui 运行 Vite dev。
3. 前端通过 /api/* 调 Gateway/Agent。

# 8.2 生产构建

1. pnpm -w --filter console-ui build
2. 产物落地到 products/console/dist
3. city console ui start 直接托管该目录

# 9. 分阶段迁移计划

# Phase 1：工程落地（1-2 天）

- 新建 console-ui 与 console-ui-kit 包。
- 建立 workspace、tsconfig、lint/build 脚手架。
- Gateway 支持读取 products/console/dist。

# Phase 2：功能平移（2-4 天）

- 把现有 Dashboard 功能逐项迁移到 React：概览、services、chat channels、tasks、logs、prompt、local_ui
- 保持接口不变，先实现功能等价。

# Phase 3：视觉重构（2-3 天）

- 基于 UI Kit 实现统一组件和版式系统。
- 去重信息结构，完善状态层级、空态、错误态、加载态。

# Phase 4：清理与收口（1 天）

- 删除 packages/downcity/public 旧 UI 页面资源。
- 更新 docs 与命令说明。

# 11. 风险与控制

- 风险：前端包构建失败导致 UI 不可用控制：CI 增加 console-ui build 必过门禁。

- 风险：Gateway 静态资源路径配置错误控制：启动时做 dist 目录健康检查并输出明确错误。

- 风险：API 变更导致前端破裂控制：为 /api/tui/ 和 /api/services/ 增加契约测试（最小 smoke）。

---

# 12. 建议的第一步（可立即执行）

先落地 Phase 1，不改业务逻辑，只搭骨架：

1. 新建 console-ui（Vite + React + TS）。
2. 新建 packages/console-ui-kit（Button/Panel/Badge 三个最小组件）。
3. 改 ConsoleUIGateway：静态目录改指向 products/console/dist。

这样 1-2 天内就能把“架构方向”定住，后续迭代只在 React 包里推进。
