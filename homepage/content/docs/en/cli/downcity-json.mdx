# legacy downcity.json

New projects no longer need `downcity.json`. `city agent create` stores agent ID, startup options, model binding, and plugin options in the CLI global DB. The project directory keeps `PROFILE.md`, `SOUL.md`, `.env`, and `.downcity/` runtime data.

If an older project still has `downcity.json`, the CLI can read it once as a legacy migration source; later writes go to the global DB.

## Old structure

```json
{
  "$schema": "./.downcity/schema/downcity.schema.json",
  "id": "my-agent",
  "version": "1.0.0",
  "start": {
    "port": 5314,
    "host": "0.0.0.0"
  },
  "execution": {
    "type": "api",
    "modelId": "default"
  },
  "plugins": {}
}
```

## Key fields

- `id` — stable agent identifier. Used for session storage and runtime directories.
- `start` — runtime startup config (port, host).
- `execution` — binds the project to a model ID from the connected Federation AIService.
- `plugins` — plugin runtime options.

In the new model, these fields are stored in the CLI global DB rather than in a project file.

## What should not live here

- Full model provider definitions (those live in City / Federation AIService).
- Plaintext API keys or bot credentials (those live in the CLI global DB or are injected through env).
- Console-level control plane settings.

## Relationship to SDK

When you use the SDK directly, you pass the same values as constructor options, for example `new Agent({ id, path, model, plugins, config })`. The SDK does not require `downcity.json` to exist in the project directory.
