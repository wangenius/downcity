# 架构总览

Downcity 可以先看成两条主干：

- `console`：全局控制与运维面
- `agent`：单项目运行与执行面

而在 agent runtime 内部，真正承接业务的是三层：

- `service`：主业务流程
- `plugin`：被动扩展层
- `asset`：plugin 依赖资源

## 分层结构

1. **Console 层（全局）**
- 负责 registry、模型池、全局配置、UI gateway

2. **Agent 层（项目）**
- 负责加载单个项目、持有单个 runtime、写入 `.downcity/`

3. **Service 层（核心工作流）**
- 例如 `shell`、`chat`、`task`、`memory`
- 拥有生命周期、工作流和稳定 action

4. **Plugin 层（可选增强）**
- 例如 `skill`、`voice`、`auth`
- 通过 service 预先定义好的扩展点接入
- 负责增强、校验、改写、解析，但不拥有独立生命周期

5. **Asset 层（依赖资源）**
- 负责 install / check / config / use
- 为 plugin 提供模型、二进制、第三方依赖等底层能力
- 应理解为 plugin dependency 基础设施，而不是独立业务流程层

## 一条最重要的边界

- `service` 主动参与 agent 执行周期
- `plugin` 不主动跑，只在 service 需要时被接入

所以：

- 主流程放在 service
- 扩展逻辑放在 plugin
- plugin 依赖资源放在 asset

## 一个直观例子

在聊天场景里：

- `chat` service 负责消息进入、入队、执行、回复
- `voice` plugin 在语音相关节点上补充转写
- `auth` plugin 在鉴权和角色解析节点上接入
- `voice.transcriber` asset 提供实际转写依赖
