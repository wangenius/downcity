# Plugin Configuration

Plugin configuration is easiest to understand in three layers:

- agent-level registration
- plugin runtime options
- Console-facing `setup` and `usage` protocols

## 1. Agent-level registration

Whether a plugin is active is determined by the Agent runtime registry.

Registering a plugin makes it available. Unregistering removes it from the configured registry immediately and stops its lifecycle after active Session step leases release.

## 2. Plugin runtime options

Runtime options for how an agent uses a plugin usually live in:

```json
{
  "plugins": {
    "chat": {
      "channels": {
        "telegram": {
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

This expresses:

- how the current agent uses the plugin
- which runtime options the plugin needs

## 3. `config.scope`

Plugin config definitions support:

- `global`
- `project`

This describes config field scope. The current built-in project plugins use project config for runtime options only.

Many current built-in plugins keep agent-side runtime options at project scope.

## What `setup` and `usage` mean

### `setup`

`setup` is about:

- install
- repair
- dependency preparation

Typical examples:

- install dependencies
- download a model
- install Python dependencies

### `usage`

`usage` is about:

- how the current agent should use that plugin
- which behavior choices should be saved

Typical examples:

- whether to inject prompts
- default model
- default format, language, or speed

## Why `setup` and `usage` are separate

Because they solve different problems:

- `setup` makes the capability exist
- `usage` controls how the current agent uses it

That split lets Console present clearer surfaces for:

- `Install / Repair`
- `Options`

## How plugin config is usually persisted

Host-side plugin settings are generally written back into the host config store.

For the CLI, that means the agent config in the global DB. SDK hosts can pass the same values through `new Agent({ config })` or their own plugin constructors.

The persistence path writes the `plugins` section instead of dumping the entire merged runtime state.

## Recommended mental model

Remember these three lines:

- registration decides whether the plugin is active
- runtime options live under host-managed `plugins.<name>`
- `setup` and `usage` are UI protocol layers
