# Chat Service

`chat` 是负责消息收发链路的 service。

## 职责

- 接收入站渠道事件并做标准化
- 解析 `contextId/chatKey`，写入 history 与 meta
- 把回复/动作下发到具体渠道

## 默认执行策略

- 对低风险且用户意图已经明确的请求，Agent 会优先直接执行，而不是在事件标题、默认平台、显然的时间表达上反复追问。
- 遇到“明天、下午两点、提前两小时”这类时间表达时，Agent 会优先按当前会话时区解析成绝对时间，并在回复里写清楚实际执行时间。
- 如果动作依赖系统权限或第三方连接（例如提醒事项、日历、聊天渠道），Agent 应先探测是否可用；若被权限或连接阻塞，会直接说明阻塞点，而不是先承诺“可以创建”再回头追问。

## 聊天权限判断

- Chat service 收到消息后，会先识别发消息的用户身份与来源平台。
- 是否允许进入 Agent，不看群本身，而是看当前发消息用户所属的权限组。
- 新用户如果还没有显式分组，会自动使用默认权限组。

## 多 Agent 授权隔离

- `chat` 的授权规则是按 agent 隔离的，不是全局共享一套。
- 即使多个 agent 绑定了同一个 channel account，它们的默认权限组、用户角色绑定和授权结果也可以完全不同。
- 换句话说，共享的是 bot 接入凭据，不是授权策略。

完整说明见：[权限说明](/zh/docs/security/permissions)

## 会话显示名（chatTitle）

- Chat service 会在 context meta 中尽量记录渠道会话的显示名（`chatTitle`），用于 Console UI 展示更可读的“群名/频道名/私聊对象名”。
- Telegram 通常可直接拿到群名/频道名；飞书与 QQ 采用 best-effort 策略（受平台事件字段与权限影响，可能缺失）。
