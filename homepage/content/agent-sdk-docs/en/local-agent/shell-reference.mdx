# Shell & Sandbox

The Agent uses `@downcity/shell` for local command execution, sandboxed runners, shell session management, and interactive approval flows.

## Install

Shell is a peer dependency of `@downcity/agent`. When you install the agent, you can add the shell package:

```bash
pnpm add @downcity/shell
```

## Shell

The `Shell` class manages command execution and the approval runtime. Application-level approvals belong to a specific Session.

### Minimal usage

```ts
import { Shell } from "@downcity/shell";
import { Agent } from "@downcity/agent";

const shell = new Shell();

const agent = new Agent({
  id: "demo",
  path: process.cwd(),
  shell,
});
```

When Shell is mounted through `Agent`, no path option is needed; the Agent configures its project root and runtime context. Pass `root_path` only when using Shell independently.

### Methods

`shell.dispose()` — Release shell resources.

Approval requests appear as an `approval-required` Tool Part in `session.subscribe((mutation, reply) => {})`. At the application layer, use `reply.approval(...)`, or `session.approvals()` and `session.resolve_approval(...)`; do not call the Shell Runtime's internal approval methods directly.

## Model tools

A Shell mounted on an Agent provides:

| Tool | Purpose |
|------|---------|
| `shell_exec` | Run one non-interactive command |
| `shell_session` | Manage long-running or interactive PTY commands |
| `read` | Read paginated text; binary files return metadata only |
| `write` | Create UTF-8 files or atomically replace them with `overwrite: true` |
| `edit` | Apply unique, non-overlapping exact-text replacements to one file |

`read` returns up to 500 lines by default, with a maximum of 2,000 lines or 256KB per call. Continue with `truncated` and `next_offset`. `write` accepts up to 1MB and creates parent directories automatically.

`edit` accepts up to ten operations. Every operation is matched against the original file, and the file is written only when all operations are valid. Pass `read.sha256` as `edit.expected_sha256` to reject stale concurrent edits.

All three file tools are fixed to the project root and reject symbolic-link escapes. They never request unrestricted access.

## Sandbox

Shell commands use the safe sandbox by default. The backend is selected automatically based on the host platform:

- `macos-seatbelt` — macOS sandbox
- `linux-bubblewrap` — Linux sandbox
- `unrestricted-host` — No sandboxing

The SDK's internal sandbox runner handles shell subprocess creation, platform-specific backend invocation, and one-shot command execution. You do not normally need to interact with it directly.

### ConfigResolver

`SandboxConfigResolver` derives the fixed safe-sandbox boundary from the Agent project root.

`SandboxPreflight` runs pre-execution checks to validate the sandbox setup.
## Approval

`ShellApprovalRuntime` manages the interactive approval loop. Unrestricted requests use the current Session approval mode; the default `ask` mode requires an explicit approve or deny decision.

## ShellTools

`ShellTools`, `FileTools`, and their schemas provide the AI-ready definitions used by the Agent. You do not normally need to use them directly; they are wired automatically when you pass `shell` to the Agent constructor.
