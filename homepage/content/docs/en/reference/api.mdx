# HTTP API Reference

> User-facing runtime API. For most operations, CLI (`downcity ...`) is recommended first.

## Core endpoints

### `GET /health`

```json
{ "status": "ok", "timestamp": "2026-03-08T12:00:00.000Z" }
```

### `GET /api/status`

```json
{ "name": "downcity", "status": "running", "timestamp": "..." }
```

### `POST /api/execute`

Request body:

```json
{
  "instructions": "Summarize today's changes",
  "chatId": "default"
}
```

## Managed plugin API

### `GET /api/plugins/list`

List managed plugin states.

### `POST /api/plugins/control`

```json
{ "pluginName": "chat", "action": "restart" }
```

Supported `action`: `start | stop | restart | status`.

### `POST /api/plugins/command`

```json
{
  "pluginName": "task",
  "command": "list",
  "payload": { "status": "enabled" }
}
```

## Plugin API

### `GET /api/plugins/catalog`

List registered plugins.

### `POST /api/plugins/availability`

```json
{ "pluginName": "sound" }
```

### `POST /api/plugins/action`

```json
{
  "pluginName": "sound",
  "actionName": "models",
  "payload": { "capability": "asr" }
}
```

## Console UI API (summary)

### `GET /api/ui/plugins`

Returns plugin summary plus availability state (used by Console UI panels).

Example:

```json
{
  "success": true,
  "plugins": [
    {
      "name": "sound",
      "actions": ["models", "asr", "tts"],
      "pipelines": ["chat.augmentInbound"],
      "guards": [],
      "effects": [],
      "resolves": [],
      "hasSystem": true,
      "hasAvailability": true,
      "availability": {
        "enabled": true,
        "available": true,
        "reasons": []
      },
      "config": {
        "actions": [
          {
            "name": "models",
            "supportsCommand": false,
            "supportsApi": false,
            "apiMethod": "",
            "apiPath": ""
          }
        ]
      }
    }
  ]
}
```

### `POST /api/dashboard/sessions/:sessionId/execute`

Execute one turn in a specific session (for example a selected chatKey).

Request body example with attachment:

```json
{
  "instructions": "Please read the attached markdown and provide an actionable summary.",
  "attachments": [
    {
      "type": "document",
      "fileName": "page.md",
      "contentType": "text/markdown; charset=utf-8",
      "content": "# Captured Web Page\n\n..."
    }
  ]
}
```

Notes:

- `sessionId` should be the exact target session returned by agent. For chat sessions, this is an opaque ID managed by the chat plugin (not a parseable platform key).
- Attachments are materialized by agent and injected as `<file ...>` tags before the user instruction.
- If `sessionId` maps to a chat conversation, execution is enqueued into the same chat plugin queue (same behavior as platform inbound messages), and chat history is appended before execution.
- Chat-context calls return immediately with queue metadata (`queued=true`, `queueItemId`, `queuePosition`).

## Action routes

Plugin actions use:

- `/api/plugins/action`
- `/api/plugins/control`
- `/api/plugins/command`

## Debug baseline

Make sure the agent is running before calling APIs. Runtime startup belongs to the [Agent start workflow](/en/docs/reference/cli); the health endpoint belongs to the running Agent HTTP surface.
