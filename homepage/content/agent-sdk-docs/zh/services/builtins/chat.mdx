# chat 内建 Service

`chat` 是负责聊天渠道运行时的内建 service。

在当前 runtime 里，它承载 Telegram、Feishu、QQ 的消息入口与回复分发。

## 它长期持有什么

从当前实现看，`chat` 会长期持有这些运行时对象：

- `channelState`
- `queueWorker`
- `queueStore`

这些都是实例级状态，不是一次调用时临时传进去的参数。

## 为什么它是 service，而不是 tool

`chat` 不是一个简单的“发消息函数”。

它需要长期维护：

- 渠道连接
- queue 状态
- 跨多个 session 稳定存在的回复分发行为

这就是非常典型的 service。

## 它和公开 SDK 的关系

这是当前最清晰、已经有显式 SDK 构造器入口的内建 service：

```ts
import { Agent, ChatService } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  tools: {},
  services: [
    new ChatService({
      telegram: {
        botToken: process.env.TELEGRAM_BOT_TOKEN!,
      },
    }),
  ],
});
```

所以这两页各自负责的内容可以这样理解：

- 这页解释内建 `chat` 在 runtime 里的角色
- [ChatService](/zh/agent-sdk-docs/services/chat-service) 解释显式 SDK 构造器与配置面

## 最适合的使用场景

当你需要这些能力时，就应该用 `chat` 这类 service 的思路来理解：

- 让渠道 bot 由 agent 进程长期持有
- 需要一个稳定的 queue worker 边界
- 需要多个 session 共享同一个聊天运行时实例
- 需要 service 级 system 规则来约束聊天行为

## 启动边界

当前 SDK 里，最稳定的自动启动边界仍然是：

- `agent.http.start()`
- `agent.rpc.start()`

service lifecycle 的启动，目前主要是在这里被有意触发的。
