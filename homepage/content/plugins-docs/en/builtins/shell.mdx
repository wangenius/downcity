# `shell` Plugin

`shell` is the built-in plugin adapter for the `@downcity/shell` package.

It provides:

- one-shot shell execution
- persistent shell sessions
- status, read, write, wait, and close actions

## Main shape

- `lifecycle`
- `actions`

## What it is for

Use `shell` when you need:

- shell sessions backed by `@downcity/shell`
- per-agent shell state boundaries
- a runtime action layer around shell execution

## How users use it

There is no root `town shell` shortcut. Use the plugin command forms:

```bash
town plugin command shell exec --payload '{"cmd":"pwd"}'
town plugin command shell start --payload '{"cmd":"pnpm test","inlineWaitMs":1000}'
town plugin command shell status --payload '{"shellId":"<shellId>"}'
town plugin command shell read --payload '{"shellId":"<shellId>","fromCursor":0}'
town plugin command shell write --payload '{"shellId":"<shellId>","chars":"q\n"}'
town plugin command shell wait --payload '{"shellId":"<shellId>","timeoutMs":5000}'
town plugin command shell close --payload '{"shellId":"<shellId>","force":true}'
```

`town plugin command ...` targets a managed agent runtime. For local direct execution inside the current project, the advanced form is `town plugin action shell <action> --payload '<json>'`.

## SDK action use

You can instantiate it directly and optionally override runtime parameters. If omitted, the built-in defaults are used:

```ts
import { ShellPlugin } from "@downcity/plugins";

const shell = new ShellPlugin({
  maxActiveShells: 64,
  defaultExecTimeoutMs: 60_000,
  defaultWaitTimeoutMs: 10_000,
  cleanupDelayMs: 10 * 60 * 1000,
});
```

```ts
await agent.plugins.runAction({
  plugin: "shell",
  action: "exec",
  payload: {
    cmd: "pwd",
  },
});
```

The action names are `exec`, `start`, `status`, `read`, `write`, `wait`, and `close`.

## Important semantics

- shell sessions are implemented by `@downcity/shell`; `ShellPlugin` binds that package to one agent instance
- constructor parameters are optional runtime tuning values; defaults preserve the built-in behavior
- sandbox backends also live in `@downcity/shell`. Each command is spawned inside the current agent sandbox, whose persistent directory is `.downcity/sandbox`
- `shellId` only identifies the session log and process lifecycle under `.downcity/shell/<shellId>`; it does not decide sandbox HOME, cache, or permissions
- Sandbox commands can read and write the project, use network by default, and share `.downcity/sandbox` as `HOME`
- Sandbox commands cannot write the real user home or host system locations such as `/usr/local`, `/opt/homebrew`, and `/System`
- `start` lifecycle binds runtime
- `stop` lifecycle closes all shell sessions and clears in-memory state

## Public status

`ShellPlugin` remains importable from `@downcity/plugins`. The command execution, sandbox backend, session lifecycle, and approval primitives live in `@downcity/shell`.
