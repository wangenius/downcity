title: "Accounts 服务"
description: "用官方账号服务提供注册、登录、OAuth 入口、会话和 user_token 签发能力。"
---

`@downcity/services` 适合还不想单独维护完整账号系统的产品。它把最常见的注册、登录、会话和 OAuth 入口打包成一个官方服务。

它的重点不是“多一个 SDK”，而是给 City 增加一个现成的 `accounts` service，让产品可以直接通过 `User Gate` / `Admin Gate` 去调用。

## 这组文档怎么读

建议按下面顺序读：

1. 先读 [注册与登录](/docs/city/packages/services/accounts/register-and-login)，理解 guest -> 登录成功 -> `user_token` -> 用户态调用这条主链路。
2. 再读 [OAuth 与会话](/docs/city/packages/services/accounts/oauth-and-session)，理解第三方登录、session 和回跳结果。
3. 如果你要接 Google 登录，继续读 [Google 登录配置](/docs/city/packages/services/accounts/google-login)。
4. 如果你要接微信网站登录，继续读 [微信登录配置](/docs/city/packages/services/accounts/wechat-login)。
5. 最后读 [管理侧操作](/docs/city/packages/services/accounts/admin-operations)，理解可信环境里的用户、session 和账号管理动作。

## 它提供什么

- 注册与登录
- 邮箱验证
- GitHub / Google / WeChat OAuth 入口
- better-auth 认证事实表
- 产品侧 profile 资料表
- 登录后返回 City `user_token`

## 它和 `@downcity/city` 的关系

把两者分开看最清楚：

- `@downcity/services`
  把 `accounts` 这组 service action 注册到 City
- `@downcity/city`
  让前端、App、服务或可信后端去调用这些 action

也就是说，这个服务负责“把账号能力接进 City”；真正的调用方式仍然统一走：

```ts
guest.service("accounts").action("login").invoke(...)
user.service("accounts").get("me")
admin.service("accounts").get("users")
```

## 什么时候用它

- 你的产品想先跑通最小账号系统
- 你想让注册/登录之后直接得到 `user_token`
- 你不想自己先建认证表、profile 表和服务路由

## 安装

```bash
pnpm add @downcity/services
```

## 启用方式

```ts
import { City } from "@downcity/city";
import { accountsService } from "@downcity/services";

const base = new City();

base.use(accountsService({
  token_ttl: "7d",
}));
```

## 它会暴露什么 service

最核心的是 `accounts` service。常见路由包括：

```txt
POST /v1/accounts/register
POST /v1/accounts/login
GET  /v1/accounts/providers
POST /v1/accounts/oauth/start
GET  /v1/accounts/oauth/result
GET  /v1/accounts/oauth/callback
GET  /v1/accounts/me
POST /v1/accounts/logout
GET  /v1/accounts/users
GET  /v1/accounts/sessions
```

其中第三方 OAuth 回跳地址固定是 `/v1/accounts/oauth/callback`。发起 `oauth/start` 时不需要产品前端自己传 `redirect_uri`，服务会按当前 City 的公网域名自动生成。

其中：

- `register`、`login`、`providers`、`oauth/start`、`oauth/result` 是 guest 入口
- `me`、`logout` 是登录后入口
- 管理侧还可以读取 users、sessions 这类数据

## 它会维护哪些表

这套服务分成两层数据：

- better-auth 认证事实表：`auth_users`、`auth_accounts`、`auth_sessions`、`auth_verifications`
- 产品资料表：`auth_profiles`
- OAuth 轮询 state 表：`service_accounts_oauth_states`

前一层负责“这个用户是谁、通过什么方式登录、当前 session 是什么”；后一层负责“产品真正要展示的 profile”。

如果产品前端需要先判断“现在到底开了哪些登录入口”，应该先请求：

```ts
const providers = await guest.service("accounts").get("providers");
```

再根据返回结果决定是否展示 GitHub、Google、WeChat 或邮箱登录入口。不要在产品端把 provider 列表写死。

## 常见调用

```ts
const session = await guest.service("accounts").action("login").invoke({
  email: "user@example.com",
  password: "password123",
  bay_id: "bay_demo",
});
```

登录或注册成功后，返回结果里会包含 `user_token`。产品 client 拿到 token 之后，再用常规 `User Gate` 去调 AI service。

`GET /v1/accounts/me` 返回的不只是 `user_token` 解出来的运行时用户，还会附带：

- `profile`：当前产品层资料

## 最常见的产品接法

```ts
const guest = new Gate({
  role: "user",
  city_url: "https://base.example.com",
});

const session = await guest.service("accounts").action("login").invoke<{
  user_token: string;
}>({
  email: "user@example.com",
  password: "password123",
  bay_id: "bay_demo",
});

const user = new Gate({
  role: "user",
  city_url: "https://base.example.com",
  bay_id: "bay_demo",
  user_token: session.user_token,
});

const reply = await user.ai.text({
  prompt: "写一段欢迎语",
});
```

这也是为什么 `accountsService()` 和 `@downcity/city` 通常是一起看的：一个提供账号能力，一个提供调用入口。
