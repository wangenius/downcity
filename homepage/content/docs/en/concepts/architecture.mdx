# Architecture Overview

Downcity is easiest to reason about with two top-level trunks:

- `console`: global control and operations plane
- `agent`: project runtime execution plane

Inside the agent runtime, `service`, `plugin`, `capability`, and `asset` form the execution layer.

## Layered Structure

1. **Console Layer (global)**
- Commands: `city console ...`, `city console config ...`
- Responsibility: process management, registry, model pool, global storage

2. **Agent Layer (project)**
- Commands: `city agent create/start/status/restart ...`
- Responsibility: load one project, own one runtime, persist traces in `.ship/`

3. **Service Layer (core workflows)**
- Stable business domains such as `chat`, `skill`, `task`, `memory`
- Own orchestration and user-facing workflows

4. **Plugin Layer (optional enhancements)**
- Optional modules such as `voice`
- Expose actions, capabilities, hooks, and system prompt fragments

5. **Capability + Asset Layer**
- `capability`: stable runtime contract such as `audio.transcribe`
- `asset`: installable dependency or runtime resource required by plugins

## Architecture Rules

- Separate control from execution: `console` governs, `agent` executes.
- Separate workflow from enhancement: `service` owns business flow, `plugin` augments it.
- Separate contract from implementation: services call capabilities, not plugin internals.
- Separate behavior from dependency: plugin declares required assets, asset owns install/check/config/use.
