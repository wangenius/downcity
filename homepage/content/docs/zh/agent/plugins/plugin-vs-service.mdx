# Plugin 和 Service 的区别

这是最容易混淆的一组概念。

## 一句话版

- plugin 更像扩展层
- service 更像长期运行实例

## 更适合写 plugin 的场景

- 你要增加一组 action
- 你要插入 hooks
- 你要提供 resolve 点
- 你要注入 system 文本
- 你要给 Console 提供 setup / usage 协议
- 你要注入某组 runtime HTTP 路由

## 更适合写 service 的场景

- 你要长期持有状态
- 你要 worker
- 你要 session / connection 池
- 你要后台运行对象
- 多个 session 要共享同一个长期实例

## 它们可以怎么配合

很常见的一种组合是：

- plugin 负责扩展点和能力声明
- service 负责长期运行态

例如：

- plugin 通过 hook 接到 chat 链路
- 真正长期存在的聊天运行态则由 `ChatService` 承载

## 一个经验法则

如果你问自己：

- “我要不要把这段逻辑做成 plugin？”

那就先看它是否主要在回答这两个问题：

- 我需要往现有链路插点吗？
- 我需要对外暴露一组能力声明吗？

如果是，更像 plugin。

如果你问自己：

- “我要不要把这段逻辑做成 service？”

那就先看它是否主要在回答：

- 我需要一个长期活着的实例吗？

如果是，更像 service。

## 当前最稳的设计顺序

1. 先判断是不是 plugin 扩展点问题
2. 再判断是不是 service 长期状态问题
3. 如果两者都需要，就拆成 plugin + service 协作
