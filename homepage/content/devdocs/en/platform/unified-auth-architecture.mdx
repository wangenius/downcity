# Unified Auth Architecture

This page describes the target authentication model for server deployment.

The design is intentionally expressed as a single system rather than multiple parallel schemes.

## Core Position

Downcity should use one account system as the only identity source.

That means:

- users log in through one auth entrypoint
- the server issues one bearer-token style credential
- protected APIs validate that token and resolve the user behind it
- browser UI, CLI, Chrome Extension, and automation are all just clients of the same system

## Why this is the preferred direction

- it removes the current “local control plane with no real auth” risk
- it avoids splitting product auth into separate local-only and server-only systems
- it gives us one permission model for all clients
- it is simple enough to implement now and strong enough to evolve later

## V1 Scope

V1 only needs:

- one admin bootstrap flow
- one login API
- one bearer token model
- one RBAC + scope layer
- one audit log path

V1 does not need:

- client-side public/private key pairs
- separate browser session and PAT systems as different first-class products
- multiple independent auth stacks

## Request Model

All protected requests should converge to:

```http
Authorization: Bearer <token>
```

Server-side flow:

1. verify token hash
2. resolve user
3. evaluate role and scopes
4. allow or deny
5. write audit log for sensitive actions

## Deployment Boundary

Recommended boundary:

- Gateway is the only external entrypoint
- Runtime stays behind Gateway
- Runtime should not be directly exposed to the public network

## Client Coverage

This single model should cover:

- Console UI
- CLI
- Chrome Extension
- automation scripts

They are not separate auth systems. They are different clients of the same account system.

## Suggested Data Model

Minimum tables:

- `auth_users`
- `auth_roles`
- `auth_permissions`
- `auth_user_roles`
- `auth_role_permissions`
- `auth_tokens`
- `auth_audit_logs`

## Planned Module Area

Suggested implementation root:

- `packages/downcity/src/main/auth/`

Suggested responsibilities:

- login and bootstrap
- bearer-token verification
- route protection policy
- permission evaluation
- auth storage
- audit logging

## Final Rule

The long-term system should not evolve into:

- one local token system
- one server account system
- one browser-only session model

Instead it should remain:

- one account system
- one login model
- one bearer-token access model
- one permission model
