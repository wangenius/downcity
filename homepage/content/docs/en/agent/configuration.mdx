# Agent Configuration

An Agent is configured through the Agent SDK. The CLI (`downcity`) is a convenience wrapper that reads `downcity.json` from the project directory and passes those values to the Agent SDK constructor.

## Constructor options

When you create an Agent directly with `@downcity/agent`, configuration is passed to the constructor:

```ts
import { Agent } from "@downcity/agent";

const agent = new Agent({
  id: "repo-helper",
  path: "/path/to/project",
  model: myModel,
  tools: [myTool],
  plugins: [myPlugin],
  env: { ... },
});
```

Key fields:

- `id` — stable agent identifier. Used for session storage and runtime directories.
- `path` — project directory. The Agent reads `PROFILE.md`, `SOUL.md`, and `.env` from here.
- `model` — the live model instance. Passed directly, not looked up from a catalog.
- `tools` — array of tool instances available to the agent.
- `plugins` — array of plugin instances to load at startup.
- `env` — host-provided environment variables, overlaid with project `.env`.

You can also pass `start` (port, host) and `execution` options if you want the Agent to start its own HTTP surface.

## Project configuration file: downcity.json

The CLI (`downcity agent start`) looks for `downcity.json` in the project directory and passes its contents to the Agent SDK constructor. It is essentially a serialized form of the constructor options.

The minimal structure:

```json
{
  "$schema": "./.downcity/schema/downcity.schema.json",
  "id": "my-agent",
  "version": "1.0.0",
  "start": {
    "port": 5314,
    "host": "0.0.0.0"
  },
  "execution": {
    "type": "api",
    "modelId": "default"
  },
  "plugins": {}
}
```

Key fields:

- `id` — same as constructor `id`.
- `start` — runtime startup config (port, host). CLI flags override this if provided.
- `execution` — binds the project to a model ID from the connected Federation AIService. `type` is always `api`.
- `plugins` — plugin-level configuration. For example, `plugins.chat.channels` binds chat channels to global accounts.

What should not live here:

- Full model provider definitions (those live in City / Federation AIService).
- Plaintext API keys or bot credentials (those live in `~/.downcity/downcity.db` or are injected through env).
- Console-level control plane settings.

The point of this file is to declare how the project should be assembled at runtime, not to store everything.

## PROFILE.md

`PROFILE.md` defines the agent's visible role:

- who the agent is
- what it is responsible for
- how it should answer
- which boundaries it should respect

Think of it as the role definition rather than runtime configuration. If you want to shape output style, priorities, or collaboration tone, this is the first place to do it.

## SOUL.md

`SOUL.md` defines deeper, more stable operating principles. It is closer to foundational working style than a one-off role prompt. Change it only when you truly want to shift the agent's long-term behavior.

In a healthy project, `SOUL.md` should normally change less often than `PROFILE.md`.

## Environment variables

The Agent merges two env layers at startup:

1. Host-provided env (from `new Agent({ env })` or the CLI runtime).
2. Project-root `.env` (overlays the host layer).

Runtime metadata vars such as `DC_AGENT_ID`, `DC_AGENT_PATH`, and `DC_SESSION_ID` are injected for the current command context, not written back into the shell session.

## Model binding

When using the CLI, the project binds to a model through `downcity.json`:

```json
{
  "execution": {
    "type": "api",
    "modelId": "quality"
  }
}
```

The model ID must exist in the connected Federation AIService catalog. The project does not store provider keys or base URLs. When using the SDK directly, the model is passed directly to the constructor or session.

## Chat channels

Chat channels are configured under `plugins.chat.channels`. The project only binds a global account ID:

```json
{
  "plugins": {
    "chat": {
      "channels": {
        "telegram": {
          "enabled": true,
          "channelAccountId": "telegram-main"
        }
      }
    }
  }
}
```

The actual bot credentials live in `~/.downcity/downcity.db` (managed by the `downcity chat` CLI or Console UI). The project only says which account to use.
