# 任务定义与 Action 协议

## 任务定义文件

```text
.downcity/task/<taskId>/task.md
```

`taskId` 由 `title` 归一化生成，`title` 需要保持唯一。

## Frontmatter 字段

| 字段 | 必填 | 说明 |
| --- | --- | --- |
| `title` | 是 | 任务名称（唯一标识） |
| `description` | 是 | 任务目标描述 |
| `sessionId` | 是 | 执行结果要回发到的会话标识 |
| `when` | 是 | 触发条件：`@manual` / cron / `time:ISO8601` |
| `status` | 是 | `enabled` / `paused` / `disabled` |
| `kind` | 否 | `agent`（默认）或 `script` |
| `review` | 否 | 仅 `kind=agent` 生效；`true` 时启用模拟用户多轮复核，默认单轮 |

## `when` 语义

- `@manual`：仅手动触发。
- cron 表达式：按周期触发。
- `time:2026-03-08T10:30:00+08:00`：一次性触发（必须带时区）。

one-shot 触发成功后，任务会自动：
- `status -> paused`
- `when -> @manual`

## Action 一览

| Action | 作用 | 关键结果 |
| --- | --- | --- |
| `list` | 列出任务定义 | `tasks[]` |
| `create` | 新建任务定义 | 可能 `reusedExisting=true` |
| `update` | 更新任务定义 | 写回 `task.md` |
| `run` | 手动触发一次执行 | 立即返回 `accepted=true` + `executionId` |
| `status` | 显式改状态 | `enabled/paused/disabled` |
| `enable/disable` | 状态快捷命令 | 等价状态更新 |
| `delete` | 删除任务与历史目录 | 返回 task 目录路径 |

## create 去重逻辑

`create` 只按 `title` 精确去重：

- 已有同名 `title`：返回复用（`reusedExisting=true`）。
- 无同名：创建新任务。

不会再做“语义近似”去重。

## run 的协议语义

`run` 是异步受理：

1. 校验任务存在。
2. 后台启动 `runTaskNow(...)`。
3. 立即返回：`accepted=true`、`message`、`executionId`。

其中 `message` 会明确告诉调用方：
- 任务已经开始执行
- 完成后 task 会自动发送给用户
- 可以直接继续后续流程
- 无需等待 task 完成

完成后的自动发送规则：
- 默认通过 chat service `send` action，把 task 的**最终结果正文**发回 `sessionId` 绑定的 chat。
- 因此正常情况下，不需要在 task 正文里重复要求执行器再调用一次 `city chat send`。
- 只有在跨会话、多目标通知、额外抄送等场景下，才应显式写入发送动作。

调用方行为约束：
- 不要在 `accepted=true` 后手动把 task 产出再转发一次。
- 不要主动读取 run 目录、`run-progress.json`、`output.md` 做轮询。
- 只有用户明确要求排查或查看产物时，才去检查 run 目录。

`agent` 任务当前规则：
- 默认单轮执行。
- 仅当 `review=true` 时启用模拟用户多轮复核。
- 默认只要求产生有效输出文本，不再默认要求 `chat_send` 成功。

## task 正文怎么写

推荐把 `body` 写成稳定结构，而不是一句模糊指令。

### agent task 推荐结构

```md
# 任务目标

- 明确这次任务最终要交付什么。

# 背景与输入

- 补充上下文、数据来源、范围、约束和假设。

# 执行步骤

1. 先理解目标与完成标准。
2. 再收集信息或执行操作。
3. 把关键中间产物写入 run 目录。
4. 输出最终结果正文。

# 输出要求

- 直接写最终结果，不要附带冗长日志。
- 如需表格 / JSON / Markdown，在这里写清楚。
- 默认不要重复调用 `city chat send`。

# 触发与状态建议

- 草稿或试运行：`@manual` + `paused`
- 稳定周期任务：cron + `enabled`
- 一次性任务：`time:<带时区 ISO 时间>`

# 注意事项

- 只在需要跨会话或额外通知时，才显式发送消息。
```

### script task 推荐结构

```bash
# 任务目标：一句话说明脚本要做什么
# 输入依赖：列出环境变量、文件、命令依赖
# 失败条件：列出应该 exit 1 的情况

set -euo pipefail

# 1. 准备输入

# 2. 执行主逻辑

# 3. 输出最终结果（这段输出会被直接发送给用户）
```

## 默认建议

- 如果用户没有明确要求自动触发，优先创建成 `@manual` + `paused`。
- task 正文应该描述“最终结果长什么样”，而不只是“去做一下”。
- 关键中间产物尽量落到 run 目录，便于后续排查。

## CLI 与 API

CLI：

```bash
city task run daily-check --reason "manual"
```

API：

```bash
city service command task run --payload '{"title":"daily-check","reason":"manual"}'
```
