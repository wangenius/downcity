`@downcity/services` is a good fit when you do not want to build a full account system first. It packages the common registration, login, session, and OAuth entry flows into one official service.

The important point is not "another SDK." It adds a ready-made `accounts` service into City, so studio-side callers can use it through `User Gate` and `Admin Gate`.

## How to read this section

The recommended order is:

1. Start with [Registration and login](/en/docs/city/packages/services/accounts/register-and-login) to understand the main guest -> login success -> `user_token` -> logged-in user flow.
2. Continue with [OAuth and sessions](/en/docs/city/packages/services/accounts/oauth-and-session) for third-party login, session behavior, and callback results.
3. If you want Google login, continue with [Google Login Setup](/en/docs/city/packages/services/accounts/google-login).
4. If you want WeChat website login, continue with [WeChat Login Setup](/en/docs/city/packages/services/accounts/wechat-login).
5. Finish with [Admin operations](/en/docs/city/packages/services/accounts/admin-operations) for trusted-side user, session, and account-management workflows.

## What it provides

- registration and login
- email verification
- GitHub, Google, and WeChat OAuth entry points
- better-auth auth tables
- studio-facing profile table
- City `user_token` issuance after login

## Relationship with `@downcity/city`

The cleanest way to separate the two is:

- `@downcity/services`
  registers the `accounts` service and its actions into City
- `@downcity/city`
  gives frontends, apps, services, and trusted backends a unified way to call those actions

So this service owns "adding auth capability into City," while the SDK still owns the actual calling pattern:

```ts
guest.service("accounts").action("login").invoke(...)
user.service("accounts").get("me")
admin.service("accounts").get("users")
```

## Use it when

- you want the fastest path to a minimal account system
- you want registration and login to return `user_token` directly
- you do not want to design auth tables, profile tables, and routes yourself first

## Install

```bash
pnpm add @downcity/services
```

## Enable it

```ts
import { City } from "@downcity/city";
import { accountsService } from "@downcity/services";

const base = new City();

base.use(accountsService({
  token_ttl: "7d",
}));
```

## Which service it exposes

The central service is `accounts`. Common routes include:

```txt
POST /v1/accounts/register
POST /v1/accounts/login
GET  /v1/accounts/providers
POST /v1/accounts/oauth/start
GET  /v1/accounts/oauth/result
GET  /v1/accounts/oauth/callback
GET  /v1/accounts/me
POST /v1/accounts/logout
GET  /v1/accounts/users
GET  /v1/accounts/sessions
```

The third-party OAuth return URL is fixed at `/v1/accounts/oauth/callback`. Studio clients do not need to pass `redirect_uri` into `oauth/start`; the service generates it automatically from the current public City origin.

As a first mental model:

- `register`, `login`, `providers`, `oauth/start`, and `oauth/result` are guest-access entry points
- `me` and `logout` are logged-in entry points
- admin-side callers can also inspect users and sessions

## Which tables it maintains

The service now keeps two layers of data:

- better-auth auth tables: `auth_users`, `auth_accounts`, `auth_sessions`, `auth_verifications`
- studio-facing table: `auth_profiles`
- oauth polling table: `service_accounts_oauth_states`

The first layer answers auth questions. The second layer answers studio-facing display questions.

If the studio frontend needs to decide which sign-in buttons to render, it should call this first:

```ts
const providers = await guest.service("accounts").get("providers");
```

Then render email, GitHub, Google, or WeChat only from the server response. Do not hardcode the provider list in the studio client.

## Common call pattern

```ts
const session = await guest.service("accounts").action("login").invoke({
  email: "user@example.com",
  password: "password123",
  studio_id: "studio_demo",
});
```

Successful registration or login returns `user_token`. After that, the studio client can switch to a normal `User Gate` for AI calls.

`GET /v1/accounts/me` also returns the current `profile`, so studio-side callers can render account data directly from one endpoint.

## The most common studio-side integration

```ts
const guest = new Gate({
  role: "user",
  city_url: "https://base.example.com",
});

const session = await guest.service("accounts").action("login").invoke<{
  user_token: string;
}>({
  email: "user@example.com",
  password: "password123",
  studio_id: "studio_demo",
});

const user = new Gate({
  role: "user",
  city_url: "https://base.example.com",
  studio_id: "studio_demo",
  user_token: session.user_token,
});

const reply = await user.ai.text({
  prompt: "Write a welcome message",
});
```

This is why `accountsService()` and `@downcity/city` are usually read together: one provides the auth capability, the other provides the calling surface.
