City 是 `@downcity/city` 导出的访问层。产品侧通过它接入 Federation。

它不只是“调一下 AI”的小工具，而是把这几类调用统一收口到一个产品侧入口里：

- `User City`：终端产品调用 AI service、custom service 和公开 service
- `Admin City`：可信环境管理 city、签发 `user_token`、维护 runtime env
- `service(name)`：统一调用你自己注册的 service，或官方服务暴露出来的接口

如果你要做的是“前端产品如何接 City”，这页应该是 package 层的第一入口。

## 两类客户端

| 客户端 | 给谁用 | 负责什么 |
| --- | --- | --- |
| `User City` | 前端、App、扩展、桌面端 | 拿 `city_id + user_token` 调 AI service、custom service、公开 service |
| `Admin City` | 产品后端、管理脚本、内部工具 | 管 city、签 token、写 runtime env、调管理侧 service |

## 什么时候该用它

- 你的产品已经拿到了 `city_id + user_token`
- 你要让产品前端去调用 `text()`、`stream()`、`image()`、custom service 或 service
- 你要在可信后端里签发 token 或写 City 的管理数据
- 你想把注册、登录、OAuth、usage、payment 这些能力也放到同一套调用模型里

## 安装

```bash
pnpm add @downcity/city
```

## 这个包到底管什么

把它理解成三层最清楚：

1. `client.ai.*`
   这是 AIService 的高层调用入口，适合 `text()`、`stream()`、`image()`、`video()` 这类 AI 能力。
2. `client.service(name).action(name).invoke()`
   这是通用 Action 调用入口，适合自定义 service，也适合服务暴露出来的业务 Action。
3. `client.service(name).get(path)`
   这是 GET Action 调用入口，适合 OAuth 轮询、列表查询、当前用户信息等读取型接口。

也就是说，City 实际上统一承载了：

- AI 调用
- 你自己写的业务 service 调用
- 官方服务 service 调用
- city / token / env 管理调用

## 四种最常见调用

### 1. AI service

```ts
const result = await client.ai.text({
  model: "deepseek-v4-flash",
  prompt: "你好 Downcity",
});
```

适合模型推理、图片、视频、语音等 AI 能力。

### 2. Custom service

```ts
const rewritten = await client
  .service("rewrite")
  .action("formal")
  .invoke<{ text: string }>({
    prompt: "把这段话改得更专业",
  });
```

适合你自己在 City 里注册的业务 service。

### 3. 官方服务

```ts
const session = await guest.service("accounts").action("login").invoke({
  email: "user@example.com",
  password: "password123",
  city_id: "city_demo",
});

const usage = await user.service("usage").get("me");
```

适合 `@downcity/services`、`@downcity/services`、`@downcity/services` 这类官方服务。

### 4. Trusted-side management

```ts
const issued = await admin.cities.tokens.apply({
  city_id: "city_demo",
  user_id: "user_123",
  ttl: "7d",
});
```

适合 city、token、runtime env 这些必须在可信环境完成的动作。

## 什么是 service

service 不是一套单独的协议。

对 `@downcity/city` 来说，它和你自己注册的 service 本质上是同一件事：都是 City 里挂在统一 `/v1/*` 路由空间下的 service，只是来源不同。

- 你自己写的 service：例如 `rewrite`、`translate`、`workspace`
- 官方服务带来的 service：例如 `accounts`、`usage`、`payment`、`payment.stripe`

所以产品侧不需要学习两套调用方式。统一就是：

```ts
client.service("service_id").action("action_id").invoke(...)
client.service("service_id").get("path", query)
```

## `User City` 最小示例

```ts
import { City } from "@downcity/city";

const client = new City({
  role: "user",
  federation_url: "https://base.example.com",
  city_id: "city_demo",
  user_token: "ub_xxx",
});

const models = await client.ai.listModels();

const result = await client.ai.text({
  model: models.primary(),
  prompt: "你好 Downcity",
});
```

这个阶段假设你已经有：

- `federation_url`
- `city_id`
- `user_token`

如果你还没有 `user_token`，通常要先走下面的登录链路。

## `Admin City` 最小示例

```ts
import { City } from "@downcity/city";

const admin = new City({
  role: "admin",
  federation_url: "https://base.example.com",
  admin_secret_key: process.env.DOWNCITY_FEDERATION_ADMIN_SECRET_KEY,
});

const city = await admin.cities.create({
  name: "Demo City",
});

const token = await admin.cities.tokens.apply({
  city_id: city.city_id,
  user_id: "user_123",
  ttl: "30m",
});
```

## 注册登录怎么串起来

最常见的是两种链路。

### 模式 A：你自己有业务后端

```txt
前端提交登录信息
  -> 你的业务后端完成账号校验
  -> 业务后端用 Admin City 向 City 申请 user_token
  -> 后端返回 city_id + user_token
  -> 前端创建 User City
  -> 前端开始调用 client.ai.* 或 client.service(...)
```

这种模式适合你已经有成熟账号系统。

### 模式 B：直接用 accounts 服务

```txt
前端先创建 guest User City（只有 federation_url）
  -> 调 guest.service("accounts").action("register" / "login")
  -> accounts service 返回 user_token
  -> 前端再创建正常 User City（federation_url + city_id + user_token）
  -> 前端继续调用 AI service 或其他 service
```

这种模式适合想先跑通最小注册登录系统的产品。

## 用 accounts 服务做登录

```ts
const guest = new City({
  role: "user",
  federation_url: "https://base.example.com",
});

const session = await guest.service("accounts").action("login").invoke<{
  user_token: string;
  user_id: string;
  email: string;
}>({
  email: "user@example.com",
  password: "password123",
  city_id: "city_demo",
});

const user = new City({
  role: "user",
  federation_url: "https://base.example.com",
  city_id: "city_demo",
  user_token: session.user_token,
});

const reply = await user.ai.text({
  prompt: "给新用户写一段欢迎语",
});
```

这里要注意：

- `register`、`login`、`oauth/start`、`oauth/result` 这类入口通常是免登录 Action
- 它们仍然通过 `User City` 调，但这时只需要 `federation_url`
- 真正进入用户态调用后，才需要 `city_id + user_token`

## 自定义 service 与服务路由

除了 `client.ai.*` 之外，产品和管理端都可以用 `service(name)` 走统一 `/v1/*` 路由：

```ts
const session = await client.service("accounts").action("login").invoke({
  email: "user@example.com",
  password: "password123",
  city_id: "city_demo",
});
```

再看几个更完整的例子：

```ts
// 你自己的业务 service
const draft = await client.service("writer").action("draft").invoke({
  topic: "Downcity onboarding",
});

// accounts service
const me = await client.service("accounts").get("me");

// usage service
const usage = await client.service("usage").get("me");

// 先读支付方式
const methods = await client.payment.methods();

// trusted-side route
const payments = await admin.service("payment").get("payments");

// 再按具体支付方式发起支付
const checkout = await client.payment.method("stripe").invoke({
  topup_id: "topup_demo",
});
```
