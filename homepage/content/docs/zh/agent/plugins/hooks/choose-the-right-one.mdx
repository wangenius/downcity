# 怎么选

这一页只回答一个问题：

- 当前这个需求，到底该放哪一层

## 先看最短判断

如果你想：

- 主动调用一个明确能力，用 `action`
- 把一个值逐步加工，用 `pipeline`
- 在继续前做拦截判断，用 `guard`
- 不改主值，只做记录或同步，用 `effect`
- 拿到一个最终唯一答案，用 `resolve`
- 持有长期状态、连接、worker，用 `service`

## 一组常见问题

### “我想给输入补默认字段”

用 `pipeline`。

因为你是在改值。

### “我想没有权限就直接拦住”

用 `guard`。

因为你是在决定能不能继续。

### “我想每次都顺手写一条审计日志”

用 `effect`。

因为你只是在做副作用。

### “我想查这个用户最终是什么角色”

用 `resolve`。

因为你要的是唯一答案。

### “我想让外部显式执行一个命令”

用 `action`。

因为你要的是一个可直接调用的能力入口。

### “我想维护一个长期缓存、连接池或后台 worker”

用 `service`。

因为这已经是长期运行实例，不再只是 plugin 扩展点。

## 最常见的误分层

### 用 `resolve` 做多插件协同增强

这通常是错的，应该回到 `pipeline`。

### 用 `effect` 偷偷承担可靠任务队列

这通常太重了，应该拆到 `service` 或更完整的后台能力。

### 用 `guard` 顺手改值

这会让语义变混乱，改值应该放回 `pipeline`。

## 一个很实用的拆法

如果一个能力同时需要：

- 显式命令入口
- 运行链路扩展
- 长期状态

那很可能应该拆成：

- `action` 负责显式调用
- `pipeline / guard / effect / resolve` 负责链路扩展
- `service` 负责长期状态
