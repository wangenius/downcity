# Env and downcity.db Design

Use one rule:

- shared secrets and shared infrastructure live in `~/.downcity/downcity.db`
- project behavior lives in `downcity.json`

## `downcity.db` owns

- `env_entries`
- `channel_accounts`
- `chat_auth_roles`
- `chat_auth_role_permissions`
- `chat_auth_channel_defaults`
- `chat_auth_user_roles`
- `model_providers`
- `models`
- other City-owned encrypted settings

## Project `downcity.json` owns

- `execution.modelId`
- `plugins.*`

## Why

- secrets should not be duplicated into project config
- plugin enablement and plugin-owned dependency config are project behavior
- City storage should not hide project-specific plugin state

## Runtime load order

1. the host may read City-managed env from `downcity.db`
2. the host passes runtime-needed values explicitly through `new Agent({ env })`
3. Agent reads project `.env`
4. project `.env` overrides duplicate host keys
5. Agent reads project `downcity.json`
6. resolve `plugins.*`
7. resolve channel credentials, chat authorization rules, and Federation AIService model catalog from `downcity.db`
