This section only does one thing: help you run a complete Downcity call chain as quickly as possible.

What you end up with is not just a demo page. It is the smallest town-shaped system:

- one reusable City
- one model record
- one `town_id`
- one `user_token` that an end user can actually use
- one `User City` that successfully calls City

You do not need a separate AI backend for every new town. A new project can start as a client first and connect to an already deployed City.

## Recommended path

1. [Use it now](/en/docs/city/quickstart/use-now): run the shortest end-to-end path with `cities/node` and `cli/city`
2. [Create City](/en/docs/city/quickstart/create-city): create City and let the first startup initialize the default database automatically
3. [Add model](/en/docs/city/quickstart/add-model): register models with Provider and AIService
4. [Issue `user_token`](/en/docs/city/quickstart/issue-token): give a town user a valid call credential
5. [Connect client](/en/docs/city/quickstart/connect-client): call City from the town with `User City`

## Assumptions in this quickstart

- the example City uses `sqlite:./data.sqlite`
- models come from runtime registration code, not handwritten model tables
- all towns currently share the same AIService model directory
- default models are defined through `Provider.model({ default })`
