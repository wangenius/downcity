# Model Config Reference

Downcity keeps model selection out of the agent project implementation. The project only stores a model ID; the shared AI service owns provider keys, provider routing, and the model catalog.

## Mental model

```txt
shared AI service model catalog
  -> town model list
  -> town model use <modelId>
  -> downcity.json execution.modelId
  -> agent session runtime
```

This keeps each agent project small and lets many products or agents reuse the same model infrastructure.

## Project binding

Agent projects bind through `execution.modelId`:

```json
{
  "execution": {
    "type": "api",
    "modelId": "quality"
  }
}
```

Notes:

- `execution.type` should be `api`
- `execution.modelId` should match a model returned by `town model list`
- provider secrets and provider connection details stay outside the project config

## Commands

List, bind, and test shared models:

```bash
town model list
town model use <modelId>
town model test <modelId>
```

Restart the agent after changing the binding:

```bash
town agent restart .
```

## What the project does not store

Project `downcity.json` should not store provider secrets or provider connection details. Those belong to the shared service runtime.
