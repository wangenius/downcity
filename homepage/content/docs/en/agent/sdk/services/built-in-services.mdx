# Built-in Services Today

The current Downcity runtime registers five built-in services:

- `chat`
- `contact`
- `task`
- `memory`
- `shell`

## First separate two layers

### 1. Runtime built-in services

These are the service classes already registered and used inside the full Downcity runtime.

### 2. Public SDK-facing service usage

This is the layer that makes sense to present directly to `@downcity/agent` users.

Those two layers are related, but they are not exactly the same thing.

Today, the most direct SDK-facing public service usage is:

- `ChatService`
- `BaseService` for custom services

## `chat`

Purpose:

- owns Telegram, Feishu, and QQ channel integration
- owns channel state
- owns the queue worker
- handles chat ingress and reply dispatch

This is the most typical and SDK-friendly built-in service today.

## `contact`

Purpose:

- handles agent-to-agent linking and approval
- stores contact relationships
- handles sharing, inbox, and contact chat

It is more of an internal collaboration runtime capability than the first service most SDK users start with.

## `task`

Purpose:

- handles task definitions
- owns cron runtime state
- supports scheduling, reloads, and run artifacts

It is a strong example of a long-lived background service, even though it is not the first SDK injection path most users need.

## `memory`

Purpose:

- memory indexing
- search
- reads
- flush operations

It behaves more like a service than a simple tool because it owns more persistent runtime state.

## `shell`

Purpose:

- owns shell session maps
- manages shell session lifecycle
- supports start, read, write, wait, close, and exec

It is a very good example of why some capabilities belong in a service instead of a one-shot tool.

## Best way to understand this as an SDK user

The safest learning order is:

1. get comfortable with `ChatService`
2. understand `BaseService` for custom services
3. treat `task`, `memory`, `shell`, and `contact` as proof-by-example of the runtime service model
