# Model Config Reference

Downcity keeps model selection out of the agent project implementation. The project only stores a model ID; the Federation AIService owns provider keys, provider routing, and the model catalog.

## Mental model

```txt
Federation AIService model catalog
  -> city
  -> Downcity selects a City model
  -> downcity.json execution.modelId
  -> agent session runtime
```

This keeps each agent project small and lets many products or agents reuse the same model infrastructure.

## Project binding

Agent projects bind through `execution.modelId`:

```json
{
  "execution": {
    "type": "api",
    "modelId": "quality"
  }
}
```

Notes:

- `execution.type` should be `api`
- `execution.modelId` should match a model exposed by the connected Federation AIService
- provider secrets and provider connection details stay outside the project config

## Where commands live

Connect Downcity to City, write the project binding, then restart the agent after changing it.

Concrete workflows live in [Downcity City docs](/en/docs/reference/cli), [Downcity config docs](/en/docs/reference/cli), and [Agent restart docs](/en/docs/reference/cli).

## What the project does not store

Project `downcity.json` should not store provider secrets or provider connection details. Those belong to the Downcity runtime.
